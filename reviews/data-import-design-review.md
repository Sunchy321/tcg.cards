# `data-import-design.md` 评审意见

## 评审结论

通过，可以进入实施计划阶段。

## 通过理由

- 设计已经明确支持多来源与来源级配置
- 字段级策略覆盖了需求中的四种核心语义：覆盖、忽略、只覆盖特定值、必须审批
- 将原来的“全量人工审批”调整为“先留痕，再按风险走自动应用、批量审批或逐条审批”
- 对 `*_data` / `*_app` 的职责划分比较清楚，避免把导入状态和用户审批动作混在一起
- 已经预留来源优先级、批量审批、字段 override 与回滚链路等扩展点

## 需要注意的问题

- `fieldPath` 必须有注册表或校验机制，否则规则容易失配
- `overwrite_when_matched` 的 matcher 需要固定表达能力，避免后续实现成过度自由的脚本系统
- `auto_apply` 的适用范围必须足够保守，建议只对白名单低风险字段开放
- `batch_review` 的分组键必须稳定且可解释，避免把不同语义的更新批在一起
- `import_review_actions` 与 `import_field_changes.decisionStatus` 之间要明确唯一事实来源
- `import_review_actions` 属于用户审批动作，实施时必须保留在 `*_app`
- 更新日志建议先预留 `inline / hash / ref / storageMode` 字段，再把对象存储、分区和归档作为进阶目标推进

## 实施建议

- 第一版先支持结构化 matcher，例如 `is_null`、`equals`、`in`
- 第一版先建立低风险字段白名单，再逐步放开 `auto_apply`
- 第一版把批量审批做成稳定分组的 service 能力，不单独设计复杂工作流引擎
- 第一版可以先以内联日志为主，但必须保留后续冷热分层所需字段
- 第一版只接一到两个真实来源，先验证规则模型是否够用
