# `agent-knowledge-design.md` 评审意见

## 结论清单

### 必须修改

无。设计文档已经明确了数据库分层、派生知识索引表、运行态表、证据模型、服务层工具、增量索引链路与多游戏扩展边界，可以进入实施计划阶段。

### 建议修改

- 在正式实现前确认 PostgreSQL 是否已启用 pgvector，并固定向量维度、索引类型和算子配置，避免后续重建成本失控。
- 在实现阶段补充一份 `evidence bundle` 的 Zod 契约，避免不同游戏的工具返回结构漂移。
- 为 `CardRelation.relation` 和 Hearthstone 相关 relation 类型建立一份服务层允许值清单，减少关系语义随实现扩散。
- 明确切块策略的默认参数，例如规则文档按节点切块，裁定按单条切块，卡牌说明按“摘要 + 正文 + 扩展说明”分层切块。
- 明确 Cloudflare embedding 模型的固定版本、超时策略和重试策略，避免 query embedding 与 document embedding 行为漂移。
- 明确统一智能体的问题路由规则，优先依赖显式游戏参数，其次再做问题分类。

建议默认值：

- 默认 embedding 模型：`@cf/baai/bge-m3`
- 默认维度：`1024`
- 默认距离度量：cosine distance
- 默认 PG 索引：HNSW + `vector_cosine_ops`
- query embedding 失败时直接降级为纯结构化检索
- document embedding 异步任务最多重试 `3` 次

### 可后置

- 个性化推荐与长期用户画像
- 多模型 embedding 回填
- 复杂反馈分析与自动质检报表
- 跨游戏混合检索

## 总体评价

这份设计的优点是边界收得比较稳，没有把“智能体知识”误解成重复建设一套新的卡牌主库。

设计明确区分了三类数据：

- 主领域事实继续留在 `magic` / `hearthstone`
- 可重建的索引、切块、embedding 进入 `*_data`
- 对话、反馈、工具轨迹等运行态进入 `*_app`

这个划分与仓库现有 schema 结构一致，后续迁移、清理、增量重建都更容易落地。

服务连接方式也合理。设计没有让模型直连数据库，而是要求模型只通过应用内知识服务调用受控工具，这一点对于精度、权限控制、可追溯性都非常关键。现有 ORPC 与搜索 DSL 已经足够支撑第一阶段的精确检索能力，因此实现风险可控。

当前版本的设计进一步收敛了向量方案：PostgreSQL 同时承担事实存储与向量存储，Cloudflare 只负责 embedding 推理。这让系统避免了“PG 主库 + 外部向量库”双系统同步的额外复杂度，也更符合当前仓库对 PG 的依赖现状。

近实时更新方案同样合理。设计复用了现有导入和 watcher 的思路，用 `knowledge_index_jobs` 做显式任务记录，比数据库触发器更容易排查与重放。需要注意的是，document embedding 应异步生成并回写 PG，避免把 Cloudflare 模型推理放进用户请求关键路径。

**结论：** 设计可行，建议进入实施计划阶段。建议项不阻塞计划编写。

## 风险评估

### 低风险

- 现有 ORPC 查询入口已经覆盖 Magic 与 Hearthstone 的核心读取场景，包装成智能体工具的成本较低。
- 现有 `packages/search` 已能承担结构化查询第一段，不需要立刻引入新的搜索框架。
- `*_data` 与 `*_app` 的分类与仓库现有规则一致，数据库落地路径清晰。

### 中风险

- pgvector 的启用状态、向量索引参数和维度契约若未提前固定，后续可能触发整库重建。
- Cloudflare embedding 推理属于外部依赖，延迟、配额、成本和模型变更都需要在服务层做兜底。
- `@cf/baai/bge-m3` 适合多语言检索，但其维度、距离度量和索引参数应在首期固定，避免上线后再换模型触发整库回填。
- 多版本文本证据若在服务层处理不严，容易出现 Hearthstone 版本混用或规则版本混用。
- 统一多游戏智能体若过早推进，可能让路由和证据引用复杂度显著上升。

### 排除的风险

- 主卡牌 schema 大规模改造风险：本设计不涉及
- 外部公开 API 稳定性风险：本设计当前不依赖
- 自动写回规则或卡牌数据的事务风险：本设计不涉及
