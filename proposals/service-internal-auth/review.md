# service-internal 登录服务评审

## 评审结论

建议按修正版方案实施。

## 通过理由

1. 方案保留了 `service-internal` 作为 `app-*` 认证服务的角色
2. 方案同时保留 `site-*` 本地认证入口，符合当前运行时边界
3. 共享 auth 工厂能够减少 `site-*` 与新服务之间的重复实现
4. 使用 `Hono + Worker` 与“内部服务”语义更一致，不再引入不必要的页面运行时
5. 不引入数据库迁移，实施风险可控

## 主要风险

### 1. 这不是单入口统一认证

本次不会把所有运行时都统一到一个认证入口；`site-*` 与 `app-*` 会继续保留不同入口。

### 2. 需要持续约束共享边界

如果后续继续把 `site-*` 的认证或 RPC 通过 HTTP 转发给 `service-*`，这份边界会再次失效。

### 3. 需要明确配置职责

`service-internal` 需要自己的认证配置；`site-*` 也需要自己的本地认证配置，但不再依赖 `INTERNAL_AUTH_URL` 这类代理地址。

## 结论建议

按当前边界继续推进：`service-internal` 服务 `app-*`，`site-*` 保留本地认证，仅复用共享 auth 代码。
