# 炉石双职业渲染逻辑收敛设计

> 稳定的运行时边界、数据归属和同步规则以 [../../docs/project-architecture.zh-CN.md](../../docs/project-architecture.zh-CN.md) 为准。本文只描述炉石图片渲染逻辑的增量设计；若有冲突，以主架构文档为准。

## 背景

近期炉石图片渲染逻辑在双职业卡牌上的处理方式发生了变化，导致当前实现与历史渲染结果不一致。

当前已知的确认信息是：

- `35.4` 是已经确认会产生双职业渲染分歧的版本边界

也就是说，至少从 `35.4` 开始，双职业卡牌的渲染结果已经不能再直接视为与此前版本完全等价。

双职业卡牌的渲染不是单纯的样式问题，而是会影响：

- 卡面最终视觉结果
- `renderHash` 和图片去重结果
- 需求导出与外部渲染器之间的兼容性
- 本地导入、缓存复用和存量图片迁移判断

如果不把双职业逻辑收敛成稳定、可解释、可回放的规则，后续每次渲染实现调整都会继续放大图片差异和缓存漂移问题。

## 目标

- 明确双职业卡牌的渲染决策边界
- 将双职业渲染逻辑固定为可版本化的协议约束
- 让导出、渲染、导入三段流程对双职业卡牌使用同一套判定结果
- 避免后续实现变更再次无意改变双职业卡牌的视觉结果

## 非目标

- 不重做整体炉石卡牌模型
- 不修改图片存储结构
- 不修改本地导入与 R2 同步的整体流程
- 不在本轮引入新的图片格式
- 不在本轮追求双职业渲染逻辑的历史全量自动回填

## 现状问题

### 1. 双职业渲染决策存在版本漂移

当前双职业卡牌的渲染处理不再与历史版本保持一致，说明“双职业是否视为渲染输入差异”的判定已经发生了隐式变化。

目前已知这类分歧至少覆盖：

- `35.4` 及其后续沿用该逻辑的版本

这类变化如果没有显式版本化，就会让同一张卡在不同时间导出的图片出现不一致。

### 2. 渲染逻辑和图片去重逻辑容易被同一份隐式规则绑定

当前图片系统依赖 `renderHash` 做去重和缓存复用。

如果双职业判定规则没有单独冻结，那么以下结果都会被连带影响：

- `renderModel` 的规范化结果
- `renderHash` 的稳定性
- 需求文件中的 `requestId` 和 `output.fileName`
- 本地和远端图片是否被判定为同一张图

### 3. 外部渲染器需要稳定输入契约

外部渲染器只应接收已经确定好的单图输入。

如果双职业逻辑继续留在渲染器内部自由演化，就会破坏“导出侧先决策、渲染侧只执行”的边界，导致协议无法稳定。

### 4. 当前本地化写库主键无法自然承载“同文案、不同渲染”分叉

当前 `entity_localizations` 的主键由以下字段组成：

- `cardId`
- `lang`
- `revisionHash`
- `localizationHash`

`renderHash` 不在主键中。

这意味着：如果游戏修改了渲染方式，导致同一份 localization 内容对应的 `renderHash` 发生变化，但 `localizationHash` 保持不变，那么当前写库路径不会把它当成一条新的 localization 记录。

更关键的是，现有写入逻辑对已存在主键的 localization 行，主要只更新版本元数据，而不是完整替换 `renderHash` 和 `renderModel`。

因此在“同一 localizationHash、不同 renderHash”的情况下，当前实现既不会自然插入新行，也不能保证旧行中的渲染 payload 被正确替换。

这说明系统当前隐含了一个前提：

- 只要 localization 主键不变，渲染 payload 就应当稳定

而本次双职业问题已经证明这个前提并不可靠。

## 设计决策

### 1. 将双职业逻辑定义为显式版本化规则

双职业渲染规则必须以版本号或等价的稳定标识来表达。

每一次影响视觉结果的双职业判定变化，都必须进入新的版本决策，而不是直接覆盖旧行为。

已知的 `35.4` 分歧应作为首个明确锚点进入版本语义，而不是继续作为实现侧隐含行为存在。

### 2. 双职业判定前置到需求导出阶段

双职业卡是否使用特定渲染路径，应当在导出阶段就确定，并写入需求模型或相关派生字段。

渲染器只负责消费已经确定的输入，不再临时推导双职业分支。

### 3. 渲染输入只携带已冻结的双职业结果

渲染请求中不应再携带“待推导的双职业意图”，而应携带导出阶段已经决定好的结果。

这样可以保证：

- 相同导出结果得到相同渲染结果
- 双职业规则变化不会悄悄污染其他卡种
- 渲染器实现可以替换，但输入契约保持稳定

### 4. 兼容策略以版本切换而不是隐式回溯为主

对于历史双职业图片，不应依赖渲染器当前版本自动猜测旧规则。

如果后续需要重跑历史图片，应基于明确版本或明确回填任务，而不是让新规则覆盖旧规则。

### 5. 版本语义应同时影响 hash 与协议说明

只要双职业规则会影响最终像素，就必须同步影响：

- 渲染输入的派生字段
- `renderHash` 或其等价上游输入
- 文档中的协议说明
- 相关测试样本

否则就会出现“协议写的是一套，图片系统算的是另一套”的再次漂移。

### 6. 需要显式定义“同文案、不同渲染”的落库策略

双职业规则收敛不能只停留在渲染器输入层，还必须回答数据库层面的处理方式：

- 是让 `renderHash` 进入 localization 身份
- 还是在现有主键下允许完整替换渲染 payload
- 还是引入单独的渲染版本维度

在没有先确定这个策略之前，单纯调整双职业渲染规则只会继续放大“导出结果变了，但写库身份没有变”的冲突。

## 风险与取舍

### 1. 版本化会增加一次迁移成本

把双职业逻辑显式版本化后，短期内会需要更新导出逻辑、测试样本和部分图片判定结果。

这是必要成本，因为当前问题本身就是由隐式漂移引起的。

### 2. 历史图片可能与新规则不一致

如果新规则被正式冻结，旧图片不一定需要重新生成。

更合理的做法是把旧图片视为旧版本产物，并通过版本字段区分，而不是强行把所有历史图片改写成最新规则。

### 3. 渲染器和导出端的责任边界需要一次性划清

如果双职业判定仍然分散在多个位置，就会再次出现版本不一致。

因此这次设计必须明确：

- 谁决定双职业结果
- 谁负责把结果写入输入
- 谁只负责执行

### 4. 本地化身份与渲染身份当前并未完全分离

当前模型把 localization 身份和 render 身份部分绑在同一条记录上，但主键又只包含 localization 相关字段。

如果继续沿用这套结构而不补充显式策略，那么任何“文案不变、渲染变化”的规则调整，都会再次撞上相同问题。

## 验收标准

- 双职业卡牌的渲染规则有明确版本边界
- 导出阶段可以稳定地产生同一张卡的固定渲染输入
- 渲染器不再依赖未冻结的双职业隐式规则
- 双职业相关的测试样本与版本语义一致
- 后续规则变更不会无意影响历史已定义版本
- “同一 localizationHash、不同 renderHash”的情况有明确且一致的写库策略
