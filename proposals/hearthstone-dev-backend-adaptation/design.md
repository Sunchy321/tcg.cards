# 炉石站点适配 dev 新后端方案

## 背景

当前分支基于旧的炉石数据链路实现了多项前端与导入修复，包括高级搜索、种族和阵营、卡牌详情页、相关卡牌、衍生物、Token、兆示、巨型、卡图与文本展示等。

`dev` 分支已经重构炉石数据后端：

- 删除旧的 `scripts/import-hearthstone.ts`
- 删除旧搜索页面与旧搜索服务
- 使用新的本地 hsdata 投影流程
- 卡牌详情接口改为基于 `CardEntityView`、`EntityRelation`、`renderHash`、`revisionHash`、`isLatest`
- 卡图改为优先使用 `renderHash` 对应的新资源路径

因此旧代码不能直接合并，需要把已确认的业务规则迁移到新后端与新前端入口。

## 目标

- 以 `dev` 的后端和数据模型为基础继续开发。
- 保留当前分支已经确认过的炉石站点体验。
- 避免把旧搜索、旧导入脚本、旧卡图路径强行带回新架构。
- 让前端适配新的 `CardEntityView`、`EntityRelation` 和新卡图资源。

## 需要迁移的既有行为

### 数据规则

- 种族来源应读取 `CARDRACE`，必要时回退到 `RACE`。
- 阵营筛选使用标签语义：
  - 所有阵营
  - 污手党
  - 玉莲帮
  - 暗金教
  - 异虫
  - 人族
  - 星灵
- 原始 `TERRAN` 应在应用层统一映射为 `human`。
- 非实际卡牌、Token、衍生物、酒馆战棋专属卡不应出现在普通搜索结果中。
- Token、衍生物、巨型附属物、泰坦技能、兆示升级形态应通过详情页相关卡区域展示。
- 兆示需要特殊处理：
  - 普通兆示来源展示基础士兵衍生物。
  - 中立兆示来源展示所有基础兆示衍生物。
  - 兆示升级形态作为对应基础衍生物自己的相关卡展示，不在来源卡页面重复摊开。

### 前端行为

- 详情页语言切换应同时刷新主卡与相关卡内容。
- 语言切换控件放在卡图区域，与普通/金卡切换保持同类样式。
- 搜索结果默认显示简体中文。
- 结果页系列标识显示系列名，不显示裸 `SET_xxx`。
- 没有有效系列的重复卡应从普通结果中隐藏。
- 费用图标使用法力水晶，而不是普通蓝色圆点。
- 文本中的 XML/HTML 残留、`$a5`、`{0}` 等占位符应清理。
- 某些卡牌文本允许压平成一行，避免无意义强制换行。
- 相关卡区域保留，版本历史放右侧。

## 适配入口

初步判断需要适配这些新入口：

- `apps/site-hearthstone/server/orpc/hearthstone/card.ts`
  - 扩展或调整 `full` 返回的 `relatedCards`
  - 确认 relation 过滤、语言、版本和 `revisionHash` 规则
- `apps/site-hearthstone/app/pages/card/[id].vue`
  - 迁移详情页布局、语言切换、相关卡展示、版本历史和文本渲染
- `apps/site-hearthstone/app/components/CardImage.vue`
  - 保留新 `renderHash` 资源路径
  - 适配普通、金卡、钻石、签名和酒馆战棋图
- `apps/site-hearthstone/app/components/RichText.vue`
  - 保留文本清理和可选换行压平能力
- 新 hsdata 投影流程
  - 找到替代旧 `scripts/import-hearthstone.ts` 的投影位置
  - 将种族、阵营、相关卡关系、文本清洗迁入新投影层

## 实施计划

- [ ] 确认 `dev` 新 hsdata 投影入口和 schema 字段来源。
- [ ] 把种族、阵营、文本清洗和 relation 生成规则迁入新投影层。
- [ ] 确认 `EntityRelation` 是否已经能承载兆示、巨型、泰坦、疫病、Token 等关系。
- [ ] 适配卡牌详情页：
  - [ ] 卡图区域语言和卡图变体切换
  - [ ] 主信息区系列名、文本、类型、种族、职业显示
  - [ ] 相关卡区域
  - [ ] 右侧版本历史
- [ ] 重新实现或接入搜索：
  - [ ] 普通搜索
  - [ ] 高级搜索
  - [ ] 系列浏览
  - [ ] 随机卡牌
- [ ] 使用之前发现问题的卡做回归验证。

## 风险

- `dev` 已删除旧搜索整套文件，因此高级搜索不能靠简单合并恢复。
- 新投影流程可能位于 desktop 或 bun runtime 侧，需要先找清楚入口再动手。
- 当前分支已有旧实现和 `dev` 大量删除重构互相冲突，建议以 `dev` 为底座迁移规则，而不是把 `dev` 合到当前旧 UI 上硬修。
- 需要小心区分真实可收集卡、Token、衍生物、酒馆战棋牌和旧版本快照，避免搜索结果再次混入非目标数据。

## 建议执行方式

建议新建基于 `dev` 的迁移分支，然后按计划逐块迁移：

1. 先迁数据投影规则。
2. 再迁详情页。
3. 再恢复搜索和系列页。
4. 最后做回归验证。

这样可以最大限度复用 `dev` 后端改动，同时把当前分支已经验证过的需求重新落到新架构上。
