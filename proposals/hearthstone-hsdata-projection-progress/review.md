# hsdata 本地投影进度事件评审

> 稳定的运行时边界、数据归属和同步规则以 [../../docs/project-architecture.zh-CN.md](../../docs/project-architecture.zh-CN.md) 为准。本文只记录本需求的评审结论；若有冲突，以主架构文档为准。

## 评审结论

建议通过。

这次改动是典型的可观察性补齐，不改变投影边界，也不引入新的数据归属复杂度。

## 阻塞问题

无。

## 主要判断

### 1. 投影应使用独立事件，而不是复用导入事件

导入事件已经绑定了 source 文件、jobId、batch 和实体数。

投影事件如果复用同一结构：

- 前端类型会被迫接受大量无意义空字段
- 阶段标签会变得含混
- 后续扩展投影专属字段更困难

独立事件更清晰，也更符合当前边界。

### 2. 百分比基于 snapshot 数量是当前最稳妥的折中

投影内部最终会产生 entity/localization/relation 多类输出，但这些数量在流程前半段并不稳定。

相比之下：

- snapshot 总量在最开始就确定
- 每完成一条 snapshot 都可以稳定递增

因此用 snapshot 进度做用户可见百分比是合理的。

### 3. 前端复用导入卡片可以控制改动面

本需求的重点是：

- 看见阶段
- 看见数量
- 看见进度

不是重新设计页面。复用现有导入卡片结构能降低实现风险，也更利于用户理解。

## 非阻塞建议

### 1. Rust 事件辅助函数放在投影模块内

投影事件当前只由投影模块使用，没有必要提前抽成通用框架。

### 2. `failed` 事件要覆盖早失败路径

不仅写库失败需要发 `failed`，前置校验、加载 raw snapshot、解析 projection tag 失败时也应发出最终失败事件。

### 3. 页面过滤条件保持最小集

只按 `sourceTag` 过滤即可，不需要额外引入 jobId 或执行 token。

投影当前就是单 sourceTag 命令，过度设计只会增加状态同步复杂度。

## 通过条件

- Rust 投影链路能够发出阶段性进度事件
- 前端能够监听并展示投影进度条
- 完成和失败时页面状态可以收口
