# 应用内重量级任务运行框架设计

> 稳定的运行时边界、能力分层和数据归属规则以 [../../docs/project-architecture.zh-CN.md](../../docs/project-architecture.zh-CN.md) 为准。本文只描述应用内重量级任务运行框架的需求级设计；若有冲突，以主架构文档为准。

## 背景与前提

当前仓库已经存在若干“重量级任务”实现，以 `hearthstone publish` 为代表，具备以下典型特征：

- 前端触发、后端执行
- 涉及大面积数据库读取与写入
- 执行时间明显长于普通页面请求
- 前端需要持续可见的进度反馈
- 任务通常不应重复并发执行
- 任务失败后需要明确的重试、终止和残留状态清理策略

现有实现已经验证了一部分可行能力，例如：

- 任务可以分阶段推进
- 任务可以在原子块边界协作式停止
- 前端可以订阅实时进度
- 数据库中可以保留批次结果与残留状态

当前已存在的重量级任务类型，至少包括以下三类：

1. `reanchor`
   - 对本地数据库中的 baseline 进行对齐
   - 主要在单一数据库域内完成候选读取、对比和写回
   - 更接近“本地权威数据重新对齐”类任务
2. `publish`
   - 将本地数据库中的批量数据写入远程数据库
   - 至少跨越“本地数据库状态”和“远程数据库写入结果”两个一致性域
   - 更接近“跨库发布”类任务
3. 批量渲染
   - 批量向第三方服务器发起请求
   - 将返回的图片写入文件
   - 将结果 summary 写回数据库
   - 至少跨越“第三方服务响应”“文件系统写入”“本地数据库 summary”多个一致性域
   - 更接近“外部副作用驱动”的批处理任务

这些现有任务共同说明，本文要覆盖的并不只是“单库内长事务”。

- 有些任务主要在单一数据库域内推进
- 有些任务天然跨本地数据库、远程数据库、文件系统或第三方服务
- 因此后续运行框架设计不能默认假设所有业务副作用都能与框架状态写入落在同一个物理事务中
- 对跨一致性域任务，框架更需要关注幂等性、重复检测、恢复边界和残留状态收束

但现有实现仍偏向具体任务专用逻辑，尚未收敛为一套应用内统一的重量级任务运行框架。随着后续更多导入、发布、重建、批量修复、批量回填类任务出现，继续为每个任务分别发明运行骨架，会带来以下问题：

1. 任务状态模型不统一
2. 暂停、取消、继续、重试语义容易漂移
3. 前端进度展示难以复用
4. 同类型任务的串行约束难以统一治理
5. 残留运行态和僵尸任务清理逻辑会在各处重复出现

因此，需要先设计一套足够通用、但仍然保持在应用内部可控范围内的“重量级任务运行框架”。

本文以下列运行背景作为明确前提；后续凡是涉及调度、并发、租约、清理和前后端职责切分的判断，均应优先服从这些背景约束，而不是按通用多租户平台或高吞吐任务系统的假设展开：

1. 单 `app`、单 `service`
   - 首轮框架服务于单个应用内部的单个后端服务
   - 不按多应用共享任务平台建模
   - 不按多服务共同消费同一任务域建模
2. 目标平台是管理维护者的个人电脑
   - 运行环境首先是假定为管理维护者本人使用的个人电脑
   - 不按长期在线、多节点、高可用生产集群建模
   - 不以多人同时操作为默认场景
3. 任务都是重型运维任务
   - 任务本质上是面向维护者的重型操作
   - 典型任务包括导入、发布、重建、批量修复、批量回填等
   - 这些任务首先追求正确性、可观测性和可控终止，而不是高吞吐调度能力
4. 任务均为手动触发、低频次操作
   - 不把高频自动调度、海量任务堆积或持续后台吞吐作为首轮前提
   - 并发风险整体较低
   - 首轮主要需要防范的是用户重复点击、重复提交或在任务未收束时再次触发同类操作
5. 调度器是轻量调度，而不是复杂并发调度系统
   - 首轮默认只有一个执行器
   - 首轮默认调度器是单线程、单任务推进模型
   - 不按多线程调度器、多 worker 协同抢占或复杂任务编排系统建模
   - 设计核心应放在任务状态控制、恢复边界、收束语义和前后端一致性上
   - 而不是优先解决多线程竞争、复杂抢占或高吞吐排队问题
   - 在当前前提下，即使未来允许不同 `taskType` 的任务同时执行，也先假设它们之间不存在执行期竞争或共享运行时写入竞争
   - 若后续发现不同任务之间确实会发生执行期竞争，再单独升级并发控制模型
6. 前端页面采用“单页面单当前任务”的产品心智模型
   - `publish` 页面始终只处理一个当前 `publish` 任务
   - `reanchor` 页面始终只处理一个当前 `reanchor` 任务
   - 页面即使支持切换 `taskScope`，当前主任务也应始终按新的 `activeScope` 重新计算
   - 页面保持的是“单页面单当前任务”的心智，而不是“切换 scope 后仍粘住旧任务实例”的心智
   - 在当前产品边界下，也不希望同类重型任务并行运行

这些背景前提直接导出若干首轮设计倾向：

- 优先保持模型收敛，避免为低概率并发场景过早引入额外表、额外调度层级或分布式协调抽象
- 优先防止“错误点击导致的重复启动”与“残留状态未收束”，而不是优先优化复杂排队、公平调度或多 worker 抢占
- 优先把调度器视为轻量单线程控制器
  - 其核心职责是发现可执行任务、推进状态、处理取消/恢复和清理残留
  - 而不是承担复杂多线程并发控制器的职责
- 优先按“单执行器 + 无跨任务执行期竞争”建模
  - 不为理论上的旧执行器竞争、跨任务写入竞争或执行期抢占预先引入重型 fencing/lease 协调体系
- 优先把任务框架建模为“低频、手动、可观测、可收束”的应用内运维能力，而不是通用批处理平台
- 优先维持每类任务页面只承载一个当前任务的稳定交互模型
  - 该当前任务始终按页面当前 `activeScope` 重新计算
- 后续若产品边界从个人维护场景扩展到多人协作或更高频自动化运行，再单独扩展并发与调度模型

这些前提还应继续约束后文的评审与建模取向：

- 不按多进程、多执行器竞争、多节点抢占或分布式 worker 系统补足模型闭合性
- 若某个复杂状态、子状态、字段或调度规则，只是为了覆盖未来可能出现的更复杂部署形态，而不是当前单进程背景下的真实问题，则首轮不采纳
- 若发现语义矛盾，优先尝试删除或收缩语义，而不是优先新增状态、派生态或额外控制层
- 任何新增字段或状态，都必须能回答：
  - 若没有它，在当前单进程、低频、手动触发的背景下，具体会出现什么错误行为
  - 若答不出当前错误行为，就视为超前设计

首轮明确禁止以下过度设计倾向：

- 不为本地前端 + 本地 service 的场景设计页面级严格排序协议
- 不为复合页面快照引入额外的全局页面版本号、复杂合并算法或多对象严格增量同步语义
- 不要求前端对 `pageTask + occupyingRun + latestScopeRun` 做复杂乱序合并
- 实时通道优先承担“提示有变化”或“补充当前主任务局部更新”的职责
- 页面真相始终以重新拉取快照为准；若两者冲突，永远以快照覆盖实时消息

## 目标

1. 提供一套应用内部统一的重量级任务运行模型，供不同任务类型接入
2. 支持前端触发、后端执行、前端实时接收进度
3. 支持协作式暂停与取消，允许在原子块边界生效
4. 支持页面切换或重新进入后自动接续当前任务的进度展示和控制入口
5. 支持任务失败后的重新执行
6. 对支持继续语义的任务，允许从中断点继续；对不支持继续的任务，框架允许显式声明不支持
7. 支持按任务类型串行执行，并为未来放宽到更细粒度并发保留扩展空间
8. 支持后端重启后的残留运行态清理，避免数据库中长期滞留“进行中但实际未运行”的任务
9. 提供足够稳定的阶段化进度模型，让前端能够计算阶段内百分比、显示明确的整体阶段指示，并展示整体已用时间
10. 提供一套统一的重任务前端组件，避免不同任务页面各自实现状态展示与控制条
11. 提供一个全局任务检视入口，让维护者可以在任意页面通过标题栏按钮与 popup 快速查看当前任务状态

## 非目标

1. 本轮不要求后端进程重启后自动恢复执行中的任务
2. 本轮不要求所有任务都必须支持中断点继续
3. 本轮不引入独立的外部工作流平台或独立调度集群
4. 本轮不把“任意时刻抢占式暂停”作为目标能力
5. 本轮不要求一开始就支持跨服务、多节点、分布式 worker 执行

## 已确认需求边界

### 1. 运行边界

- 任务由前端触发
- 任务实际在后端执行
- 前端需要随时感知进度变化

### 2. 任务特征

- 涉及大面积数据库读取与写入
- 通常执行时间较长
- 通常同一类任务一次只能执行一个
- 任务需要具备完整进度条展示能力

### 3. 暂停与取消语义

- 所有重量级任务都必须支持取消
- 暂停与继续作为成对出现的可选能力，由任务类型显式声明是否支持
- 暂停与取消不要求在任意时刻精确生效
- 任务允许拆分为若干“中间不可打断”的原子块
- 暂停与取消在原子块边界协作式生效

### 4. 恢复与继续语义

- 页面重新进入时，如果后端存在正在运行的任务，前端必须能够重新接续进度显示与控制入口
- 默认不要求后端重启后自动断点续跑
- 需要支持清理数据库中残留的“运行中但实际未执行”的任务状态
- “继续”能力允许是任务类型可选能力，不要求所有任务都支持

### 5. 重试语义

- 默认“重试”表示从头重新执行
- “继续”表示从中断点继续执行
- 部分任务可以不支持继续，但必须明确声明

### 6. 进度模型

- 任务必须拆分为多个阶段
- 首轮默认阶段应为可预先给出总量的 `bounded` 阶段
- 少数无法预先给出总量的任务允许声明 `unbound` 阶段，但必须显式作为例外接入
- `unbound` 阶段仍必须持续提供当前已完成量，不允许长期显示空进度条
- 只有初始化、收尾、总结报告等极短的简易阶段，才允许声明为 `simple` 阶段，并省略当前量与总量
- 每个阶段内部的推进速度应尽量稳定，以便前端估算剩余时间
- 后端应向前端提供：
  - 当前阶段
  - 当前阶段已完成量
  - 当前阶段总量（若该阶段为 `bounded`）
  - 当前阶段进度模式
- 前端基于这些数据自行计算阶段内百分比
- 前端基于阶段列表与顺序显示整体阶段指示
- 前端基于任务开始时间显示整体已用时间
- 对于 `unbound` 与 `simple` 阶段，只展示与其模式一致的进度信息

### 7. 串行约束

- 当前首轮默认按“每种任务类型的占槽运行实例最多同时存在一个”设计
- `paused` 不作为并发执行实例，但仍保留同类型任务的创建槽位
- 不把“全局永远只能运行一个任务”写死到模型中
- 设计上应保留未来扩展到更细粒度锁或多任务并行的空间

## 方案比较

### 方案一：纯数据库状态机 + 前端轮询

#### 思路

- 后端把任务状态、阶段、进度和当前控制意图都写入数据库
- 前端通过轮询读取数据库快照
- 执行器在原子块边界读取暂停或取消控制意图

#### 优点

- 结构简单
- 数据库天然成为唯一真相来源
- 页面刷新后的进度恢复实现直接
- 残留状态清理容易实现

#### 缺点

- 进度刷新依赖轮询，实时体验较弱
- 高频轮询会带来额外请求成本
- 暂停、取消与阶段切换的反馈会存在可见延迟
- 难以提供足够丝滑的前端进度响应

#### 结论

该方案适合作为保守实现，但不满足当前对“前端要随时响应进度”的体验要求，因此不作为首选方向。

### 方案二：数据库权威快照 + `session_bound` 内存恢复上下文 + 实时进度通道 + 协作式执行器

#### 思路

- 数据库保存任务运行的权威快照
- 仅对命中 `session_bound` 恢复契约的暂停边界，额外依赖受约束的进程内恢复上下文
- 后端执行器按“任务 -> 阶段 -> 原子块”推进
- 每完成一个原子块，就落库最新进度、刷新心跳并检查当前控制意图
- 前端先读取数据库快照，再通过实时通道持续接收进度更新

#### 优点

- 同时满足正确性与前端体验
- 适合协作式暂停、取消与部分任务继续
- 页面重新进入后可以先恢复快照，再接续实时状态
- 便于统一治理任务类型级串行约束
- 便于统一实现残留运行态清理

#### 缺点

- 明显比纯轮询方案更复杂
- 需要同时维护数据库快照、`session_bound` 内存恢复上下文与实时事件
- 需要建立统一的任务状态模型，而不能只做单个任务特化逻辑

#### 结论

该方案最贴合当前需求，作为本设计的首选方向。

### 方案三：外部作业系统 / 队列系统 + 应用壳层

#### 思路

- 应用只负责创建任务、读取状态、展示进度和发控制动作
- 真正的执行由外部作业系统或队列 worker 负责
- 应用数据库与外部作业系统共同组成任务运行面

#### 优点

- 更适合未来多节点、多 worker、更多任务并行的扩展
- 执行压力更容易从主应用中拆分出去

#### 缺点

- 基础设施复杂度明显更高
- 一致性边界从“应用内部”扩展成“应用 + 外部作业系统”
- 暂停、继续、阶段总量、原子块边界等业务语义仍需自行实现
- 对当前首轮需求来说偏重

#### 结论

该方案可作为远期演化方向或特定高吞吐场景的备选，但不作为当前首轮设计方向。

## 现成框架调研结论

### 1. 对方案二的结论

方案二没有“开箱即用且完全匹配”的现成框架。

原因不是缺少通用队列系统，而是方案二的关键能力带有强业务语义：

- 每个阶段必须可提前得知总量
- 阶段内速度要足够稳定以支撑 ETA
- 暂停与取消要在业务定义的原子块边界生效
- 不同任务类型对继续能力的支持不同
- 页面重新进入后要自动接管当前任务的展示与控制

通用 job queue 或 workflow 框架通常只能提供：

- 调度
- 派发
- 重试
- 并发控制
- 任务持久化

但不能直接替代应用内的任务阶段模型、协作式控制语义和前端进度协议。因此，方案二的核心运行框架应当自研。

### 2. 对方案三的结论

方案三存在可借用的现成框架，当前最值得关注的是：

- `Graphile Worker`
- `pg-boss`

它们与当前约束的贴合点在于：

- 都以 PostgreSQL 为基础
- 都面向 Node.js 生态
- 都具备成熟的调度、重试、延迟执行或并发控制能力

但它们更适合作为“作业系统底座”，而不是直接替代本设计要解决的业务任务框架。

### 3. Bun 兼容性结论

当前 service 运行在 Bun 驱动的 Node 程序中，数据库固定为 PostgreSQL。

对上述现成框架，需要保持以下判断：

- 从 Bun 的 Node API 兼容方向看，这些框架有较大概率可以运行
- 但它们的官方支持前提通常仍然写的是 Node 版本，而不是 Bun
- 因此不能把它们视为在 Bun 环境下“已被官方明确支持”的稳定方案

若未来要选择方案三或借用这些框架作为调度底座，应先做最小可行验证，而不能直接默认兼容。

## 当前代码实现的参考结论

以 `hearthstone publish` 为例，当前实现已经具备部分和目标框架相似的特征：

- 具备实时进度流
- 具备协作式停止
- 具备批次和基线等数据库落点
- 具备残留批次清理入口

但它并不是目标方案的完整实现，原因在于：

- 当前实现还没有把“数据库快照、`session_bound` 内存恢复上下文、实时事件”三者的职责边界收敛成统一模型
- 当前进度实时流更接近“进程内 job 状态推送”，而不是“数据库权威快照 + 实时事件投递”
- 当前能力主要服务于单一 publish 任务，而不是统一的通用任务框架

因此，现有实现可以作为经验参考，但不应直接约束本设计的最终模型。

## 选定方向

当前确认选择 **方案二：数据库权威快照 + `session_bound` 内存恢复上下文 + 实时进度通道 + 协作式执行器**。

选择原因如下：

1. 最符合“前端实时感知进度”的要求
2. 最适合承载协作式暂停与取消
3. 允许页面重新进入后自动接续显示与控制
4. 能把任务状态、阶段、控制意图、串行约束和残留清理收敛到统一模型中
5. 在不引入外部任务平台的前提下，复杂度仍保持在应用内部可控范围内

## 首轮核心设计方向

基于当前已确认结论，首轮框架固定采用以下方向。

首轮设计分为两个层次：

- 第一部分首先是架构与对象层面的模型，用来固定系统边界和核心概念
- 第二部分开始进入行为层面的模型，用来固定状态迁移和控制语义

这两部分不是纯概念讨论，后续都会继续约束代码模型和数据库模型。

## 第一部分：整体架构与核心对象模型

第一部分首先作为系统的架构心智模型存在，用来回答：

- 任务系统分为哪些层
- 每层分别负责什么
- 哪些核心对象是运行框架中的一等概念
- 哪些状态必须落数据库，哪些只属于实时通知

它不要求代码目录、类名、文件名与文档中的术语一一对应，但后续的代码结构与数据库设计必须服从这一层边界。

### 1.1 分层结构

首轮按以下四层拆分：

#### 1.1.1 任务定义层

负责定义某类任务“怎么跑”，包括：

- 任务类型标识
- 阶段列表
- 每个阶段如何计算总量
- 每个阶段如何切分原子块
- 是否支持继续
- 若支持继续，恢复点的最小契约是什么

该层主要是代码模型，而不是数据库配置中心。

首轮不要求把 `TaskDefinition` 写入数据库。更合适的实现方向是：

- 在代码中维护任务定义 registry
- 在任务运行记录中只保存 `taskType` 和必要的定义版本信息

#### 1.1.2 任务运行层

负责真正推进任务执行。

统一执行器按“任务 -> 阶段 -> 原子块”推进，每完成一个原子块后执行以下动作：

- 更新阶段进度
- 刷新心跳
- 检查当前控制意图
- 在需要时保存恢复点
- 向实时通道发布最新快照

该层不直接面向前端，也不应自由拼装临时 UI 状态。

#### 1.1.3 任务控制层

负责控制面能力，包括：

- 创建任务
- 校验同类型任务是否允许进入运行
- 下发暂停、继续、取消控制动作
- 触发执行尝试
- 清理残留运行态

前端不直接操纵执行器，只和这一层交互。

#### 1.1.4 任务观测层

负责把运行状态暴露给前端和其他调用方。

该层包含两条职责不同的链路：

- 数据库快照链路
  - 提供权威快照
- 实时事件链路
  - 提供状态变化通知

前端进入页面时，先读取数据库中的当前任务快照，再接入实时通道继续接收增量变化。

### 1.2 权威状态原则

首轮固定以下原则：

- 默认情况下，数据库中的任务运行快照是当前任务状态的权威来源
- 命中 `session_bound` 恢复契约的暂停边界，继续前提由“数据库快照 + 当前服务本次启动期间保留的进程内恢复上下文”共同决定
- `session_bound` 之外的阶段，不引入额外的内存状态作为继续前提
- 实时通道只负责通知状态变化，不负责承载权威快照
- 页面重新进入时，前端以数据库快照为起点恢复展示
- 实时事件丢失时，系统仍可以依赖数据库快照恢复一致状态

### 1.3 核心对象

首轮固定以下一等概念：

- `TaskDefinition`
- `TaskScope`
- `TaskRun`
- `TaskStage`

这些对象既是领域模型，也会继续向后约束实现模型，但它们在首轮中的承载方式并不完全相同。

#### 1.3.1 `TaskDefinition`

表示某类任务的静态定义，例如：

- `hsdata_publish`
- `image_rebuild`
- `tag_repair`

它主要是代码模型，而不是数据库模型。

首轮不要求单独建表持久化 `TaskDefinition`，原因是：

- 任务定义本质上依赖执行代码
- 即使写进数据库，最终仍需回到代码 handler 执行
- 首轮不需要引入动态任务注册中心

更合适的首轮方向是：

- 代码中维护任务定义 registry
- `TaskRun` 记录 `taskType`
- `TaskRun` 必须记录 `definitionVersion`，用于区分运行实例所依赖的定义版本

只要某个任务声明支持继续，`definitionVersion` 就不是可选调试信息。它必须参与恢复点解释契约，框架不能默认用“当前最新定义”去猜测解释旧运行实例的 `resumeToken`。

继续方式由任务定义静态声明，而不是由单次运行动态猜测。这里的“继续方式”首先是阶段契约，而不是整个任务运行实例的一次性总标签。首轮至少区分三类：

- `none`
  - 不支持继续
- `durable`
  - 当前暂停边界的继续只依赖持久化恢复点
- `session_bound`
  - 当前暂停边界的继续除持久化恢复点外，还依赖当前服务本次启动期间仍保留的进程内上下文

同一个任务定义中的不同阶段或块，可以进一步细化自己的恢复边界，但都必须落在这组静态恢复方式之内。

这里需要固定一个根本边界，避免后续和暂停语义混淆：

- `durable / session_bound` 描述的是“当前暂停边界如何继续”
- 它们不是“这个任务整体属于哪一类”的静态分类

因此：

- 一个 `TaskRun` 可以经过多个恢复契约不同的阶段
- 某次具体 `paused` 状态，只能命中一个明确的恢复契约
- 首轮默认按阶段声明恢复契约；若未来某阶段内部还要混用多种恢复边界，再继续细化到块级

首轮显式接受 `durable` 与 `session_bound` 这两种并列成立的阶段级恢复契约。二者都要求存在稳定可解释的持久化恢复点，但继续前提并不相同：

- `durable`
  - 继续前提完全由持久化状态决定
- `session_bound`
  - 除持久化恢复点外，还额外依赖当前服务本次启动期间仍然存在的进程内上下文

因此，首轮框架不把“继续语义的全部前提都来自数据库”当作硬约束。对绝大多数阶段，数据库快照就是继续判断的全部依据；只有对 `session_bound` 来说，数据库负责表达“任务停在了哪个可恢复边界”，当前服务本次启动期间保留的进程内上下文负责表达“这个边界当前是否仍可继续”；若两类信息缺一，则该运行实例不能继续。

这里还需要补一个与现有任务类型直接相关的首轮边界：

- 任务定义不只要声明恢复契约
- 还要声明原子块的副作用模型

首轮至少区分两类：

- `atomic`
  - 原子块的业务副作用与框架确认可以落在同一一致性边界内
  - 更适合单一数据库域内的任务
- `reconcilable`
  - 原子块的业务副作用可能跨越多个一致性域
  - 框架不要求这些副作用与任务快照提交天然处于同一物理事务
  - 任务必须依赖幂等写、重复检测、apply marker、对账或等价机制恢复一致性

`reanchor` 更接近 `atomic` 候选；`publish` 和批量渲染默认应按 `reconcilable` 设计，除非后续某个具体阶段能证明自己满足更强的一致性边界。

#### 1.3.2 `TaskScope`

表示某次任务在业务语义上“是对谁做事”。

它回答的核心问题是：

- 这次任务作用于哪个业务对象
- 哪些上层业务上下文会把这次任务视为相关任务
- 哪些暂停任务在同一业务上下文里应被视为同一类继续候选

它不等同于任务参数 JSON，也不负责表达当前执行器占用关系。

首轮建议把 `TaskScope` 固定为一组稳定字段，而不是隐含地埋在 `params` 里。更合适的方向是：

- `taskScopeType`
  - 作用对象类型
- `taskScopeKey`
  - 作用对象稳定键
- `taskScopeSnapshot`
  - 可选，给前端与调试使用的结构化对象快照

其中：

- `taskScopeKey` 用于稳定标识“这次任务作用于哪个业务对象”
- `taskScopeSnapshot` 用于保留便于展示和调试的上下文
- `taskScope` 负责稳定表达任务与业务对象的归属关系

首轮运行实例唯一性按 `taskType` 约束，而不是按 `taskScope` 约束。但这里的“唯一性”直接约束当前占用该 `taskType` 创建槽位的运行实例。也就是说，同一 `taskType` 下，同一时刻最多只允许存在一个占用创建槽位的 `pending / running / pausing / paused / resuming / canceling` 运行实例。

`paused` 不再处于执行中，但仍然落入同类型任务的创建槽位约束。也就是说：

- 进入 `paused` 后，任务不再推进执行
- 在它被继续、取消或收束前，不能再创建新的同类型运行实例
- 这个规则对 `durable` 与 `session_bound` 一视同仁

`taskScope` 仍然只负责回答“这个运行实例属于哪个业务上下文”。因此，前端接续时不能拿 `taskScope` 去推导是否存在多个正在占槽的并行运行实例；并行性约束仍先由 `taskType` 级单槽规则收掉。

这里还要把首轮页面模型的边界写清楚：

- `taskScope` 负责说明“任务属于哪个业务对象”
- 它不负责决定“页面当前主任务是谁”
- 对同一个任务页面来说，当前主任务由“当前 `activeScope` + 页面查询到的任务候选”共同决定
- `activeScope` 不只是展示过滤条件
  - 它直接参与页面当前主任务的选择
- 因此页面切换 `activeScope` 时，应重新计算当前主任务

但首轮还需要补一个显式边界，避免“能不能创建”和“当前页面能不能看到阻塞源”脱节：

- `taskType` 级创建槽位可以阻塞不同 `taskScope` 之间的新建请求
- 因此阻塞关系本身必须对上层可见，而不能只隐含在“创建时返回冲突”里
- 框架应把“当前是谁占用了该 `taskType` 的创建槽位”作为显式可查询事实暴露出来
- 该事实至少需要包含：
  - `taskRunId`
  - `taskType`
  - `status`
  - `taskScopeType`
  - `taskScopeKey`
  - 必要的 `taskScopeSnapshot`

这样页面或控制面即使当前 `activeScope` 与阻塞实例不匹配，也能明确知道：

- 当前不是“没有任务”
- 而是“有别的 scope 的同类型任务正在占位”
- 当前不能新建的原因是什么

这样可以避免同一业务上下文下积累多个暂停运行实例，导致继续入口、恢复候选和当前责任边界失去确定性。

`taskScope` 是业务归属键，而不是页面标识；`taskType` 只能回答“这是什么任务类别”。即使首轮串行约束按 `taskType` 建模，页面也不能只凭 `taskType` 自动接管任务；页面是否接续任务，应由页面当前解析出的 `activeScope` 与任务的 `taskScope` 做匹配判断。

这里要明确区分两层不同语义：

- `taskScope` 是业务标识，用来回答“这是谁的任务”
- `taskType` 是首轮串行类别，用来回答“这类任务能不能并发运行”

现有 `publish / reanchor` 的经验可以直接视作一个示例：

- `taskScopeType = publish_stream`
- `taskScopeKey = publishTarget + environment + publishType`
- `taskScopeSnapshot = { publishTarget, environment, publishType }`

#### 1.3.3 `TaskRun`

表示一次具体任务运行实例。

它是前后端交互的主对象，也是数据库中的核心权威记录。任务状态、参数、运行时间、错误信息、恢复能力、当前阶段定位等都围绕它组织。

它回答的核心问题是：

- 这次任务现在处于什么整体运行状态
- 当前推进到哪个阶段
- 最近一次错误和心跳是什么

它承载任务级当前控制意图的快照，但不负责保存细粒度不可变控制历史。

这里还需要固定一个容易被忽略但非常关键的边界：

- `TaskRun` 是逻辑运行实例
- `TaskRun` 不等同于某一段页面会话

同一个运行实例可以经历：

- 首次启动
- 暂停后的继续
- 清理流程收束

这些都仍然属于同一个 `TaskRun` 生命周期。

#### 1.3.4 `TaskStage`

表示某次任务运行中的阶段快照。

每个阶段至少应具备：

- `stageKey`
- 阶段总量
- 当前已完成量

前端基于它自行计算阶段百分比，显示阶段推进位置，并结合任务开始时间展示整体已用时间。

`TaskStage` 既是领域对象，也预期会在数据库中存在对应持久化结构。

### 1.4 首轮固定的运行原则

#### 1.4.1 任务推进方式

- 执行器按“任务 -> 阶段 -> 原子块”推进
- `bounded` 阶段必须能提供明确总量
- `unbound / simple` 阶段不要求提供 `total`
- 每个原子块结束后刷新进度、检查当前控制意图并刷新心跳
- 每次会影响前端可见状态的提交，都必须原子更新对应的 `TaskRun` 与 `TaskStage`

#### 1.4.2 控制语义

- `pause`
- `resume`
- `cancel`
- `retry`

其中：

- `retry` 默认表示创建一个新的运行实例，并从头重新执行
- `resume` 仅适用于声明支持继续的任务类型

#### 1.4.3 串行语义

- 首轮同一 `taskType` 下，占用类型级创建槽位的运行实例保持唯一
- 这不是临时实现细节，而是首轮重型任务框架的产品级约束
- 首轮有意让 `taskType` 同时承担“任务类别标识”和“串行控制键”这两层语义
- 这不是待拆分的设计债，也不是为了将来必然演进而刻意保留的半成品抽象
- 后续 review 不应把“`taskType` 尚未与并发资源键解耦”默认视为缺陷；在当前单人运维应用边界内，这是明确选择的简化模型
- 这条约束同时服务于前端页面心智模型：
  - `publish` 页面只承载一个当前 `publish` 任务
  - `reanchor` 页面只承载一个当前 `reanchor` 任务
  - 页面切换 `taskScope` 不会直接把主任务切换成别的运行实例
- 首轮不提供同类重型任务的排队或 backlog 能力
- 类型级创建槽位唯一性不能只停留在应用层检查，必须由数据库约束强制保证
- 当前执行层只按“单执行器 + 单线程推进”建模
- 不为当前背景额外引入独立的执行权对象、租约对象或多执行器竞争协议

#### 1.4.4 恢复语义

- 页面重新进入时，应自动发现当前运行任务并接续显示
- 后端重启后不自动续跑
- 需要有统一的残留运行态清理机制

## 第二部分：生命周期与状态机

第二部分开始进入行为模型，用来固定：

- 任务运行状态有哪些
- 哪些状态只是“未终态”
- 哪些状态真正表示“正在执行”
- 控制意图如何驱动状态迁移
- 哪些终态来自正常执行，哪些终态来自残留清理

### 2.1 状态集合

首轮建议把 `TaskRun` 固定为以下状态：

- `pending`
  - 任务已创建，但尚未开始执行
  - 属于可被调度层重复发现并重复尝试启动的等待态
- `running`
  - 执行器正在推进阶段
- `pausing`
  - 已收到暂停请求，等待当前原子块结束
- `paused`
  - 已在安全边界停下，尚未发起恢复尝试，可等待继续或取消
- `resuming`
  - 已收到继续请求，正在发起恢复尝试
  - 恢复尝试包括恢复前提校验与重新进入执行循环
  - 属于纯调度态，此时尚未重新进入执行循环
  - 属于可被调度层重复发现并重复尝试恢复的等待态
- `canceling`
  - 已收到取消请求，等待当前原子块结束
- `canceled`
  - 任务已被协作式终止
- `failed`
  - 任务已被系统明确判定为失败
  - 包括执行期失败、恢复尝试失败，以及等待态无法继续推进后的失败收束
- `completed`
  - 任务已成功完成
- `abandoned`
  - 数据库中残留的运行态在确认执行器已不存在后，被清理流程收束成的终态

其中应明确拆分三组概念：

- `pending / running / pausing / paused / resuming / canceling` 视为未终态
- `running / pausing / canceling` 视为执行中状态
- `canceled / failed / completed / abandoned` 视为终态

这里不再把“当前活跃任务”作为单一概念使用，原因是：

- `pending` 表示任务尚未结束，但并未开始执行
- `paused` 表示任务尚未结束，但也不再持有执行权
- `resuming` 表示任务请求继续，但尚未重新拿到执行权
- 只有执行中状态才表示任务正在真实占用运行资源

因此，后文涉及前端自动接管、执行占用、残留运行态清理时，都应以“执行中状态”或更明确的专用集合为准，而不是笼统复用“活跃态”一词。

`pending` 只表示当前唯一占槽任务已经创建，但尚未被调度层真正启动。它不是同类任务排队队列中的一个候选，也不是残留运行态，不要求存在有效心跳。

但这里要补一个首轮硬边界：

- `pending / resuming` 虽然不属于执行中状态
- 但它们不是被动停放态，而是调度层负责持续尝试启动或恢复的等待态
- 它们不参加“残留运行态 -> abandoned”清理语义
- 但必须参加独立的调度巡检与失败收束语义

也就是说，一个 `pending` 或 `resuming` 长时间停留，不应直接按“执行器残留”解释；但如果调度层持续无法把它推进到真正执行，也不能无限期占用同类型创建槽位。首轮要求框架通过调度巡检与必要的失败收束管理这些状态，而不是让它们永远停留在数据库中等待人工发现。

首轮的类型级单槽约束，默认约束 `pending / running / pausing / paused / resuming / canceling`。`paused` 虽然不再占用执行权，但仍属于同类型任务的强占位对象；在其被继续、取消或自动收束前，不允许再创建新的同类型运行实例。这样可以把 `pause` 语义固定为“任务安全停下，等待当前运行实例继续或终止”，而不是“让出执行权后允许同类型新任务插队”。

这里再补一条与当前产品边界直接相关的首轮结论：

- 在当前“单服务、个人电脑、手动触发、低频运维任务”的前提下
- `paused` 长时间保留创建槽位不是首轮要自动消除的系统风险
- 更合适的首轮语义是：
  - 由操作者自己决定是继续还是取消
  - 框架不对 `durable paused` 默认增加自动过期回收
  - UI 必须持续暴露该暂停实例及其显式 `cancel` 入口
  - 这是一种有意保留的非自愈阻塞态
  - 框架不会试图在后台自动替操作者决定“应继续还是应放弃”
  - 该阻塞态的解除依赖显式人工操作或后续恢复尝试失败后的收束

也就是说，`paused` 在首轮被视为一种明确的人工运维挂起态，而不是必须被后台自动回收的异常态。

### 2.2 状态迁移规则

本节状态迁移表是首轮唯一权威的 `TaskRun.status` 迁移集合。后文所有控制意图、巡检、恢复与清理规则，都只能落在这里显式允许的边上。

首轮固定以下迁移方向：

- `pending -> running`
- `pending -> failed`
- `pending -> canceled`
- `running -> pausing -> paused`
- `pausing -> canceling`
- `pausing -> failed`
- `running -> canceling -> canceled`
- `canceling -> failed`
- `running -> abandoned`
- `pausing -> abandoned`
- `canceling -> abandoned`
- `running -> failed`
- `running -> completed`
- `paused -> canceled`
- `paused -> resuming -> running`
- `resuming -> failed`
- `resuming -> canceled`

首轮不允许以下语义：

- `failed -> running`
- `failed -> pending`
- `completed -> running`
- `canceled -> running`

也就是说，失败或终止后的“重试”不应通过原地复活旧运行实例实现，而应创建新的 `TaskRun`。

这里还要明确一条此前容易混淆的边界：

- `paused -> resuming -> running` 只表示“已在安全边界停下的同一运行实例，重新竞争到执行权后继续推进”
- 它不用于表达“执行器异常消失后自动把原运行实例重新拉起”
- 首轮中，一旦运行中的执行器异常退出或失去执行上下文，该次连续执行期不会被系统自动视为同一运行实例的隐式继续入口
- 后续只允许通过：
  - 清理流程收束为 `abandoned`
  - 或在明确进入 `paused` 后再由用户显式触发 `resume`
  来继续管理该运行实例
- 对 `pausing / canceling`，首轮再固定一条失败优先级规则：
  - 已受理的控制意图只在当前原子块成功完成后才能落成目标终态
  - 若该原子块本身失败，则该次运行优先收束为 `failed`
  - 不允许把已经失败的块结果再解释为成功暂停或成功取消
- 若 `pause` 已受理、任务处于 `pausing`，而后续又收到合法 `cancel`
  - 则应走 `pausing -> canceling`
  - 这条边只表示控制意图从“暂停”切换为“取消”
  - 不表示任务已经立即终止
- `abandoned` 只允许由清理流程触发
- 不允许作为普通控制动作或正常执行路径主动写入的目标终态
- 首轮唯一允许保留同一 `TaskRun` 继续语义的前提，是任务先明确进入 `paused`
- 对命中 `session_bound` 契约的暂停边界，一旦继续所需的进程内上下文丢失，任务不得继续停留在 `paused`
- 若此时尚未发起 `resume` 尝试，系统应自动把该运行实例收束为 `canceled`
- 若已经进入 `resuming`，则应按恢复尝试失败收束为 `failed`

### 2.3 控制意图与状态的关系

首轮固定以下原则：

- 用户发出的不是“直接终态”，而是“控制意图”
- 当前开放控制意图直接写入 `task_runs`
- 执行器或调度层读取当前控制意图后，在合法边界推进状态迁移

这意味着：

- 点击暂停时：
  - 先写入 `pause` 控制意图
  - 任务进入 `pausing`
  - 执行器在安全边界落成 `paused`
- 点击取消时：
  - 若任务仍处于 `pending`，由控制层直接收束为 `canceled`
  - 若任务已处于执行循环中，则把当前控制意图改写为 `cancel`
  - 对 `running / pausing`，任务进入 `canceling`
  - 执行中的任务由执行器在安全边界落成 `canceled`
  - 对 `paused / resuming`，由控制层或调度层直接收束为 `canceled`
- 点击继续时：
  - 先写入 `resume` 控制意图
  - 只有 `paused` 且任务定义声明支持继续时，才允许进入 `resuming`
  - 调度层完成恢复前提校验并成功唤起执行器后，再进入 `running`

这里需要把控制意图与状态之间的不变量写得更硬一些：

- 若 `TaskRun.status = pausing`，则 `controlRequestKind` 必须为 `pause`
- 若 `TaskRun.status = resuming`，则 `controlRequestKind` 必须为 `resume`
- 若 `TaskRun.status = canceling`，则 `controlRequestKind` 必须为 `cancel`
- 若 `TaskRun.status` 为 `pending / running / paused / canceled / failed / completed / abandoned`：
  - 不得依赖旧中间态语义解释控制意图
  - 若 `controlRequestKind` 已不再需要继续驱动状态迁移，应及时清空
- 若恢复或巡检发现上述不变量被破坏，必须将该运行实例判定为状态不一致，并由框架收束或修复，而不是交给前端猜测

这样可以明确区分两类状态：

- 控制意图已被受理
- 任务已真正停下或终止

### 2.4 残留运行态语义

`abandoned` 不属于正常业务成功或失败语义，而属于运行态清理语义。

它表示：

- 数据库中曾经存在执行中任务
- 但执行器进程或心跳已无法证明任务仍在运行
- 系统通过启动清理或巡检清理，将其收束为明确终态

这样可以避免把“用户主动取消”和“进程崩溃后残留未完成任务”混成同一种含义。

## 第三部分：数据模型与表结构草案

第三部分的目标不是一次把所有审计、事件流和历史细节都建齐，而是先提供一套最小可用的数据模型，满足以下硬要求：

- 数据库中存在权威快照
- 对 `session_bound` 暂停边界，框架额外维护受约束的内存恢复上下文
- 前端可以读取当前任务快照
- 执行器可以读取当前控制意图
- 同类型任务可以做串行单槽控制
- 支持取消、重试，以及由任务类型显式声明的暂停与继续
- 支持页面重新进入后的状态接续
- 支持残留运行态清理

### 3.1 首轮表结构选择

首轮建议固定为两张核心表：

- `task_runs`
- `task_stages`

首轮不单独引入 `task_events` 表。

原因是当前最关键的是先把“当前权威快照”和“控制面”做正确；首轮把当前控制意图直接收敛到 `task_runs`，不再额外引入独立命令表。如果一开始就把完整控制命令审计和事件溯源都引入，会明显放大复杂度。实时通道仍然可以存在，但首轮把它视为数据库状态变化的投递层，而不是独立真相来源。

### 3.2 `task_runs`

`task_runs` 是主表，表达一次任务运行实例的整体状态。

建议至少包含以下字段：

- `id`
  - 任务运行实例 ID
- `taskType`
  - 任务类型，例如 `hsdata_publish`
- `definitionVersion`
  - 标识该运行实例绑定的任务定义版本
- `taskScopeType`
  - 任务作用对象类型
- `taskScopeKey`
  - 任务作用对象稳定键
- `taskScopeSnapshot`
  - 可选，任务作用对象快照，JSON
- `status`
  - 对应第二部分定义的状态集合
- `params`
  - 任务参数，JSON
- `supportsResume`
  - 该次运行是否存在可暂停并继续的阶段
- `currentStageKey`
  - 当前阶段标识，可为空
- `currentStageIndex`
  - 当前阶段序号，可为空
- `currentResumeMode`
  - 当前阶段的恢复契约快照：`none / durable / session_bound`
- `pausedResumeMode`
  - 若当前处于 `paused / resuming`，记录本次暂停边界命中的恢复契约；否则可为空
- `selectionAnchor`
  - 可选，运行级候选宇宙锚点
  - 仅作为从阶段级恢复事实派生出来的运行级缓存摘要
- `resumeToken`
  - 可选，任务级恢复点缓存摘要
- `runtimeBootId`
  - 若本次暂停边界命中 `session_bound`，记录服务本次启动的唯一标识；否则可为空
- `resumeContextKey`
  - 若本次暂停边界命中 `session_bound`，记录进程内恢复上下文键；否则可为空
- `runRevision`
  - 当前任务快照版本号，单调递增
- `controlRequestKind`
  - 当前开放控制意图：`pause / resume / cancel`，可为空
- `heartbeatAt`
  - 最近心跳时间
- `startedAt`
  - 真正开始执行时间
  - 首次 `pending -> running` 成功时必须写入
  - 后续 `resume` 不得改写，必须保留首次真正进入执行循环的时间
- `finishedAt`
  - 终止或完成时间
- `errorCode`
  - 失败码，可为空
- `errorMessage`
  - 失败信息，可为空
- `terminalReason`
  - 终态原因；用于统一表达 `canceled / failed / abandoned` 的结束原因
  - `completed` 时可为空，其余终态必须可稳定给出原因码
- `retryOfTaskRunId`
  - 若这是一次重试运行，指向原任务
- `createdAt`
- `updatedAt`

首轮约束建议如下：

- 运行实例必须显式记录 `taskScopeType` 与 `taskScopeKey`
- `taskScopeSnapshot` 只用于展示、调试与辅助恢复页面上下文，不应成为唯一匹配依据
- 类型级创建槽位唯一性仍由 `task_runs` 自身承载
  - 首轮应通过数据库层的部分唯一约束或等价机制强制保证
- `params` 固定为输入快照，不允许任务执行过程中被修改
- `task_runs` 负责保存任务级当前快照与阶段指针
  - 不重复保存当前阶段自己的进度模式与数值事实
- `task_stages` 是阶段级事实源
  - `task_runs.currentStageKey / currentStageIndex / currentResumeMode` 只是为快照接口、筛选和调试准备的运行级派生快照
  - 若恢复、巡检或人工修复时发现两者不一致，应优先以 `task_stages` 重建 `task_runs` 的阶段快照
- `currentStageKey / currentStageIndex / currentResumeMode` 只表达“已 durable 进入的当前阶段”
  - `running / pausing / canceling`
    - 指向当前正在执行的阶段
  - `paused / resuming`
    - 指向暂停时停下、且恢复后准备重新进入的阶段
  - `failed / canceled / completed / abandoned`
    - 这三个字段必须为空
- `pending`
  - 不得把“计划进入但尚未 durable 开始”的下一阶段写进 `currentStage*`
  - 若页面需要知道下一计划阶段，应使用独立的派生读模型，而不是复用 `currentStage*`
- 每次影响前端可见状态的提交，都必须让 `runRevision` 单调递增
- `task_runs` 与 `task_stages` 中会影响当前快照可见结果的更新，应在同一事务内提交
- `supportsResume` 是运行实例级摘要字段
  - 它只回答“这个任务是否存在可暂停并继续的阶段”
- `selectionAnchor` 是恢复语义的一部分
  - 它不回答“从哪里继续”
  - 它回答“继续的仍然是不是原来那一批候选对象”
  - 对支持 `resume` 的任务或阶段，若候选依据会随时间漂移，则必须存在稳定可解释的 `selectionAnchor`
- `currentResumeMode` 与 `pausedResumeMode` 是运行实例级快照字段
  - 它们只反映当前阶段或当前暂停边界命中的恢复契约
  - 不应被解释为整个任务天然属于某一种恢复方式
- `currentResumeMode` 允许为空
  - 为空表示当前不存在已 durable 进入的当前阶段
- 当前存在已 durable 进入的当前阶段，且该阶段不支持继续时，`currentResumeMode` 必须为 `none`
- 当前阶段若允许继续，则 `currentResumeMode` 必须为 `durable` 或 `session_bound`
- 进入 `paused / resuming` 时，必须把命中的阶段恢复契约固化到 `pausedResumeMode`
- `runtimeBootId` 与 `resumeContextKey` 只在 `pausedResumeMode = session_bound` 时必填
- `pausedResumeMode != session_bound` 时，`runtimeBootId` 与 `resumeContextKey` 必须为空
- 除 `paused / resuming` 外，`pausedResumeMode / runtimeBootId / resumeContextKey` 必须为空
- `resuming -> running` 成功提交时，必须同事务清空：
  - `pausedResumeMode`
  - `runtimeBootId`
  - `resumeContextKey`
- 任务收束为任一终态时，也必须同事务清空：
  - `pausedResumeMode`
  - `runtimeBootId`
  - `resumeContextKey`
- `terminalReason` 不应只是自由文本
  - 首轮至少应收敛为“稳定错误码 + 可选补充文本”的结构
  - 不能让前端、日志和运维只靠自然语言字符串做分支判断
- `completed` 之外的所有终态都必须写出 `terminalReason`
  - 包括 `canceled / failed / abandoned`
  - 首轮不允许把这三类终态落成“状态已知，但原因为空”
- `task_runs` 负责保存运行实例的当前快照，也负责保存首轮唯一的当前控制意图
- `controlRequestKind` 只表达“当前还在等待落成的控制意图”
  - 不表达完整控制历史
  - 不应用来替代终态原因
- 创建新 `TaskRun` 时，必须通过数据库约束保证同一 `taskType` 下不存在其他占用类型级创建槽位的运行实例
- 该校验代表首轮单槽模型约束，而不是可选优化
- 首轮应按以下规则判断是否占用该槽位：
  - `pending / running / pausing / paused / resuming / canceling`
    - 始终占用
- 更具体地说，首轮应要求数据库在 `task_runs(taskType)` 上对上述占位状态建立部分唯一约束，或提供等价的单槽约束机制
- 因此“当前是谁在占位”首先是 `task_runs` 上的受约束事实，而不是控制层临时扫描后得出的软判断
- 若需要重新开始该任务类型的新一轮运行，必须通过：
  - 对现有实例执行取消
  - 或对终态实例执行重试
  - 或等待当前占槽实例结束
  - 而不是绕过这条占槽规则直接再插入新的冲突运行实例

### 3.3 `task_stages`

`task_stages` 表达某次运行中的阶段快照。

首轮固定要求：每次任务运行创建时就预先建立完整阶段骨架。这样前端、调试和恢复重建都能依赖稳定的完整阶段序列，不再允许“进入阶段时再惰性创建”作为首轮实现分支。

`task_stages` 只表达阶段级状态，不承载任务总体生命周期状态；任务总体生命周期仍只由 `task_runs.status` 表达。

建议字段：

- `taskRunId`
- `stageKey`
- `stageIndex`
- `status`
  - 可取 `pending / running / paused / canceled / failed / completed / abandoned`
- `label`
  - 给前端展示的阶段名，或可由前端映射
- `progressMode`
  - `bounded / unbound / simple`
- `resumeMode`
  - 该阶段在可暂停边界上采用的恢复契约：`none / durable / session_bound`
- `total`
  - 阶段总量；仅 `bounded` 阶段必填
- `done`
  - 当前已完成量；`bounded` 与 `unbound` 阶段必填
- `resumeToken`
  - 阶段级恢复点，可为空
- `selectionAnchor`
  - 可选，阶段级候选宇宙锚点
  - 用于冻结该阶段在 pause / resume 之间所面对的候选集合边界
- `startedAt`
- `finishedAt`
- `updatedAt`

首轮约束建议：

- `bounded` 阶段必须在真正开始执行前确定 `total`
- `bounded` 阶段的 `total` 必须来自可冻结、可重复解释的输入事实
  - 不允许直接依赖会持续漂移的即时查询视图
  - 若候选集合会随时间变化，则必须先通过稳定锚点、快照或等价边界把它冻结后，才能把该阶段声明为 `bounded`
- 若 `bounded` 阶段的 `total` 依赖候选集合冻结边界：
  - 则用于解释该 `total` 的 `selectionAnchor` 必须与该 `total` 一起 durable 落库
  - 两者必须属于同一次原子事实提交
- 对已经进入 `paused` 的 `bounded` 阶段，`resume` 时必须复用暂停前已 durable 写入的 `total`
  - 不允许在 `resume` 路径上重新计算并覆盖 `total`
  - 否则会破坏 `done / total` 的可比性，以及暂停前后对同一阶段进度的稳定解释
- 若 `paused bounded` 阶段在恢复前发现：
  - `total` 缺失
  - `total <= 0`
  - `done > total`
  - 或等价的阶段进度事实损坏
  则本次 `resume` 尝试不得继续执行
  - 必须直接收束为 `failed`
  - 并写入稳定 `terminalReason = resume_incompatible`
  - 若后端在暂停态读取快照时就已经能判定上述损坏事实成立：
    - 则该实例不应继续停留在“可继续暂停”语义下占槽
    - 框架应直接将其收束为 `failed`
    - 并统一写入 `terminalReason = resume_incompatible`
    - 在该收束完成前，`canResume` 必须折叠为 `false`
- 对 `bounded` 阶段，“切换到该阶段”与“该阶段的 `total` 已 durable 可见”必须作为同一原子状态切换成立
- `bounded` 阶段的 `done` 不得大于 `total`
- `unbound` 阶段可以没有 `total`，但必须持续推进 `done`
- `simple` 阶段允许同时省略 `done` 与 `total`，但应只用于初始化、收尾、总结报告等极短阶段
- `resumeMode` 是阶段契约的一部分
  - 它描述当前阶段若停在可恢复边界上，应按什么前提继续
  - 不应用 `task_runs` 的任务级静态字段替代它
- `selectionAnchor` 与 `resumeToken` 不能混用为同一概念
  - `resumeToken` 表达“从哪里继续”
  - `selectionAnchor` 表达“继续的还是不是原来那一批候选对象”
- 对恢复语义本身，首轮固定单一事实源：
  - `task_stages.resumeToken` 与 `task_stages.selectionAnchor` 是权威恢复事实
  - `task_runs.resumeToken` 与 `task_runs.selectionAnchor` 只作为运行级缓存摘要
  - 若恢复、巡检或人工修复时发现两层不一致，应优先以 `task_stages` 重建 `task_runs`
- `progressMode` 只保留在 `task_stages`
  - `task_runs` 不重复保存这类阶段局部事实
- 前端的阶段百分比与阶段内 ETA 仅对 `bounded` 阶段基于 `done / total` 自行计算
- `unbound` 阶段不承诺百分比与 ETA，但仍必须展示明确的已完成量
- 当需要从数据库重建当前阶段快照时，应优先以 `task_stages` 的事实状态和顺序推导当前阶段
- 对未终态 `TaskRun`，`task_stages` 必须满足“单义可重建”的序列不变量
  - 对 `pending`
    - 所有阶段保持 `pending`
    - 不存在当前活跃阶段
  - 对 `running / pausing / paused / resuming / canceling`
    - 同一时刻至多只允许存在一个当前活跃阶段
    - 当前活跃阶段只允许表现为：
      - `running`
      - `paused`
      - 或在 `resuming` 成功提交前保留其暂停前最后一个稳定状态
    - 阶段序列必须保持：
      - 前缀阶段已终结
      - 单个当前阶段活跃
      - 后缀阶段尚未开始
  - 不允许出现两个并列“当前阶段”同时可被解释为活跃的状态组合
- 当 `paused / resuming` 指向某个暂停阶段时：
  - `task_runs.currentStageKey / currentStageIndex / currentResumeMode / pausedResumeMode`
    必须与该唯一可恢复阶段保持一致
  - 若恢复、巡检或人工修复时发现这些字段无法与唯一阶段单义对应：
    - 必须将其判定为状态不一致
    - 由框架收束或修复，而不是交给前端自行猜测
- `task_stages.status` 不复制 `TaskRun` 的请求中间态
  - `pausing / resuming / canceling` 只存在于 `TaskRun.status`
  - 当前阶段在这些中间态期间的阶段状态固定按以下规则解释：
    - `TaskRun.status = pausing`
      - 当前阶段保持 `running`
    - `TaskRun.status = resuming`
      - 当前阶段保持上一次已落库的稳定阶段状态
      - 若暂停前该阶段已停在可恢复边界，则通常表现为 `paused`
    - `resuming -> running` 的认领提交一旦成功：
      - 当前恢复阶段必须在同一提交中同步回到 `running`
      - 不允许出现 `TaskRun.status = running` 而当前 `TaskStage.status` 仍停留在 `paused` 的对外可见窗口
    - `TaskRun.status = canceling`
      - 当前阶段保持 `running`
- 若终止发生在 `resuming` 且尚未重新进入阶段执行：
  - 当前阶段保持其暂停前最后一次稳定状态
  - 通常表现为 `paused`
  - 不把这类失败或取消改写成阶段级 `failed / canceled`
  - 这条规则优先级高于后文普通 `failed / canceled` 的阶段终止回写规则
- 也就是说，前端若需要展示“正在暂停 / 正在继续 / 正在取消”
  - 必须读取 `TaskRun.status`
  - 不能试图从 `TaskStage.status` 反推这些中间态

### 3.4 当前控制意图放在哪里

首轮把当前控制意图直接收敛到 `task_runs`，不再单独引入 `task_commands`。

原因是：

- 当前产品边界下，同一任务同一时刻只需要一个开放控制意图
- 首轮控制动作集合是封闭集合，只包含 `pause / resume / cancel`
- 当前前提是单 `app`、单 `service`、低频手动触发、单执行器
- 若再单独引入命令表、命令序号、意图覆盖与处理状态，会明显放大复杂度

首轮规则建议：

- `controlRequestKind` 是首轮唯一的开放控制意图字段
- 同一时刻最多只允许存在一个开放控制意图
- `resume` 只能作用于 `paused`
- `resume` 被受理后，应先把任务推进到 `resuming`
- `pause` 只对 `running` 生效
- `cancel` 对 `pending / running / pausing / paused / resuming` 都可生效
  - `pending` 上的 `cancel` 由控制层直接收束为 `canceled`
  - `running / pausing` 上的 `cancel` 由执行器按协作式终止语义收束
  - `paused / resuming` 上的 `cancel` 由控制层或调度层直接收束
- 当控制意图真正落成后，应同步清空 `controlRequestKind`
- `TaskRun.status` 的 `pausing / resuming / canceling` 负责表达任务当前对外可见的运行中间态
- `controlRequestKind` 只负责表达“当前正在等待落成的控制意图”
- 若启动恢复、巡检清理或人工修复时发现：
  - 中间态存在但 `controlRequestKind` 为空
  - `controlRequestKind` 存在但 `TaskRun.status` 不匹配
  则应将该运行实例判定为状态不一致，并由框架收束或修复

这套规则的核心不是“控制动作可以排队”，而是：

- 同一时刻只保留一个当前有效控制意图
- 只有更强且仍然合法的意图，才允许覆盖旧意图
- `TaskRun.status` 与 `controlRequestKind` 共同表达首轮控制面真相

首轮还应把“可覆盖”规则收成明确口径，避免不同实现各自理解“更强”：

- `pause`
  - 只允许在 `running` 上新建
  - 不允许覆盖任何已有开放控制意图
- `resume`
  - 只允许在 `paused` 上新建
  - 不允许覆盖任何已有开放控制意图
- `cancel`
  - 允许在 `pending / running / pausing / paused / resuming` 上新建或受理
  - 若当前开放控制意图为 `pause`，且任务已进入 `pausing`：
    - 允许用 `cancel` 覆盖为更强意图
    - 任务转入 `canceling`
  - 除上述 `pausing -> canceling` 外，首轮不允许再定义其他控制意图覆盖关系
- 重复点击同类控制动作：
  - 若与当前开放控制意图相同，则应按幂等受理或直接返回当前最新结果
  - 不应再生成新的覆盖语义

### 3.5 当前快照与历史的边界

首轮固定以下边界：

- `task_runs` 保存任务级当前快照
- `task_stages` 保存阶段级当前快照和简要历史
- 首轮不单独保存不可变控制命令审计

首轮暂不单独引入：

- 全量进度事件表
- 细粒度原子块历史表
- 实时事件补读日志表
- 独立控制动作审计表

这些都可以作为后续增强项，但不应阻塞首轮框架成立。

### 3.6 恢复点放在哪里

首轮固定单一权威模型：

- `task_stages.resumeToken`
  - 是阶段级恢复点的唯一权威事实
- `task_stages.selectionAnchor`
  - 是阶段级候选集合锚点的唯一权威事实
- `task_runs.resumeToken`
  - 仅作为当前恢复阶段的运行级缓存摘要
- `task_runs.selectionAnchor`
  - 仅作为当前恢复阶段的运行级缓存摘要

也就是说，首轮不再允许“有些任务只以任务级恢复点为权威、有些任务只以阶段级恢复点为权威”这种双轨模型。只要某个阶段支持 `resume`，就必须能在对应 `task_stages` 行里回答：

- 从哪里继续
- 继续的仍然是不是原来那一批候选对象

但首轮应固定一个原则：

- 恢复点必须是“可由代码解释的稳定结构”
- 不允许把不可控的临时内存对象序列化进去
- 恢复点应尽量只表达：
  - 当前阶段
  - 当前游标
  - 已确认完成的块边界
  - 必要的校验锚点
- 恢复点必须与 `taskType + definitionVersion` 一起解释
- `resumeToken` 不应单独承担“冻结候选集合边界”的职责
  - “从哪里继续”与“继续的还是不是原来那一批候选对象”必须分开表达
- `selectionAnchor` 必须是“可由代码解释的稳定结构”
  - 它可以是 source tag、baseline version、change-log upper bound、snapshot watermark 或等价锚点
  - 它必须足以冻结 pause 前后面对的候选宇宙
- 对支持 `resume` 的任务，如果当前阶段的候选依据会随时间漂移，而任务又无法稳定保存 `selectionAnchor`，则该阶段不应宣称支持 `resume`
- 对 `durable` 恢复方式，如果当前代码无法兼容解释该版本的恢复点：
  - 在 `resume` 尝试期间，应把该运行实例收束为 `failed`
  - 并通过 `terminalReason` 明确标记为恢复契约不兼容
- 对 `session_bound` 恢复方式，若继续所需的进程内上下文丢失：
  - 在 `resume` 尝试期间，应把该运行实例收束为 `failed`
  - 在尚未发起 `resume` 尝试前的启动清理或巡检清理中，允许直接收束为 `canceled`
  - 并通过 `terminalReason` 明确标记为恢复上下文丢失
- 对 `session_bound`，这里的“进程内上下文”固定指当前服务本次启动期间保留的服务内状态
  - 启动会话切换后，不允许用新的启动会话去解释旧的 `resumeContextKey`
- 这两种情况下都允许用户重试，但不允许框架静默套用新定义继续执行旧恢复点

### 3.7 页面接续依赖哪些字段

前端页面重新进入时，至少需要依赖以下字段恢复展示：

- 页面先解析自己当前选中的 `activeScope`
- 再用 `activeScope` 与任务的 `taskScopeType + taskScopeKey` 做匹配判断

从 `task_runs` 读取：

- `id`
- `taskType`
- `taskScopeType`
- `taskScopeKey`
- `taskScopeSnapshot`
- `status`
- `currentStageKey`
- `currentStageIndex`
- `supportsResume`
- `currentResumeMode`
- `pausedResumeMode`
- `startedAt`
- `finishedAt`
- `errorMessage`

从 `task_stages` 读取：

- 当前阶段的 `total`
- 当前阶段的 `done`
- 所有阶段的顺序与状态

这样前端即使错过实时事件，也能只靠数据库快照恢复主界面状态。

这里再固定一个读模型边界，避免把恢复校验字段误暴露成前端主状态字段：

- `selectionAnchor`
  - 属于恢复语义与后端校验语义
  - 默认不进入前端页面主读模型
- 前端主页面是否展示“可以发起继续尝试”
  - 由后端折叠成 `canResume`
  - `canResume = true` 在首轮只表示：
    - 当前存在一个允许用户发起 `resume` 请求的暂停实例
    - 不表示系统已经预先证明这次恢复尝试一定会成功
  - 对 `durable paused`
    - `canResume` 只基于状态机合法性与产品权限判断
    - 真正的恢复兼容性校验仍发生在点击 `resume` 之后
  - 对 `session_bound paused`
    - 后端还必须额外确认当前启动会话与进程内恢复上下文仍然存在
    - 只有这类前提仍成立时，才允许返回 `canResume = true`
- 只有在调试、运维或专门诊断视图中
  - 才建议额外暴露 `selectionAnchor` 或其可读摘要

也就是说，`selectionAnchor` 虽然是恢复语义的关键字段，但它不是首轮页面主交互需要直接消费的事实。

### 3.8 首轮不做的建模

为了避免首轮过重，第三部分暂不把以下能力写成必须项：

- 跨任务依赖 DAG
- 子任务树
- 每个原子块的完整审计历史
- 通用事件溯源
- 多租户/多命名空间任务隔离
- 动态数据库驱动的任务定义中心

### 3.9 这一版表结构的取舍结论

这套数据模型的核心取舍是：

- 先把“当前权威快照”建稳
- 只把 `session_bound` 所需的内存恢复上下文作为受约束补充，而不是把所有运行态都推回进程内
- 先把“暂停 / 继续 / 取消 / 串行单槽”建清楚
- 先让前端能靠快照恢复
- 不在首轮引入过重的事件溯源模型

## 第四部分：执行器契约与任务接入模型

第四部分的目标是固定一件事：

- 具体任务怎样接入这套框架，而不是每个任务自己发明运行方式

### 4.1 首轮接入方式

每种重量级任务都要在代码里注册一个 `TaskDefinition`。

它至少要回答下面这些问题：

- 任务类型是什么
- 有哪些阶段
- 每个阶段的总量如何计算
- 每个阶段怎样拆成原子块
- 哪些阶段支持继续
- 每个阶段采用哪种恢复契约
- 如果支持继续，恢复点怎样解释
- 失败后是否允许继续，还是只能重试

也就是说，任务接入不是“给框架一个函数就行”，而是要提交一份完整的运行契约。

### 4.2 `TaskDefinition` 的最小职责

首轮建议把它固定成这几个职责：

#### 4.2.1 声明任务元信息

至少包括：

- `taskType`
- `definitionVersion`
- `supportsResume`
- `effectModel`
  - `atomic / reconcilable`
  - 任务级默认副作用模型
- `stageKeys` 或等价阶段声明
- 各阶段的 `resumeMode`
- 若支持继续，候选宇宙如何冻结为稳定的 `selectionAnchor`

#### 4.2.2 构建阶段计划

创建 `TaskRun` 时，任务定义需要一次性产出完整阶段计划，并回答：

- 当前有哪些阶段
- 阶段顺序是什么
- 每个阶段属于 `bounded / unbound / simple` 中的哪一种
- `bounded` 阶段的总量如何得到
- `unbound` 阶段的已完成量如何稳定推进
- 若该阶段支持 `resume`，该阶段的候选集合如何冻结为稳定可解释的 `selectionAnchor`

首轮固定要求：

- 阶段集合与阶段顺序在 `TaskRun` 创建时冻结
- `task_stages` 的完整骨架也在此时一并建立
- 运行过程中只允许推进既有阶段的状态、进度与恢复事实
- 不允许在阶段切换时新增、删除或重排阶段
- 对 `bounded` 阶段，任务定义必须能在进入该阶段前算出 `total`
- “进入该阶段”与“该阶段 `total` 已可见”必须作为同一原子切换提交

这里的关键约束是：

- `bounded` 阶段的总量必须在进入该阶段前确定
- `unbound` 阶段只能作为少数显式例外存在，不能成为首轮默认阶段类型
- `simple` 阶段只能用于极短且不值得维护数值进度的辅助步骤

#### 4.2.3 生产原子块

每个阶段都必须能拆成稳定的原子块，例如：

- 一批 row keys
- 一个主键游标区间
- 一个 change log 时间窗
- 一个 staging batch

原子块必须满足两个要求：

- 处理中间不可打断
- 块结束后必须可以安全地更新进度、检查控制意图、保存恢复点

这里再增加一条首轮硬约束：

- 原子块的业务副作用边界必须服从该任务声明的 `effectModel`
  - 这里的任务级 `effectModel` 只提供默认值
  - 若某个阶段或原子块需要更细粒度的副作用模型，必须在该阶段或该块上显式覆盖
  - 对 `atomic`
    - 业务副作用与块确认应尽量落在同一一致性边界内
    - 更合适的方向是：
      - 块内业务写入
      - checkpoint / done / 心跳 / 必要快照更新
      共同落在同一个数据库事务或等价一致性边界内
  - 对 `reconcilable`
    - 框架不要求业务副作用与任务快照提交天然同事务
    - 但任务必须能回答：
      - 如何判断某个块已经成功应用
      - 如何避免重复应用写坏数据
      - 恢复或重试前如何对账
      - 是否需要 apply marker、版本对比、幂等键或等价保护

也就是说，首轮不再把“所有任务都必须同事务确认副作用”当作统一前提。统一要求只有两条：

- 框架快照写入必须与块级确认一起稳定提交
- 真实业务副作用必须在所声明的副作用模型下可恢复、可重试、可检测重复执行

#### 4.2.4 执行原子块

框架不关心某个块内部具体是：

- 批量读
- 批量写
- merge
- copy into staging
- apply projection

但要求它通过统一上下文回写状态，而不是自己随意改数据库运行态。

这里要再固定一个首轮实现边界：

- 但要求任务作者把块内业务写入设计成：
  - 在声明的副作用模型下可安全确认
  - 且满足幂等或可检测重复应用

也就是说，首轮不追求为所有业务副作用建立一套独立执行权协议，而是要求块级副作用在当前单执行器模型下可安全确认、可安全重放、可检测重复执行。

### 4.3 执行器上下文

框架应向任务执行器提供一个统一上下文，至少包含这些能力：

- 读取当前 `TaskRun`
- 读取当前阶段
- 更新阶段进度
- 保存恢复点
- 刷新心跳
- 读取 `TaskRun` 上当前开放控制意图
- 标记阶段完成
- 标记任务失败 / 完成 / 取消

这样可以保证：

- 业务任务只关心“怎么处理数据”
- 框架统一管理“怎么表达运行状态”

### 4.4 执行循环

首轮建议把统一执行循环固定成下面这个模型：

1. 读取 `TaskRun`
2. 校验当前状态仍允许启动或继续
3. 准备本次即将进入的阶段事实
   - 若是首次启动，则确定首个要进入的阶段
   - 若是 `resume`，则确定本次要重新进入的暂停阶段
   - 若该阶段是首次进入的 `bounded` 阶段，在进入事务前先准备好该阶段 `total`
     - 这里的 `total` 必须已经基于冻结后的输入边界计算完成，而不是依赖进入事务后仍可能漂移的实时视图
     - 若该 `total` 依赖候选集合冻结边界，则对应 `selectionAnchor` 也必须在进入事务前一并准备好
   - 若这是 `resume` 回到一个已暂停的 `bounded` 阶段：
     - 必须直接复用该阶段暂停前已 durable 存在的 `total`
     - 不允许在 `resume` 路径上重新估算或覆盖
4. 以单个原子事务同时完成：
   - 校验该 `TaskRun` 仍处于本次允许认领的等待态
   - 将 `TaskRun.status` 从 `pending` 或 `resuming` 推进到 `running`
   - 写入本次进入 `running` 后的首个有效 `heartbeatAt`
   - 若这是首次 `pending -> running`：
     - 必须在同一提交中写入 `startedAt`
     - 且该次 `startedAt` 必须不晚于同事务写入的首个 `heartbeatAt`
   - 若这是 `resuming -> running`：
     - 不得改写既有 `startedAt`
     - 只刷新新的 `heartbeatAt`
   - durable 写入 `task_runs.currentStageKey / currentStageIndex / currentResumeMode`
   - durable 初始化或恢复对应 `task_stages` 的当前阶段状态
   - 对首次进入的 `bounded` 阶段，同时 durable 写入该阶段 `total`
   - 对依赖冻结候选边界解释的阶段，同时 durable 写入与该阶段事实相匹配的 `selectionAnchor`
5. 迭代原子块
6. 每个原子块执行后：
   - 更新 `done`
   - 保存恢复点
   - 刷新心跳
   - 检查当前是否存在 `pause / cancel` 控制意图
7. 阶段完成后进入下一阶段
8. 所有阶段完成后标记 `completed`

这意味着任务作者不应该自己控制总流程，只负责提供阶段和块的实现。

这里的关键约束是：

- 首轮不允许出现“`TaskRun.status = running`，但尚未 durable 进入任何当前阶段”的中间窗口
- 因此等待态认领成功与“进入当前阶段”必须是同一次提交
- 若阶段准备信息尚未就绪，则该次认领不得提交为 `running`
- 若同一轮启动尝试在认领提交前就已明确失败：
  - 则该实例仍停留在等待态语义
  - 应按等待态启动失败收束为 `failed`
  - 并统一写入 `terminalReason = schedule_exhausted`
- 一旦阶段已经 durable 进入，且该阶段的候选集合解释依赖 `selectionAnchor`：
  - 首跑期间后续所有原子块的候选选择，都必须只基于该次已 durable 的 `selectionAnchor` 或其等价冻结边界
  - `resume` 后的后续原子块候选选择也必须遵守同一约束
  - 不允许在阶段已经开始后回退到新的实时候选查询，重新解释该阶段面对的对象集合

这里还要补一条和残留清理直接相关的首轮硬约束：

- 若某个原子块的正常执行时长可能逼近或超过心跳超时阈值，则任务实现不能只依赖“块结束后再刷新一次心跳”
- 这类长块必须二选一：
  - 要么进一步拆小，保证单块最坏时长稳定小于心跳风险窗口
  - 要么在块内部提供框架可见的周期性续心跳点，而且这些续心跳点不能伪造“该块已经完成”的语义
- 若做不到上述任一条件，则该实现不满足首轮接入约束，因为框架将无法区分“合法长块仍在执行”与“执行器已经失活”

### 4.5 暂停与取消在执行器中的语义

执行器必须在原子块边界检查当前控制意图。

#### 4.5.1 暂停

- 如果检测到 `pause`
- 当前块完成后停止推进
- 保存恢复点
- 当前阶段置为可恢复状态
- 将当前阶段的 `resumeMode` 固化到 `pausedResumeMode`
- 若 `pausedResumeMode = session_bound`，同时保存 `runtimeBootId + resumeContextKey`
- 任务进入 `paused`

#### 4.5.2 取消

- 如果检测到 `cancel`
- 当前块完成后停止推进
- 做最小必要清理
- 不再推进后续阶段
- 任务进入 `canceled`

该框架遵循以下原则：

- 框架不要求块内抢占
- 但要求块边界响应必须一致、可预测

### 4.6 继续语义

继续不是对所有任务都默认成立，而是由任务定义中的阶段恢复契约显式声明。

继续操作本身不是“保证恢复成功”，而是一次恢复尝试。它是一个两段过程：

1. 控制层受理 `resume` 请求，把任务从 `paused` 推进到 `resuming`
2. 调度层完成恢复前提校验并成功唤起执行器后，任务才真正回到 `running`
3. `resuming -> running` 成功提交时，同事务清空：
   - `controlRequestKind`
   - `pausedResumeMode`
   - `runtimeBootId`
   - `resumeContextKey`

`resuming` 期间任务尚未重新进入执行循环，因此该状态下若收到 `cancel`，不能等待一个并不存在的执行器来消费控制意图；正确做法是由控制层或调度层直接把任务收束为 `canceled`。

继续之前还必须做恢复前提校验：

- 对 `durable` 恢复方式：
  - 调度层或执行器在真正恢复前，必须校验当前代码是否能按 `definitionVersion` 解释该任务的恢复点
  - 若兼容，则按原恢复点继续
  - 若不兼容，则本次恢复尝试失败，该运行实例应直接收束为 `failed`
- 对 `session_bound` 恢复方式：
  - 调度层或执行器在真正恢复前，必须校验 `runtimeBootId` 是否仍等于当前服务本次启动的标识
  - 并校验 `resumeContextKey` 所指向的进程内上下文是否仍然存在
  - 若存在，则按原恢复点继续
  - 若已丢失，则本次恢复尝试失败，该运行实例应直接收束为 `failed`
- 上述两类恢复尝试失败，都必须写出明确的 `terminalReason`

首轮对 `durable` 再补一个明确边界：

- 框架不主动对所有 `durable paused` 做后台兼容性扫描
- 是否发起继续，由操作者显式点击 `resume` 决定
- 点击 `resume` 时，再校验 `definitionVersion + resumeToken` 是否仍可解释
- 若不兼容，再当场收束为 `failed`

因此，`durable paused` 在首轮的产品语义不是“系统已经证明一定可以继续”，而是“任务已经安全停下，允许操作者发起一次继续尝试，届时再校验恢复契约”。

之所以需要显式的 `resuming`，是因为继续本身就是一个独立的恢复尝试阶段，而不是把 `paused` 直接同步切回 `running`。

暂停后的并发语义在首轮被统一收紧：

- `pause` 之后，该运行实例一律保留同类型任务的创建占位
- 因此不能先启动新的同类型运行实例
- `durable / session_bound` 的区别不只影响“继续前需要检查什么”
  - 还影响“这个暂停态在系统中是否会被自动降级收束”

因此首轮应把 `paused` 的产品语义明确拆成两类：

- `durable paused`
  - 是稳定的人工挂起态
  - 默认不会因后台恢复前提巡检而自动收束
- `session_bound paused`
  - 是带易失恢复前提的暂停态
  - 一旦所需进程内上下文丢失，必须由系统自动收束为 `canceled`

因此系统必须能区分两种不同语义：

- 任务只是停在 `paused`，还没人请求继续
- 任务已经请求继续，但正在等待重新启动

首轮固定以下规则：

- 只有显式声明支持暂停与继续的任务，才允许进入 `pausing / paused / resuming`
- 不支持继续的任务，也不提供暂停入口
- `paused` 不是“所有已停下但未完成任务”的统称
  - 它只表示“当前已经安全停下、但尚未发起恢复尝试的挂起态”
- `pausedResumeMode` 才是暂停后并发语义与恢复校验的直接依据
- `currentResumeMode` 只反映当前阶段契约，不能替代已暂停实例的恢复语义
- `resuming` 的产品语义不是“已经恢复成功”，而是“正在尝试恢复这个已安全停下的运行实例”
- 但这种优先是有界优先，而不是对同类型所有新任务的无限期绝对抢占
- 因此首轮产品语义中，`pause` 与 `resume` 视为成对能力

这样可以避免 UI 和用户语义混乱，同时保留后续把“可暂停”和“可继续”拆成独立能力位的扩展空间。

### 4.7 重试语义

首轮固定：

- `retry` 不复用旧 `TaskRun`
- `retry` 创建一个新的 `TaskRun`
- 新任务复制原任务参数
- 新任务通过 `retryOfTaskRunId` 指向原任务

这样有几个好处：

- 历史清晰
- 不污染旧阶段状态
- 不会把“失败过的运行实例”重新伪装成全新成功任务

### 4.8 对任务作者的硬约束

首轮框架应明确禁止以下接入方式：

- 不声明阶段总量
- 一次性全量读入内存
- 不拆原子块
- 在任务内部绕过框架直接修改任务运行状态
- 不保存块边界恢复点却声称支持继续
- 以“清空目标再整表重写”作为默认实现

这部分实际上是框架的接入门槛，而不是建议项。

### 4.9 这一部分的核心结论

第四部分最关键的决定是：

- 任务只提供定义和块级执行逻辑
- 框架掌控运行循环、状态写入、控制意图处理与心跳
- 暂停 / 取消 / 继续都必须通过统一执行循环在块边界生效
- 不支持继续的任务，不应伪装成支持暂停

## 第五部分：实时进度协议与前端接续模型

第五部分的目标是固定：

- 后端对前端暴露什么权威快照
- 实时通道推送什么状态变化
- 页面重新进入时，怎样先恢复当前状态，再接上实时进度
- 前端如何判断当前任务可暂停、可继续、可取消、可重试

### 5.1 双通道原则

首轮固定采用双通道模型：

- 快照通道
  - 读取数据库中的当前权威快照
- 实时通道
  - 推送状态变化通知

这两条通道职责不同：

- 快照通道负责“当前真相是什么”
- 实时通道负责“真相刚刚发生了什么变化”

这里的“当前真相”需要限定解释：

- 对 `session_bound` 之外的阶段，数据库快照即可完整表达当前状态
- 对命中 `session_bound` 的暂停边界，数据库快照只负责表达可恢复边界与恢复上下文标识
- 该暂停边界当前是否仍可继续，还需要结合当前服务本次启动期间保留的进程内恢复上下文判断

首轮必须保证：

- 前端即使完全错过实时事件，也能只靠快照恢复主界面
- 实时通道中断时，页面不会失去任务控制权，只是暂时失去增量刷新
- 页面真相始终以重新拉取快照为准
- 实时消息只承担轻量提示或当前主任务的局部更新职责

### 5.2 前端首次进入页面的接管流程

页面进入时，不直接先连实时通道，而是固定按下面顺序：

1. 查询当前任务类型下的当前占槽任务快照
2. 如果没有当前占槽任务：
   - 展示空闲态
3. 如果有当前占槽任务，且其状态属于 `pending / resuming`：
   - 仅在该任务与当前页面当前选中的 `activeScope` 匹配时，才自动接管该任务
   - 页面应把它视为“当前任务的等待态”，而不是空闲态
   - `pending` 更适合展示为“等待启动”
   - `resuming` 更适合展示为“正在尝试继续”
4. 如果有当前占槽任务，且其状态属于 `running / pausing / canceling / paused`：
   - 仅在该任务与当前页面当前选中的 `activeScope` 匹配时，才自动接管该任务
   - 渲染当前任务主状态
   - 渲染所有阶段状态
   - 按当前阶段的 `progressMode` 渲染对应进度信息
   - 基于 `status`、`supportsResume` 与暂停边界命中的恢复契约决定操作按钮
5. 在快照渲染完成后，再连接实时通道
6. 后续收到实时更新时：
   - 可直接局部刷新当前主任务
   - 或更简单地，直接重新拉取页面快照

这个顺序的意义是：

- 页面恢复不依赖实时连接成功
- 即使实时通道稍后才连上，前端也能先恢复主界面
- 可以避免“页面空白等事件流”的体验问题

对 `durable paused` 的展示语义，首轮建议再明确：

- 页面可以展示“继续”入口
- 但不应把它表达成“已确认一定可继续”
- 更准确的产品语义是：
  - 当前存在一个可由操作者发起继续尝试的暂停实例
  - 实际继续前，后端仍会校验恢复契约

这里还需要补一个首轮前端产品边界：

- 某个任务页面自己的主任务视图，仍然是该页面内最完整的任务详情承载面
- 但框架还应同时提供一个全局任务检视入口
  - 入口形态固定为标题栏按钮
  - 点击后打开轻量 popup
  - popup 负责展示当前全局占槽任务的摘要、阶段和可用控制动作
- 全局 popup 不是第二套独立状态机
  - 它消费与页面相同的快照语义
  - 但只展示跨页面快速检查所需的摘要层信息
- 页面主视图与全局 popup 必须共享同一套组件语义和状态文案
  - 不允许一个任务页面自己解释 `paused`，另一个 popup 再解释成别的含义

### 5.3 快照接口的最小返回结构

首轮建议存在一个“当前任务快照”接口，至少返回：

该快照接口的请求侧输入也必须写清楚，避免“`pageTask` 到底由谁推导”出现两种实现：

- 请求必须显式提供：
  - `taskType`
  - 当前页面当前选中的 `activeScope`
- 响应中的 `pageTask`
  - 由后端基于 `taskType + activeScope + 当前任务快照` 统一计算
- 前端不应只拿原始占槽事实后自行重建 `pageTask`
  - 否则 scope 切换、阻塞展示与能力位解释会再次分叉

这里先固定一个页面读模型边界：

- `pageTask`
  - 指当前页面主视图对应的唯一任务状态
  - 它是页面唯一主读模型
  - 后端内部即使需要先查询当前占槽事实，也应在后端折叠后再返回给前端
  - 更适合建模为判别联合，而不是多个并列摘要对象拼装出现
  - 它在首轮只回答一件事：当前主任务是谁

因此，这个接口不应再把“当前页面主任务是谁”“当前是谁占着该 `taskType` 的槽位”“当前 scope 最近一次结果是谁”并列暴露成多套主读模型。更合适的方向是：

- `pageTask`
  - 必填
  - 取值固定为以下互斥形态之一：
    - `idle`
    - `attached`
    - `blocking`

并满足：

- `pageTask` 任一时刻只能命中一个 `kind`
- 前端不再需要解释“多个任务位同时有值时谁优先”

也就是说，首轮更合理的快照接口不是：

- `attachedTask?`
- `blockingTask?`

而是：

- `pageTask: PageTask`

其中 `PageTask` 建议按判别联合表达：

- `idle`
  - 当前页面没有可附着的当前任务
  - 不等于“当前 `activeScope` 下从未发生过任务”
- `attached`
  - 当前页面真正附着到一个主任务
  - 该任务必然与当前 `activeScope` 匹配
  - 若该任务当前处于 `paused`，页面仍把它作为当前主任务展示，并通过 `canResume` 暴露继续入口
  - `attached` 只表示页面已经附着到主任务实例
  - 不等于该任务必然已经进入某个当前阶段
- `blocking`
  - 当前存在占槽任务，但它不属于当前 `activeScope`
  - 页面展示阻塞来源，而不是把它当成当前主任务

若页面确实需要展示“当前 `activeScope` 下最近一次终态结果”或对应重试入口，更合适的首轮方向是：

- 由独立结果接口或独立结果卡片读取
- 不与当前主 `pageTask` 快照并列绑定成单一主接口返回结构
- 这样页面主状态始终围绕“当前任务是谁”收敛，而不是同时承载“最近一次结果是谁”
- 因此主快照接口不负责承载最近结果、历史结果或重试入口

#### 5.3.1 任务级字段

对 `pageTask.kind = attached`，至少返回：

- `taskRunId`
- `runRevision`
- `taskType`
- `taskScopeType`
- `taskScopeKey`
- `taskScopeSnapshot`
- `status`
- `supportsResume`
- `currentResumeMode`
- `pausedResumeMode`
- `currentStageKey`
- `currentStageIndex`
- `progressMode`
- `startedAt`
- `finishedAt`
- `errorCode`
- `errorMessage`
- `terminalReason`

这里还应再固定一个首轮边界：

- `pageTask.kind = attached` 只承载当前主任务渲染所必需的字段
- 不承载历史结果、最近终态摘要或重试能力

其中：

- `currentStageKey / currentStageIndex` 可为空
- `progressMode` 也可为空
- 只有当任务已 durable 进入某个当前阶段时，才返回：
  - `currentStageKey`
  - `currentStageIndex`
  - `progressMode`
  - 当前阶段的 `done / total`
- 当任务仍处于等待启动，或终止发生在阶段切换间隙时：
  - `currentStageKey = null`
  - `currentStageIndex = null`
  - `progressMode = null`
  - 页面展示任务等待态，而不是展示阶段内进度
- 当任务处于 `resuming`：
  - 若它是从某个已暂停阶段发起继续尝试，则必须继续返回该暂停阶段的：
    - `currentStageKey`
    - `currentStageIndex`
    - `progressMode`
  - 页面应把它展示为“正在尝试继续当前阶段”
  - 不得把这类 `resuming` 再折叠成“没有当前阶段”的空等待态
- 当运行尚未进入任何阶段、或终止发生在阶段切换间隙时，前端应结合完整阶段列表理解“当前没有可指向的单一当前阶段”

对 `pageTask.kind = blocking`，至少返回：

- `taskRunId`
- `taskType`
- `taskScopeType`
- `taskScopeKey`
- `taskScopeSnapshot`
- `status`
- `canCancel`

对 `pageTask.kind = idle`：

- 不额外内嵌最近一次结果摘要
- 是否展示“最近一次结果”由页面按需通过独立结果读取决定

#### 5.3.2 阶段级字段

- 所有阶段的有序列表
- 每个阶段的：
  - `stageKey`
  - `stageIndex`
  - `status`
  - `total`
  - `done`
  - `startedAt`
  - `finishedAt`
- 当前阶段在完整阶段序列中的位置

这里不建议把 `selectionAnchor` 直接放进页面主快照返回结构。原因是：

- 它主要服务于恢复契约校验，而不是主页面展示
- 让前端直接理解它，会把“恢复是否成立”的后端判定逻辑重新泄露回前端
- 首轮更合理的边界是：
  - 前端消费 `canResume`
  - 后端自己解释 `selectionAnchor` 是否仍与当前恢复尝试兼容

若运维或调试页面确实需要，可单独提供诊断字段，例如：

- `resumeDiagnostics`
  - 可包含 `selectionAnchor` 的可读摘要
  - 但不作为主页面渲染前提

#### 5.3.3 控制面字段

对 `pageTask.kind = attached`，建议同时返回前端已解释好的能力位，而不是让前端自己从所有状态里推导全部规则：

- `canPause`
- `canResume`
- `canCancel`

其中应明确：

- `canPause = true`
  - 只表示当前允许发起一次暂停请求
  - 首轮固定要求同时满足：
    - `status = running`
    - 当前阶段支持暂停与继续语义
- `canResume = true`
  - 只表示当前允许用户发起一次继续尝试
  - 不表示后端已经保证本次恢复一定成功
  - 首轮固定要求同时满足：
    - `status = paused`
    - `supportsResume = true`
    - `pausedResumeMode != none`
  - 对 `session_bound`
    - 是否显示为 `true`，还应受当前恢复上下文是否仍存在影响

若当前 `taskType` 被其他 `taskScope` 的实例占用创建槽位，还应返回阻塞摘要，而不是只返回“当前 scope 下没有任务”：

- `pageTask.kind = blocking`
  - 阻塞摘要直接内嵌在 `pageTask` 对应分支里

原因是：

- 控制语义最终属于后端规则
- 前端不应复制一套完整状态机推导逻辑
- 这样更容易在后续调整规则而不破坏页面行为
- 页面需要区分“当前 scope 没有任务”和“当前 scope 被别的 scope 的同类型任务阻塞”
- 页面也需要区分：
  - `pageTask.kind = attached`
  - `pageTask.kind = blocking`
  - `pageTask.kind = idle`

对 `pageTask.kind = blocking` 里的 `canCancel`，也应明确不是“只要阻塞就天然可取消”。更合理的后端判定语义是：

- `canCancel = true`
  - 只表示当前操作者被后端允许终止这个阻塞实例
- 它必须同时受以下因素约束：
  - 该阻塞实例当前状态是否仍允许取消
  - 当前产品是否允许跨 `taskScope` 直接取消阻塞任务
  - 当前操作者是否被授予对应控制权限
- 若当前产品边界不希望页面直接取消其他 `taskScope` 的阻塞实例：
  - 应默认返回 `false`
  - 页面只展示阻塞信息，不展示取消入口

### 5.4 实时通道推送什么

首轮不要求实时通道推送所有历史事件，而只要求推送最新任务快照的变化结果。

也就是说，实时通道可以是事件形态，但它的有效载荷应该尽量贴近当前页面需要的快照字段，而不是只发非常原始的内部信号。

在 5.3 已经把页面主状态固定为 `pageTask: PageTask` 后，实时消息不需要承担复合页面快照的严格同步职责。更合适的方向是：

- 实时消息至少携带：
  - `pageTask`
  - 可选的任务级版本信息
- 其中：
  - `pageTask` 可与快照接口返回的判别联合保持同构
  - 但实时通道不要求承载额外的辅助结果读模型
- 结果型辅助视图更适合通过独立重新拉取刷新
- 主实时消息不负责把历史结果视图和当前主任务视图绑在一起

建议至少包含：

- `pageTask`
  - 其分支结构与 5.3 的快照接口保持一致
  - 即 `idle / attached / blocking`
- 若 `pageTask.kind = attached`：
  - 若 `currentStageKey != null`：
    - 当前阶段的 `progressMode`
    - 当前阶段的 `done`
    - 当前阶段的 `total`
    - 当前阶段在完整阶段序列中的位置
  - 若 `currentStageKey = null`：
    - 不返回阶段内进度字段
    - 页面按等待态展示
  - 操作能力位：
    - `canPause`
    - `canResume`
    - `canCancel`
- 若 `pageTask.kind = blocking`：
  - `canCancel`
    - 与快照接口中的阻塞摘要字段保持一致
- 若 `pageTask.kind = attached`：
  - `taskRunId`
  - `status`
  - `taskScopeType`
  - `taskScopeKey`
  - `taskScopeSnapshot`
  - `currentStageKey`
  - `currentStageIndex`
  - `startedAt`
  - `finishedAt`
  - `terminalReason`
  - `errorMessage`
- 若 `pageTask.kind = blocking`：
  - `taskRunId`
  - `status`
  - `taskScopeType`
  - `taskScopeKey`
  - `taskScopeSnapshot`
  - `startedAt`
  - `finishedAt`
  - `terminalReason`
  - `errorMessage`

首轮建议把实时推送视为“快照增量刷新”，而不是要求前端自己消费低层控制事件再重建状态。

### 5.5 前端进度计算职责

首轮固定：

- 后端提供结构化进度数据
- 前端负责计算展示层进度

也就是：

后端提供：

- 当前阶段
- 当前阶段 `progressMode`
- 当前阶段 `done`
- 当前阶段 `total`
- 阶段列表与顺序

前端负责：

- `bounded` 阶段的当前阶段百分比
- 基于阶段顺序显示明确的阶段推进指示
- 基于任务开始时间显示整体已用时间
- 估算当前阶段 ETA
- `unbound` 阶段的已完成量展示
- `simple` 阶段的状态文案展示
- 文案格式化

这样做的原因是：

- 后端只维护事实数据
- 前端更适合做展示层计算
- 同一份数据可以支持不同页面样式和不同精度的展示

### 5.6 ETA 计算前提

因为任务约束已经要求 `bounded` 阶段内部推进速度应尽量稳定，所以首轮 ETA 能力先建立在下面这个假设上：

- `bounded` 阶段内的 `done` 推进速度近似稳定

基于这个假设，前端可以按：

- 当前阶段开始时间
- 当前阶段 `done / total`
- 当前时间

先估算当前阶段剩余时间。

但文档里应明确：

- 阶段内 ETA 是首轮默认可支持的展示层估算，不是后端承诺值
- `unbound` 与 `simple` 阶段默认不承诺 ETA
- 若 `bounded` 阶段内部速度显著不稳定，则该阶段设计本身不合格，应重新切分阶段或原子块

### 5.7 实时通道中断时的行为

首轮建议固定以下行为：

- 实时通道断开时，前端保留当前页面状态，不立即清空
- 前端进入“连接中”或“更新延迟”态
- 前端应定期重新拉一次快照，作为断线兜底
- 实时通道恢复后，再继续接收实时更新

也就是说：

- 快照通道是最终兜底
- 实时通道是体验增强，不是唯一依赖

### 5.8 当前任务发现规则

首轮需要固定“页面如何知道要接哪个任务”。

这里先补一个页面侧前提：

- 一个页面可以支持多个 `taskScope`
- 但任一时刻只应存在一个当前选中的 `activeScope`
- 页面自动接续与继续入口展示，都只针对这个 `activeScope` 进行判断
- 对某一类任务页面，前端始终按“当前只有一个该类任务实例可能正在占槽或等待继续”来组织主界面
- 页面每次切换 `activeScope` 时，都应基于新的 `activeScope` 重新计算当前主任务
- 因此页面不存在脱离当前 `activeScope` 长期粘住旧主任务实例的读模型语义

这里必须区分两类发现对象：

- 当前等待态任务
  - 指 `pending / resuming`
  - 它表示当前任务已经被受理，但尚未真正进入执行循环
- 当前执行中任务
  - 指 `running / pausing / canceling`
  - 它才是页面自动接管的首要对象
- 当前暂停任务
  - 指 `paused`
  - 它虽然不处于执行循环中，但若仍占用创建槽位，页面仍应把它视为当前占槽任务的一种状态

因此建议采用：

- 先由当前页面解析自己当前选中的 `activeScope`
- 先按任务类型查询当前占槽任务
- 若查到当前占槽任务，再校验其 `taskScope` 是否与当前页面的 `activeScope` 匹配

这里看起来像是两套不同规则，但首轮其实是同一个模型的两个投影：

- 当前占槽任务先按 `taskType` 缩小候选，是因为首轮已经固定“同一 `taskType` 下占槽任务最多同时存在一个”
- 页面是否形成 `pageTask.kind = attached`，仍然要靠 `activeScope` 与任务 `taskScope` 的匹配做最终判断

这样做的原因是：

- `taskType` 只说明任务类别，不说明它是对谁做事
- `taskScopeKey` 才能稳定回答“任务现在作用于哪个业务对象”
- 当前占槽任务即使在串行层面按 `taskType` 全局唯一，也不代表它一定属于当前页面上下文
- 页面切换 `activeScope` 时，应直接按新的上下文重新决定当前主任务是谁

因此首轮最自然的是：

- 某个任务页面先解析自己当前选中的 `activeScope`
- 再查询自己所属 `taskType` 下的当前占槽任务
- 如果存在当前占槽任务且其 `taskScope` 与当前 `activeScope` 匹配：
  - 页面形成 `pageTask.kind = attached`
  - `pending / resuming` 作为当前主任务的等待态展示
  - `running / pausing / canceling` 作为当前主任务的执行中态展示
  - `paused` 作为当前主任务的暂停态展示，并通过 `canResume` 暴露继续入口
- 如果存在当前占槽任务但 `taskScope` 不匹配：
  - 页面形成 `pageTask.kind = blocking`
- 如果不存在当前占槽任务：
  - 页面形成 `pageTask.kind = idle`
  - 只在此时展示创建或启动入口

这里的阻塞实例摘要不应靠前端自行拼装推断，而应由后端基于 `task_runs` 上受数据库约束保护的唯一占位实例直接返回。

这样可以避免前端把“未结束任务”和“当前正在运行的任务”混为一谈。

以现有 `publish / reanchor` 为例：

- 页面上下文就是一个 `publish stream`
- 页面自动接管的是当前 `publish` 任务类型下、且 `taskScope` 与当前 `activeScope` 匹配的占槽任务
  - 若该占槽任务处于 `pending / resuming`，页面展示等待态
  - 若该占槽任务处于 `running / pausing / canceling`，页面展示执行中态
  - 若该占槽任务处于 `paused`，页面展示暂停态与继续入口
- 如果唯一占槽任务属于别的 `publish stream`，页面展示的是 `pageTask.kind = blocking`
- 而不是所有 `publish` 类任务都被视为当前主任务

### 5.9 操作按钮的前端规则

前端不应该只看字符串状态直接硬编码按钮，而应遵循后端返回的能力位。

首轮建议规则是：

- 页面展示状态文案和进度条
- 是否显示或启用：
  - 暂停
  - 继续
  - 取消
- 由后端在快照与实时消息里统一提供
- 对“继续”按钮，前端产品语义应按“发起继续尝试”理解
- 若继续尝试失败，前端应把它展示为一次明确失败，而不是把它误解成“暂停态被异常打断”
- 对 `paused / resuming` 这类继续前未完成但仍占用创建槽位的实例，必须显式暴露手动 `cancel` 能力
- 对来自其他 `taskScope` 的阻塞实例，前端至少要能展示“当前被哪个任务阻塞”；若当前用户有权限且后端允许，也可以直接展示取消该阻塞实例的入口

“重试”更适合作为终态结果视图或历史结果视图中的动作，而不是当前主 `pageTask` 的一部分。首轮应把它与“当前主任务是谁”的页面模型分开。

这样前端只负责消费，不重新实现状态机。

这里还需要补一个统一前端实现边界：

- 重任务前端不应只共享 composable，还应共享组件
- 首轮至少应抽出一组统一的任务 UI 组件，用于：
  - 当前任务状态卡
  - 阶段进度列表
  - 操作按钮条
  - 阻塞提示
  - 全局 popup 中的任务摘要卡
- 这些组件更适合放在 console 共享层，而不是散落在单个页面里
- `publish` 页面和全局 popup 必须优先复用同一套组件
- 后续 `reanchor` 或其他任务页面也应在这套组件上扩展，而不是重写自己的任务展示骨架

### 5.10 首轮不要求的实时能力

为了避免过重，首轮不要求：

- 每条原子块处理日志实时展开
- 完整历史事件回放
- 多页面同时编辑控制的一致性提示优化
- 跨浏览器标签页协同去重显示
- 离线缓存完整任务时间线

这些都可以以后加，但不属于首轮必需能力。

### 5.11 写侧控制接口契约

前面已经固定“前端不自己实现状态机，而是消费后端能力位”。与之对应，写侧接口也应固定为“发送控制意图”，而不是“请求直接切到某个终态”。

首轮建议最小控制接口集合固定为：

- `create task`
  - 输入：
    - `taskType`
    - `taskScopeType`
    - `taskScopeKey`
    - `taskScopeSnapshot`
    - `params`
  - 结果：
    - 返回任务级写侧结果，至少包含：
      - `taskRunId`
      - `runRevision`
      - `status`
- `pause task`
  - 输入：
    - `taskRunId`
    - 可选 `expectedRunRevision`
  - 结果：
    - 返回受理后的任务级写侧结果
- `resume task`
  - 输入：
    - `taskRunId`
    - 可选 `expectedRunRevision`
  - 结果：
    - 返回受理后的任务级写侧结果
    - 这里的“成功”只表示继续请求已被受理
    - 不表示恢复尝试已经成功完成
- `cancel task`
  - 输入：
    - `taskRunId`
    - 可选 `expectedRunRevision`
  - 结果：
    - 返回受理或直接收束后的任务级写侧结果
- `retry task`
  - 输入：
    - `taskRunId`
    - 可选 `expectedRunRevision`
  - 结果：
    - 若满足重试前提，则创建新的 `TaskRun`
    - 否则拒绝并返回当前任务级最新结果

这里还需要把 `create / retry` 的调度语义固定成单一口径，避免“创建成功”到底意味着什么出现两种解释。首轮更合理的约束是：

- `create task`
  - 只负责：
    - 校验输入
    - 创建新的 `TaskRun`
    - 把它写成“立即可调度”的等待态
    - 返回创建后的任务级写侧结果
  - 不要求在接口返回前已经成功触发一次调度尝试
- `retry task`
  - 也只负责：
    - 校验当前不存在其他占用同 `taskType` 创建槽位的运行实例
    - 创建新的 `TaskRun`
    - 继承必要的重试关联信息
    - 把它写成“立即可调度”的等待态
    - 返回创建后的任务级写侧结果
  - 不要求在接口返回前已经成功触发一次调度尝试
- 调度层可以在创建后异步尽快尝试启动
  - 但这属于后续调度职责
  - 不是 `create / retry` 成功语义的一部分
- 因此 `create / retry` 与“创建后顺手触发一次调度尝试”之间是解耦关系
  - 实现层可以在返回后尽快触发
  - 但即使这一步暂时没发生，`create / retry` 仍然视为成功

也就是说，首轮的 `create succeeded` / `retry succeeded` 明确表示：

- 新运行实例已经成功持久化
- 且已进入框架可继续推进的等待态
- 不表示它已经开始执行
- 也不表示本次返回前已经完成同步调度

若希望实现层在创建后顺手触发一次调度尝试，可以做；但文档语义上应把它视为最佳努力优化，而不是接口成功判定的一部分。

- `create task`
  - 一旦创建成功，前端应随后重新拉取页面快照
  - 若该新实例就是当前 `taskType` 的唯一占槽实例，则新的页面快照应把它视为新的 `pageTask`
- `retry task`
  - 命中重试前提时创建新的 `TaskRun`
  - 新实例通过 `retryOfTaskRunId` 指向旧实例
  - 前端应随后重新拉取页面快照，而不是继续停留在旧终态实例上
  - 若当前已有别的占槽实例，则应直接拒绝
- 对被重试的旧实例：
  - 它保持原终态不变
  - 不因新实例创建而被改写、复活或隐藏成别的状态
- 页面主视图层面：
  - `create / retry` 成功后，主任务身份应切换到新实例
  - 旧实例若仍需展示，只应作为历史或关联信息出现
  - 不应继续占据主任务位与新实例并列竞争

也就是说，前端在 `retry` 成功后的首轮默认心智应是：

- “我现在正在看新的运行实例”
- 而不是“我还停留在旧失败实例，只是旁边多了一个新任务”

这里建议显式接受 `expectedRunRevision`，不是为了做复杂乐观锁协议，而是为了防止页面拿着过旧快照重复点击时误以为自己还在操作同一个状态。

若校验失败，接口更合理的行为是：

- 不静默覆盖
- 返回当前任务级最新结果
- 让前端随后重新拉取页面快照并重渲染

写侧接口的错误语义也要和前面的状态机保持一致：

- `pause / resume / cancel / retry` 被拒绝时
  - 表示“该控制意图在当前最新状态下不合法”
  - 不表示后端故障
- 真正的后端错误
  - 仍按普通接口失败处理
- 若请求虽然被受理，但后续恢复或调度失败
  - 不在当前写接口同步返回“恢复失败”
  - 而是通过后续快照或实时消息把任务推进到对应终态

也就是说，写侧接口不再直接承担“返回某个页面视角下完整 `pageTask`”的职责。页面视图仍统一由 `taskType + activeScope` 对应的快照接口计算；写侧只返回任务级结果，前端在写入成功后再重新拉取页面快照。

### 5.12 这一部分的核心结论

第五部分最关键的决定是：

- 前端永远先拿数据库快照，再接实时通道
- 实时通道推的是“最新状态变化”，不是唯一真相
- 后端提供事实型进度数据，前端负责展示型计算
- 操作按钮能力由后端统一解释，不让前端复制状态机
- 写侧接口发送的是控制意图，并返回任务级结果；页面视图仍通过快照接口统一计算
- 实时断线时，快照拉取是兜底机制
- 页面主视图与全局任务 popup 必须共享同一套任务组件语义

## 第六部分：轻量调度与残留运行态清理机制

第六部分的目标是固定三件事：

- 谁来决定任务什么时候开始执行
- 同类型任务如何保证单槽串行
- 执行器消失后，数据库里的执行中任务如何被识别并清理

这里先固定一个首轮边界：

- 调度器是轻量执行控制器，不是复杂任务平台
- 它的职责只包括：
  - 发现可执行任务
  - 判断当前是否允许启动或继续
  - 推进任务进入执行
  - 处理取消、继续与残留收束
- 它不承担：
  - 公平排队
  - 优先级调度
  - worker 池编排
  - 复杂退避策略
  - 任务依赖图求解
  - 高吞吐 backlog 消化

### 6.1 首轮调度模型

首轮不要求独立的外部调度平台。

建议采用：

- 应用内部调度器
- 应用内部执行器
- 数据库快照 + 心跳 作为状态与存活判断基础

也就是说，首轮模型不是：

- 前端直接调用长函数跑完整任务

而是：

1. 前端创建 `TaskRun`
2. 控制层尝试调度该任务
3. 调度器确认当前允许启动后启动执行器
4. 执行器周期性刷新心跳
5. 任务结束时主动写入终态

这样即使首轮仍跑在同一个后端进程里，也已经把“请求入口”和“任务执行”解耦了。

### 6.2 为什么单独需要调度层

虽然首轮不引入外部 job system，但仍然建议单独建调度层，而不是让“创建任务接口”直接把任务跑起来。

原因只有两个：

#### 6.2.1 解耦创建与执行

创建任务只负责：

- 写 `TaskRun`
- 校验参数
- 返回任务 ID

执行任务则由调度层决定何时启动。

#### 6.2.2 统一处理状态收束

调度层统一负责：

- 判断当前是否允许启动或继续
- 启动时扫描残留活跃态
- 对等待态、暂停态和残留态做必要收束

### 6.3 心跳与时间窗口

首轮文档里不必写死具体毫秒值，但建议固定这类规则：

- 心跳超时阈值应明显长于一次正常数据库事务提交时间
- 心跳超时阈值不应长到“进程挂了还要等很久才能回收”
- 心跳刷新间隔应明显短于超时阈值

例如在实现层通常会形成这样的关系：

- 心跳间隔 < 心跳超时阈值
- 清理阈值 >= 心跳超时阈值
- 若只在块边界刷新心跳，则原子块最大时长必须稳定小于心跳风险窗口
- 若原子块无法稳定满足这一条件，则实现必须提供块内周期性续心跳点；否则该块不允许进入首轮框架

也就是说，块不能无限大，否则暂停、取消和残留识别都会失真。

### 6.4 启动失败怎么处理

启动执行或块边界提交失败时，首轮建议固定为：

- 立即停止继续推进后续原子块
- 将任务转入失败或放弃收束流程
- 返回明确的失败原因

不要允许“框架已经确认该轮执行失败，但任务逻辑还继续推进后续块”。

更严格地说，不要允许“当前轮执行已经判定失效，但任务逻辑仍凭旧上下文继续提交后续状态写入”。

原因很直接：

- 当前框架是单执行器模型
- 一旦本轮执行已经确认失败，后续推进只能制造新的不一致

因此这类失败应视为运行时一致性错误，而不是普通可忽略告警。

首轮还必须把这条约束推广到所有执行期状态写入：

- 只要某轮执行上下文已经被判定失效：
  - 后续任何基于该旧上下文的 `done / checkpoint / heartbeat / stage status / terminal` 写入都必须被数据库前置条件拦下
- 也就是说，首轮不只要求“启动认领”具备原子前置条件
  - 执行期后续所有状态提交也必须校验：
    - 当前 `TaskRun` 仍处于该执行上下文允许写入的状态
    - 且未被更晚的清理、取消、失败或其他执行轮次收束覆盖
- 若校验失败：
  - 该旧执行上下文必须立即停止继续提交
  - 不允许把迟到写入再覆盖到已经 `abandoned / canceled / failed / completed` 的新事实之上

对 `session_bound`，首轮还固定一条恢复边界：

- 一旦后端进程重启，原先暂停实例记录的 `runtimeBootId` 与当前服务启动标识就不再一致，其所依赖的进程内上下文也视为失效
- 因此这类运行实例不得在重启后继续停留在“可继续”的暂停语义下
- 启动清理时，除执行中残留任务外，还应额外扫描 `paused + pausedResumeMode = session_bound` 运行实例
- 对确认上下文已失效的实例，应直接收束为 `canceled`
- 并通过 `terminalReason` 明确标记为恢复上下文丢失

### 6.5 调度器如何发现可执行任务

首轮建议调度器只关心两类可执行候选：

- `pending`
- `resuming`

由于首轮同一 `taskType` 下占用创建槽位的运行实例保持唯一，调度器不会在同一 `taskType` 内面对多个并列候选。

也就是说：

- `pending` 表示当前唯一占槽任务尚未启动
- `resuming` 表示当前唯一占槽任务已请求继续，正在等待重新进入执行循环
- 首轮不存在“多个同类型 `pending` 或 `resuming` 候选之间如何公平排序”的问题

因此首轮这里不要求同类型 backlog 公平性算法。调度器只需要保证：

- 调度决策是确定性的
- `resuming` 不会被错误当成新的 `pending`
- 当唯一任务状态变化时，调度器能按其当前状态正确尝试启动或继续

这里还要再固定一条首轮活性原则：

- `pending / resuming` 必须能够被再次发现并再次尝试启动
- 不允许把“曾经尝试调度过一次”当成后续永远有人会接管的前提
- 在个人电脑、单服务、低频人工触发场景下，首轮不额外引入独立调度 claim 表
- 但“不引入独立 claim 表”不等于允许多个入口直接并发唤起同一个等待态实例
- 对同一个 `TaskRun` 的启动或继续尝试，首轮必须先经过一次原子认领
  - 认领前提只能是该实例当前仍处于 `pending` 或 `resuming`
  - 认领动作不能只是孤立更新 `task_runs.status`
  - 它必须和“把该实例 durable 推进到当前阶段”放在同一原子提交里完成
  - 更具体地说：
    - `pending` 的启动认领必须以 `status = pending` 为前提，在同一提交中原子推进到 `running + 已进入首个当前阶段`
    - `resuming` 的继续认领必须以 `status = resuming` 为前提，在同一提交中原子推进到 `running + 已重新进入暂停前的当前阶段`
  - 若更新影响行数为 `0`
    - 说明该实例已被其他入口接管，或状态已变化
    - 当前入口必须放弃本次启动，不得继续进入执行循环
- 因此，创建后触发、巡检补触发、启动时清理补触发、控制入口顺手补触发，虽然都可以再次发现同一个等待态实例，但都只能把“原子认领成功”的那一次真正推进到执行循环
- 首轮只要求：
  - 创建或继续请求先把任务推进到 `pending / resuming`
  - 调度器随后尽快尝试启动
  - 若本次尝试没有成功进入 `running`
    - 则在后续轻量巡检、启动时清理或下一次控制入口触发时再次尝试
    - 或在已经明确不可恢复时收束为终态

这样可以避免非执行态运行实例在数据库中长期占槽，却没有任何框架内机制继续推动它；同时又不把首轮调度器设计成复杂等待队列系统。

### 6.6 残留运行态的识别条件

首轮需要一个统一规则来判断：

- 数据库里这个任务看起来还活着，但实际上已经没有执行器了

建议基于以下条件综合判断：

- `task_runs.status` 仍处于执行中状态
- `heartbeatAt` 已超过阈值
- 当前进程内没有对应执行器实例
  - 这一条只在同进程模式下可作为辅助判断，不应成为唯一依据

这里最关键的是：

- 不要只靠“进程内没有对象了”判断
- 必须以数据库中的心跳超时作为主依据
- `pending` 和 `paused` 不应进入这一判断集合
  - `pending` 只是等待启动
  - `paused` 表示任务已在安全边界停下

这里还需要再明确区分两类“看起来没动”的状态：

- 执行中状态没心跳
  - 属于残留运行态候选，应进入 `abandoned` 清理语义
- `pending / resuming` 长时间未推进
  - 不属于残留运行态
  - 但必须进入调度巡检和必要时的失败收束语义

也就是说，首轮不允许把 `pending / resuming` 的长期停滞解释成“反正不是运行中，所以先放着不管”。

### 6.7 启动清理流程

首轮固定要求：

- 后端启动时，必须执行一次残留运行态清理

清理流程建议是：

1. 扫描所有执行中状态的任务
2. 判断其 `heartbeatAt` 是否超时
3. 对确认失活的执行中任务：
   - 把 `TaskRun` 收束到 `abandoned`
   - 写入 `terminalReason = abandoned_stale_run`
   - 必须把当前仍未终结的阶段同步标记为 `abandoned`
4. 额外扫描所有 `paused + pausedResumeMode = session_bound` 运行实例
5. 将这类实例直接收束为 `canceled`
6. 写入 `terminalReason = resume_context_lost`

这样页面在启动后重新打开时，不会继续看到伪运行中的任务。

除上述启动清理外，启动流程还应顺手对 `pending / resuming` 执行一次轻量调度巡检：

- 对等待态实例，重新纳入调度候选
- 对已经明显不可恢复的实例，允许直接收束为终态
- 这里的收束不属于 `abandoned`
  - 更合适的方向是：
    - 首次启动类等待态确认无法启动后收束为 `failed`
    - 恢复类等待态若恢复尝试失败，则收束为 `failed`

这里还要补一个首轮最小判据，避免不同实现各自决定“什么时候算等不到了”：

- `pending / resuming` 不能仅因为“已经等待了一段时间”就直接收束为 `failed`
- 首轮允许把等待态直接收束为 `failed` 的情况，只限于框架已经拿到明确的不可推进证据，例如：
  - 启动认领后，在真正进入执行循环前发生了明确启动失败
  - `resuming` 的恢复前提校验明确失败
    - 例如 `definitionVersion + resumeToken` 不再可解释
    - 或 `session_bound` 所需上下文已丢失
  - 巡检或人工修复已经确认该记录存在状态不一致，且框架无法安全恢复推进
- 除上述“明确不可推进”外：
  - `pending / resuming` 只能保留为等待态，并在后续巡检、启动时清理或控制入口补触发中继续重试
  - 不允许仅基于实现自定的重试次数、等待时长或瞬时资源抖动，随意把等待态提前收束为 `failed`

### 6.8 巡检清理流程

除了启动清理，首轮还建议保留一个轻量巡检能力。

原因是：

- 有些异常不是发生在重启时
- 进程可能长时间不重启
- 如果某次执行器静默消失，仅靠“下次重启”清理不够稳

因此首轮必须至少提供一种稳定的等待态再驱动机制：

- 一个定时巡检任务
- 或等价的、可持续发现并再次推进 `pending / resuming` 的稳定机制

但这里不要求先做复杂后台守护系统；首轮硬要求只是：

- 不能把 `pending / resuming` 的继续推进完全寄托在“也许之后有人再点一次按钮”
- 在长时间不重启、也没有后续人工操作时，框架自己仍必须有能力再次发现并推进这些等待态

这里的“巡检”在首轮实际上应拆成三类职责：

- 残留运行态巡检
  - 面向 `running / pausing / canceling`
  - 解决执行器消失、心跳失活
- 调度等待态巡检
  - 面向 `pending / resuming`
  - 解决任务已占槽但尚未真正进入执行的问题
- `session_bound paused` 恢复前提巡检
  - 面向 `paused + pausedResumeMode = session_bound`
  - 解决“启动后未立即清理到，或运行中恢复上下文后来失效”的问题

对 `session_bound paused` 巡检，首轮建议固定以下规则：

- 只扫描 `status = paused` 且 `pausedResumeMode = session_bound` 的实例
- 每次巡检都要校验：
  - `runtimeBootId` 是否仍等于当前服务本次启动标识
  - `resumeContextKey` 所指向的进程内上下文是否仍存在
- 若任一条件不成立：
  - 不允许该实例继续停留在“可继续暂停”语义下
  - 应直接收束为 `canceled`
  - 并写入 `terminalReason = resume_context_lost`

也就是说，`session_bound paused` 的失效收束不能只依赖“服务启动时顺手清一次”。只要服务还活着，框架就应能在周期性巡检中持续识别这类已不再可继续的暂停实例。

对调度等待态巡检，首轮建议固定以下规则：

- 只扫描 `status in (pending, resuming)` 的实例
- 若发现该实例仍可尝试启动或继续，则再发起一次轻量启动尝试
- 若已确认恢复前提永久失效：
  - `resuming` 应收束为 `failed`
  - 若原因是恢复点或恢复阶段事实已不可解释或已损坏：
    - 统一写入 `terminalReason = resume_incompatible`
  - 若原因是 `session_bound` 恢复上下文已失效：
    - 统一写入 `terminalReason = resume_context_lost`
- 若已确认该等待态无法再由框架继续推进：
  - 允许直接收束为 `failed`
  - 且首轮统一写入 `terminalReason = schedule_exhausted`
- 这里的“已确认无法再由框架继续推进”在首轮必须按上文同一口径解释：
  - 需要存在明确不可推进证据
  - 不能只因为等待过久或已经重试过若干次就直接判失败
  - 且这里仅指调度侧无法把等待态推进到执行
  - 不包括已经落入明确恢复失败语义的 `resuming`

这样首轮即使不引入额外调度表，也能保证：

- `pending / resuming` 不是永远占槽的静态记录
- 等待态失败会被框架自己收束
- 并发控制仍主要集中在 `task_runs` 的单槽约束与状态机上

### 6.9 `abandoned` 与 `failed` 的边界

这两个状态必须严格区分。

#### 6.9.1 `failed`

表示：

- 系统已经明确知道这次运行不能按成功路径继续收束
- 失败原因是可解释的，而不是“执行器消失后只能猜测”
- 该终态可以来自多类明确失败来源，例如：
  - 执行期真实失败
  - `resume` 尝试失败
  - `pending / resuming` 无法再由框架继续推进，最终被判定为失败

这里刻意不把 `failed` 继续收窄为“只有执行器真实跑起来并抛错”。原因是首轮已经接受以下两类失败也应被系统主动收束，而不是长期挂起：

- `resuming`
  - 恢复契约检查失败
  - `session_bound` 恢复上下文丢失
- `pending / resuming`
  - 调度层反复尝试后仍无法进入真正执行
  - 且已被判定无法继续由框架推进

因此，`failed` 在首轮表示“系统明确知道这次运行失败了”，而不是“系统只知道执行器曾经抛错了”。

但为了避免 `failed` 重新退化成一个不可解释的大类，首轮要求：

- `status = failed` 时，`terminalReason` 必填
- 前端、日志与运维判断失败来源时，应优先解读 `terminalReason`
- `errorCode / errorMessage` 主要补充机器可判定错误码与人类可读错误信息
- 不应只靠 `status = failed` 推测是执行失败、恢复失败还是等待态收束失败

首轮建议至少把 `terminalReason` 细分到以下来源层级：

- `execution_*`
  - 执行器真实进入执行循环后的业务或执行失败
- `resume_*`
  - 恢复前提不满足、恢复校验失败、恢复上下文丢失
- `schedule_*`
  - `pending / resuming` 无法再由框架继续推进

但这里还应再补一个首轮最小枚举边界，避免只停留在命名前缀层面。更合适的方向是：

- `execution_failed`
  - 执行器已进入执行循环，并发生明确业务或执行失败
- `resume_incompatible`
  - 当前定义版本已无法兼容解释旧恢复点
- `resume_context_lost`
  - `session_bound` 所需的进程内恢复上下文已失效
- `schedule_exhausted`
  - 等待态已被框架判定无法再由调度侧继续推进
  - 包括认领后、进入执行循环前的明确启动失败
- `manual_cancel`
  - 操作者显式发起取消，且运行最终按 `canceled` 正常收束
- `system_cancel`
  - 控制层或调度层基于明确系统判定主动终止，且运行最终按 `canceled` 收束
- `abandoned_stale_run`
  - 执行中残留实例被清理流程收束

这里再明确一条首轮口径，避免把命名误读成别的维度：

- `manual_cancel` 与 `system_cancel` 在首轮都被定义为稳定的终止原因码
- 它们回答的是“任务为什么落入 `canceled` 终态”
- 不应被解释为独立于 `terminalReason` 之外的来源字段、操作者字段或审计字段

首轮这里还要固定一条 reason code 一致性要求：

- 对同一类终止来源，所有收束路径都必须写入同一个稳定 `terminalReason`
- 不允许启动清理、周期巡检、控制入口补收束、恢复尝试失败各自发明近义但不相等的原因码
- 至少以下来源在首轮必须全路径统一：
  - 操作者显式取消，且未被更高优先级失败覆盖
    - 统一写入 `manual_cancel`
  - 控制层或调度层基于明确系统判定主动终止，且未命中更具体原因码
    - 统一写入 `system_cancel`
  - `session_bound` 恢复上下文丢失
    - 统一写入 `resume_context_lost`
  - 恢复点、阶段进度事实或恢复契约已不可解释
    - 统一写入 `resume_incompatible`
  - 执行中残留实例清理
    - 统一写入 `abandoned_stale_run`
  - 等待态已被框架确认无法再由调度侧继续推进
    - 统一写入 `schedule_exhausted`

首轮不要求现在就把所有终止原因穷举完，但这里的 `terminalReason` 必须始终按“终止原因”单一维度分类，而不是混入“谁发起了这个终止动作”之类第二维语义。至少应保证：

- 前端和运维常见分支不需要解析自然语言
- 失败、取消、放弃三大类都有稳定可比较的原因码
- 后续新增原因时，应以扩展枚举为主，而不是改写旧原因语义

这样可以在不扩充终态集合的前提下，保留统一失败状态，同时维持失败来源的可解释性。

#### 6.9.2 `abandoned`

表示：

- 系统无法证明任务完整结束
- 数据库里只留下过期的运行态
- 是由清理流程收束，而不是由正常执行路径主动写入

若 `TaskRun` 被清理为 `abandoned`，阶段快照也不应借用 `failed` 表达该语义，因为 `failed` 仍表示系统明确知道执行路径发生了业务或执行失败。首轮固定要求：

- 当前活跃阶段或最后可明确识别的活跃阶段，应同步收束为 `abandoned`
- 已经处于 `completed / failed / canceled` 的历史阶段可保持原状态
- 尚未开始的后续阶段保持 `pending`

这两种状态后续在前端和运维上都应区分展示。

#### 6.9.2.1 `failed / canceled` 时的阶段收束规则

除了 `abandoned`，首轮还需要把普通终态的阶段收束规则固定下来：

- 若终止发生在 `resuming` 且尚未重新进入阶段执行：
  - 当前阶段保持暂停前最后一次稳定状态
  - 通常表现为 `paused`
  - 该例外覆盖下面普通的“把最后活跃阶段改写成 `failed / canceled`”规则
- 若任务从 `paused` 被系统直接收束为 `canceled`，且在此之前未重新进入阶段执行：
  - 当前阶段保持 `paused`
  - 由 `TaskRun.status = canceled` 与对应 `terminalReason` 表达“该暂停点已不可继续”
  - 不把这类系统终止改写成阶段级 `canceled`
- 若任务从 `paused` 被用户直接收束为 `canceled`，且在此之前未重新进入阶段执行：
  - 当前阶段同样保持 `paused`
  - 由 `TaskRun.status = canceled` 与对应 `terminalReason` 表达“该暂停点已被明确放弃继续”
  - 不把这类取消改写成阶段级 `canceled`
- 当 `TaskRun` 收束为 `failed`：
  - 当前正在推进或最后活跃的阶段应收束为 `failed`
  - 已经 `completed` 的历史阶段保持 `completed`
  - 尚未开始的后续阶段保持 `pending`
- 当 `TaskRun` 收束为 `canceled`：
  - 当前正在推进或最后活跃的阶段应收束为 `canceled`
  - 已经 `completed` 的历史阶段保持 `completed`
  - 尚未开始的后续阶段保持 `pending`
- 若终止发生在任何阶段 durably 开始之前：
  - 所有阶段保持 `pending`
- 若终止发生在阶段切换间隙：
  - 已经 `completed` 的历史阶段保持 `completed`
  - 后续尚未开始的阶段保持 `pending`
  - 不得把终态回写到已经完成的上一阶段

也就是说，首轮不把“所有未来阶段都一起改成运行终态”作为默认规则。阶段快照仍然表达“这次运行实际上推进到了哪里”，而不是简单复制 `TaskRun` 的最终状态。

### 6.9.3 `canceled` 的边界

`canceled` 不只表示用户主动取消。

它统一表示：

- 运行实例未完成
- 但系统已经明确决定不再继续推进

其来源可以包括：

- 用户主动取消
- 控制层或调度层显式终止
- `session_bound` 恢复前提丢失后的自动收束

首轮对这些来源的 `terminalReason` 口径进一步固定为：

- 用户主动取消，且最终确实以取消语义收束：
  - 写入 `manual_cancel`
- 控制层或调度层基于一般系统判定主动终止，且没有更具体的失败或恢复原因：
  - 写入 `system_cancel`
- 若终止同时命中更具体原因：
  - 例如 `resume_context_lost`
  - 则优先写更具体原因，而不是退化成 `system_cancel`

因此前端与运维不应只靠 `status = canceled` 猜测来源，而应结合 `terminalReason` 解读真正的终止原因。

### 6.10 首轮不要求的调度能力

为了避免过重，首轮不要求：

- 分布式多节点 leader election
- 抢占式优先级调度
- 多租户公平调度
- 任务依赖图调度
- worker 池分片调度
- 自动扩缩容

这些都属于以后可能的演进方向，不属于当前应用内重任务框架的首轮目标。

### 6.11 这一部分的核心结论

第六部分最关键的决定是：

- 任务执行必须经过调度层，而不是由创建接口直接同步跑完
- 类型级创建槽位唯一性靠 `task_runs` 上的数据库约束保证，而不是靠控制层先查后写碰运气
- 心跳是存活判断基础
- `paused` 释放执行推进权，但继续保留同类型创建槽位
- 启动清理和巡检清理都属于框架职责
- 残留任务要收束到 `abandoned`，而不是继续伪装成运行中

## 第七部分：错误处理、幂等性与执行策略约束

第七部分的目标是把前面所有运行模型真正落到最核心的问题上：

- 大批量读写可以做
- 但不能让执行策略失控
- 失败后要能重试
- 支持暂停与继续的任务要能在块边界安全恢复
- 每个原子块必须声明明确的副作用模型与一致性边界

### 7.1 首轮最重要的硬约束

首轮框架应明确规定：

- 每个重量级任务都必须声明自己的执行策略
- 执行策略至少要回答：
  - 候选数据如何确定
  - 这些候选数据在 pause / resume 之间如何冻结为稳定的 `selectionAnchor`
  - 阶段总量如何解释
  - 原子块如何切分
  - checkpoint 如何落点
  - 原子块属于 `atomic` 还是 `reconcilable`
    - 该声明粒度至少要到阶段；若同一阶段内仍存在不同块语义，则继续细化到块级
- 增量执行是首轮推荐默认策略
- 全量执行允许存在，但必须显式声明，并额外说明为什么当前无法安全增量化

也就是说，框架不能只解决“长任务怎么跑”，还要解决“长任务怎么不失控”。

### 7.2 什么叫合格的执行策略

首轮这里需要下一个非常明确的定义：

一个任务只有满足下面条件，才算符合本框架的接入要求：

1. 能说明“本次为什么处理这些数据”
2. 对默认的 `bounded` 阶段，能在进入阶段前算出该阶段总量；若使用 `unbound` 阶段，必须显式说明为什么无法预估总量
3. 能把处理对象切成可边界恢复的原子块
4. 每个原子块只处理当前块已声明的候选数据
5. 若支持 `resume`，能说明 pause / resume 之间如何冻结同一批候选对象，而不是在恢复时重新落入新的候选宇宙
6. 能说明原子块重复执行时如何保持幂等或检测重复应用
7. 能说明失败、暂停和继续分别在什么边界上安全生效
8. 能说明该任务的原子块副作用属于 `atomic` 还是 `reconcilable`，以及框架如何确认块完成
9. 若采用全量执行，还必须说明：
   - 为什么当前没有可信增量依据
   - 为什么全量执行仍然可以被分块、观测、取消和重试
   - 为什么不会退化成不可控的大事务或不可恢复的一次性脚本

### 7.3 每个任务必须声明候选依据

首轮要求每个任务定义都必须显式声明自己的候选依据。

常见形式可以包括：

- `updatedAt > checkpoint`
- 单调递增版本号
- change log / delta log
- source tag / build range
- 上次 baseline 与当前候选集比较
- 显式候选键集合
- 明确声明“本次是全量候选集”

但无论采用哪种方式，任务定义都必须回答：

- 为什么这次只处理这批数据
- 为什么没被选中的数据可以安全跳过

如果答案是“因为这次就是全量候选集”，也必须显式写出来，而不是让框架默认替任务补全这个语义。

### 7.4 每个阶段必须声明总量来源

前面已经要求首轮默认阶段应当是 `bounded`，这里要补一层更严格的约束：

- `bounded` 阶段的总量不能靠“先把全量数据全扫一遍再慢慢算”得到
- 总量计算本身也必须服从该任务声明的候选依据
- `unbound` 阶段虽然可以没有总量，但仍要能解释 `done` 的推进单位是什么，以及为什么无法提前给出可信 `total`

举例来说：

- 如果某任务的候选依据是 `change_log`
- 那阶段总量就应该来自：
  - 候选 key 数
  - 候选批次数
  - 或同等可解释的增量集合
- 而不是重新扫描整个业务主表

对于 `unbound` 阶段，也应至少满足：

- 已完成量来自稳定可解释的处理单位
- 前端可以持续看到 `done` 推进
- 该阶段被明确标记为少数例外，而不是默认模式

### 7.5 原子块必须具备幂等性或可检测重复执行

首轮应固定一个核心要求：

- 每个原子块要么是幂等的
- 要么至少能检测自己是否已经成功应用

原因很直接：

- 暂停与继续可能让块边界被重新进入
- 失败重试可能让某些块再次被执行
- 异常中断或恢复重入时，需要防止重复写坏数据

常见可接受方式包括：

- `upsert`
- `insert ... on conflict do update`
- `update where version = expected`
- staging table + merge
- 写入前先比对 baseline / row hash
- 为块写入明确的 apply marker

但首轮至少要把这个原则写死：

- 不允许块的重复执行天然导致不可检测的数据破坏

- 幂等或可检测重复执行，不能只停留在“框架快照可重放”
- 它必须覆盖块内真实业务副作用
- 如果某个任务无法把业务写入设计成天然幂等
  - 就必须提供可检测的块级应用标记、版本比对或等价保护机制
- 不允许把“旧执行器理论上应该已经停了”当作防止重复写入的主要依据

### 7.6 失败边界必须按原子块定义

首轮要求任务失败语义必须以原子块边界为最小单位，而不是“跑到哪算哪”。

这意味着：

- 当前块执行前，系统知道自己要处理哪一批
- 当前块执行成功后，系统知道这批已经完成
- 如果块内失败，则必须明确：
  - 是整块回滚
  - 还是块内已有局部提交，但可被安全重放或识别

更推荐的首轮方向是：

- 对 `atomic` 块：
  - 每个原子块尽量包在单一一致性边界内
- 对 `reconcilable` 块：
  - 不强求跨域副作用与框架确认同事务
  - 但必须有可重复执行或可对账确认的完成语义
- 块成功，才推进 `done`
- 块失败，不推进 `done`

对 `atomic`，这里的“一致性边界”在首轮应明确包括：

- 该块的业务副作用
- 该块的 checkpoint 或恢复点推进
- 该块的 `done` 推进

对 `reconcilable`，首轮必须改成另一组要求：

- 该块的业务副作用可以先于框架确认单独生效
- 但任务必须能识别这些副作用是否已经成功应用
- 在恢复或重试时，必须能基于 apply marker、目标侧状态、版本锚点或等价机制决定：
  - 跳过已完成块
  - 重放未完成块
  - 或做必要对账

这样块级失败才真正意味着：

- 不是只有快照没前进
- 而是该块即使留下局部副作用，也不会留下不可解释的半成功结果

这样恢复和重试语义会清晰很多。

### 7.7 暂停与继续的安全边界

支持继续的任务必须满足：

- 每个块结束时都有稳定 checkpoint
- checkpoint 能唯一确定“从哪里继续”
- 持久化 checkpoint 本身必须独立表达“当前停在什么已确认边界”
- 对 `durable`
  - 继续前提不得依赖失效的内存上下文
- 对 `session_bound`
  - 除持久化 checkpoint 外，允许额外要求仍然存活的进程内上下文
- checkpoint 重新加载后，不需要重新解释一大片不确定中间态

也就是说，继续的最小语义应该是：

- 回到最近一个已确认完成的块边界
- 从下一个待执行块重新开始

而不是：

- 试图从某个块中间的半截内存状态继续

### 7.8 重试与继续必须严格区分

这一点要在文档里反复强调，因为很容易混淆。

#### 7.8.1 重试

- 创建新的 `TaskRun`
- 默认从头重新执行
- 不依赖旧 checkpoint

#### 7.8.2 继续

- 只适用于支持继续的同一运行实例
- 从已保存的恢复点继续
- 保留原有运行上下文和阶段状态

这两者的目的完全不同：

- 重试解决“这次运行失败了，再来一次”
- 继续解决“这次运行被安全停下了，从边界接着跑”

### 7.9 禁止的默认实现模式

首轮框架应明确禁止以下模式作为默认实现：

- `select * from big_table` 后在内存里全量比对
- 先清空目标表，再整表重建
- 不声明候选依据，只能靠运行时临时猜测处理范围
- 把大事务无限做大，以至于心跳、暂停、取消失真
- 块执行成功前就先推进进度
- 声称支持继续，但没有稳定 checkpoint
- 失败后靠人工猜测“可能执行到哪了”

这些不是“实现细节不好”，而是直接违背框架目标。

### 7.10 首轮允许的例外

虽然首轮推荐优先增量化，但仍然可以允许显式的全量任务存在，不过必须满足：

- 在任务定义中明确标注它是全量模式
- UI 上可明确区分它和普通增量任务
- 不允许把它包装成通用任务骨架的默认路径
- 后续若找到可信增量依据，应优先被重构掉

也就是说，框架可以容忍特例，但不能把特例当常规。

### 7.11 错误分类建议

首轮虽然不必做一整套复杂错误系统，但建议至少把错误分成下面几类：

- 业务错误
  - 参数非法、目标不存在、前置条件不满足
- 执行错误
  - SQL 执行失败、数据冲突、块处理失败
- 控制错误
  - 非法状态迁移、无效继续、无效暂停
- 运行时一致性错误
  - 心跳失效、状态不一致、恢复上下文失效
- 清理错误
  - 残留任务收束失败

这样后续前端与运维能更清楚地区分：

- 可以重试
- 不该重试
- 需要人工介入
- 需要框架修复

终态原因的建模边界如下：

- `status` 表达任务最终落入哪一类结果
- `terminalReason` 表达任务为什么落入该终态
- `errorCode / errorMessage` 只在确实存在错误诊断信息时承载错误细节，不应用来替代终态原因

### 7.12 前端展示上对错误的最小要求

前端至少需要知道：

- 当前任务是否失败
- 是普通失败还是被清理为 `abandoned`
- 是否允许重试
- 是否允许继续
- 最近错误信息是什么
- 若任务已终态，还应知道其 `terminalReason`

这要求后端在 `TaskRun` 层至少保存稳定的错误摘要，而不是只在日志里打印异常。

### 7.13 这一部分的核心结论

第七部分最关键的决定是：

- 框架默认要求任务声明可解释的执行策略，而不是替任务隐式猜测
- 增量执行是首轮推荐默认策略，但不是框架存在的唯一前提
- 每个任务都必须声明候选依据
- 每个阶段的总量也必须来自候选依据，而不是回退到不受控的全量扫描
- 原子块必须幂等或可检测重复执行
- 重试和继续严格分离
- 全量执行只能作为显式策略，不能伪装成默认路径

## 首轮落地范围与模块映射

为避免后续实现时把“通用框架”和“任务专用逻辑”重新混在一起，首轮先固定一版从设计对象到现有代码位置的实现映射。

这一节不新增需求，只把当前 proposal 已经选定的首轮范围收束成可执行边界。

### 1. 首轮明确包含的范围

首轮实现明确包含以下内容：

1. desktop runtime 内部的统一重量级任务框架
2. 本地数据库中的权威任务快照表与阶段快照表
3. 通用控制接口、快照接口与实时进度通道
4. 启动清理、周期巡检、心跳与等待态重试推进
5. `publish` 作为首个 `reconcilable` 任务接入
6. `reanchor` 作为首个 `atomic` 任务接入
7. desktop publish 页面切换到统一 `pageTask` 读模型
8. console 侧统一任务组件与全局任务检视 popup

### 2. 首轮明确不包含的范围

首轮实现明确不包含以下内容：

1. `site-console`、`service-internal` 或其他远端 runtime 的统一任务框架接入
2. 外部 job system、独立 worker 服务或跨进程分布式调度
3. 通用事件溯源表、完整控制命令审计表或原子块历史表
4. 除 `publish` / `reanchor` 外的其他任务全面迁移
5. 批量渲染等外部副作用任务的首轮接入
6. 自动断点续跑执行中的任务
7. 多任务公平排队、优先级调度或 backlog 消化模型
8. 独立于任务框架语义之外的第二套任务 UI 或第二套任务状态解释模型

### 3. 通用框架层的现有代码映射

首轮通用框架应以 `apps/service-desktop-runtime` 为唯一运行承载面，优先收敛到新的统一模块，而不是继续扩展现有专用 publish 控制流。

建议映射如下：

- runtime HTTP / RPC 入口
  - 现有入口位于 `apps/service-desktop-runtime/src/index.ts`
  - 负责本地 Bun 服务监听与 `/rpc/*` 暴露
  - 首轮不改变这一层运行形态
- runtime ORPC 聚合入口
  - 现有入口位于 `apps/service-desktop-runtime/src/orpc/service.ts`
  - 当前聚合 `runtime / hsdata / image / tag`
  - 后续通用任务路由应在这里注册，而不是继续只塞进 `hsdata`
- hsdata 专用任务接口入口
  - 当前主要位于 `apps/service-desktop-runtime/src/orpc/hsdata.ts`
  - 这里现在同时承担：
    - publish / reanchor 控制入口
    - import / projection 入口
    - 进度事件流暴露
  - 首轮目标不是继续扩张这个文件，而是把重任务框架语义拆出去
- 统一任务框架模块目标位置
  - 计划中的新增位置是 `apps/service-desktop-runtime/src/lib/tasks/` 或等价目录
  - 其中承载：
    - `definition`
    - `registry`
    - `store`
    - `scheduler`
    - `executor`
    - `control`
    - `snapshot`
    - `events`
    - `cleanup`

### 4. 现有任务专用逻辑的迁移映射

首轮不从零发明 `publish` / `reanchor` 业务逻辑，而是把它们从现有专用控制流迁移到统一框架。

当前映射如下：

- `publish` 业务主逻辑
  - 现有位置：`apps/service-desktop-runtime/src/lib/hearthstone/hsdata-publish.ts`
  - 后续角色：
    - 保留业务计算、差异推导、远端写入与批次收束逻辑
    - 不再继续持有独立的任务状态机与进度真相
- `publish` 进度与控制器
  - 现有位置：`apps/service-desktop-runtime/src/lib/hearthstone/hsdata-publish-progress.ts`
  - 当前问题：
    - 维护的是进程内单 job 状态
    - 更接近 publish 专用内存控制器，而不是数据库权威快照
  - 后续角色：
    - 被统一任务框架替代
    - 只保留可复用的进度字段映射经验，不继续作为权威实现
- `reanchor` 入口
  - 当前挂在 `apps/service-desktop-runtime/src/orpc/hsdata.ts` 中，与 publish 共用专用控制流
  - 后续角色：
    - 拆成独立 `TaskDefinition`
    - 通过统一控制接口和统一执行器运行
- import / projection 相关逻辑
  - 当前位于 `hsdata-import.ts`、`hsdata-project.ts`、`hsdata-progress.ts`
  - 首轮不迁移到统一框架
  - 只保留为后续任务接入的参考样本

### 5. 前端页面与读模型映射

首轮统一任务页面读模型只先落在 publish 页面。

但首轮前端共享层不只包含页面本身，还包含一组任务框架共用 UI：

- 页面级任务组件
- 标题栏任务入口按钮
- 全局任务 popup

当前映射如下：

- publish 页面
  - 现有位置：`apps/app-console-desktop/src/pages/hearthstone/publish/index.vue`
  - 当前问题：
    - 页面直接消费 publish 专用状态与事件流
    - 控制能力和页面心智仍绑定在专用 publish job 语义上
  - 首轮目标：
    - 改为消费统一快照接口返回的 `pageTask`
    - 改为消费统一能力位 `canPause / canResume / canCancel`
    - 改为复用统一任务组件，而不是页面内继续维护 publish 专用任务卡
    - 继续保留 publish 自己的业务表单输入与结果展示
- publish 设置页
  - 现有位置：`apps/app-console-desktop/src/pages/settings/publish.vue`
  - 首轮不作为任务框架接入目标
  - 仅在其配置结果被 publish `taskScopeSnapshot` 或参数读取时间接受影响
- 全局任务检视入口
  - 首轮新增一个标题栏任务按钮与 popup
  - 该入口不绑定单一 task page
  - 它读取当前全局占槽任务摘要，并提供快速查看与控制能力
  - popup 只展示轻量摘要，不替代任务页面的完整细节区

### 6. 数据库 schema 映射

首轮统一任务快照是 local track 内的 runtime 数据，而不是 remote 控制面数据。

因此数据库映射固定为：

- schema source / deployment track
  - 落在 `packages/db/src/schema/local/`
- 首轮表
  - `task_runs`
  - `task_stages`
- 不继续扩展现有 `publish` 专用表来承载通用任务框架语义
  - `publish` 现有批次与基线表仍保留其业务数据职责
  - 通用任务状态与阶段快照由新表承担

### 7. 首轮实现边界结论

首轮实现应始终保持下面这条边界不变：

- 通用框架负责：
  - 状态机
  - 快照
  - 调度
  - 执行循环
  - 控制语义
  - 实时推送
  - 清理语义
- 每个任务单独负责：
  - `TaskDefinition`
  - 阶段计划
  - 总量构建
  - 候选冻结
  - 原子块生成
  - 原子块业务执行

也就是说，首轮目标是“一套通用运行框架 + 多个任务定义接入”，而不是为 `publish`、`reanchor` 分别维护独立运行骨架。
