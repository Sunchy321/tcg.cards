# desktop 配置文件 JSON 编辑器与配置入口设计评审

> 稳定的运行时边界、数据归属和 workspace 责任以 [../../docs/project-architecture.zh-CN.md](../../docs/project-architecture.zh-CN.md) 为准。本文只记录本需求的评审结论；若有冲突，以主架构文档为准。

## 评审结论

建议通过。

这份设计解决的是一个真实的 desktop 配置管理问题：

- 表单入口只适合高频稳定项
- 原始配置文件已经成为真实数据源
- 但当前还缺少直接查看、编辑和定位该文件的能力

在这个前提下，同时提供：

- 表单入口
- 原始 JSON 编辑器
- 配置文件路径入口

是合理的增量方向。

## 阻塞问题

无。

## 主要判断

### 1. 原始 JSON 编辑器是表单设置的补充，而不是替代

这点判断是正确的。

如果直接把设置页改成只剩一块 JSON 文本框，会让普通用户的高频设置体验显著变差；如果继续只保留表单，又会让配置演进速度受限。

两层入口并存是更稳的折中：

- 普通用户用表单
- 高级用户与排障场景用原始 JSON

### 2. 配置文件入口应该显式暴露，而不是继续隐藏真实存储位置

既然本地数据库连接已经明确放入配置文件，就没有必要继续把文件位置藏起来。

提供：

- 打开配置文件夹
- 复制配置文件路径

可以明显降低排障成本，也更符合 desktop 应用的预期。

### 3. Monaco 更适合当前需求

当前需求已经明确是“接近 VS Code 的配置文件源码编辑”。

在这个前提下，Monaco 比轻量编辑器更匹配：

- 原文编辑手感更强
- JSON diagnostics 更成熟
- 基于 schema 的 key 提示与校验路径更直接

它的成本更高，但这次需求优先考虑效果，这个取舍成立。

### 4. 需要明确区分“配置文件内容”和“系统凭据内容”

设计中保留了这个边界，这点很重要。

否则用户会误以为：

- 所有 desktop 凭据都能在 JSON 文件里看到

但实际仍然存在：

- publish target connection string
- 登录相关凭据

这类继续留在 keyring 的内容。

## 非阻塞建议

### 1. 原始编辑器的保存逻辑要优先保证“不丢未知字段”

这是原始编辑入口最重要的性质之一。

只要保存时不是“整段文本写回”，而是先映射成一个受限模型再序列化，就很容易把未知字段丢掉。实现时需要避免这类回写方式。

### 2. key 提示应控制在“静态提示”级别

首轮不要尝试完整 JSON Schema IDE 体验。

更稳的范围是：

- 顶层 key
- 常见嵌套 key
- 简单结构引导

这样可以显著降低实现和维护成本。

### 3. 配置文件路径入口优先做文件夹而不是文件本身

“打开配置文件夹”跨平台更稳，也更少依赖系统默认编辑器行为。

“打开配置文件”可以作为可选增强项，而不是首轮阻塞项。

## 通过条件

- 原始 JSON 编辑器定位为高级入口，不替代表单设置
- 保留配置文件与 keyring 的边界提示
- 至少提供打开配置文件夹与复制路径能力
- 保存流程保证未知字段不丢失
