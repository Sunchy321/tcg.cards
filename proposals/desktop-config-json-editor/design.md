# desktop 配置文件 JSON 编辑器与配置入口设计

> 稳定的运行时边界、数据归属和 workspace 责任以 [../../docs/project-architecture.zh-CN.md](../../docs/project-architecture.zh-CN.md) 为准。本文只描述 desktop 配置文件编辑与定位入口的增量设计；若有冲突，以主架构文档为准。

## 背景

当前 desktop 已经把一部分本地配置收敛到了 `desktop-config.json`，例如：

- 本地数据库连接字符串
- 各游戏 repo 路径
- 其他 desktop 本地设置

但配置入口仍主要以分散的表单字段存在于设置页中。这会带来几个直接问题：

- 高级用户无法一次性查看完整配置结构
- 新增配置项时，必须先补一套单独表单，迭代成本偏高
- 排障时很难快速定位“这个值到底写进了哪个文件”
- 当前设置页只能操作“已显式暴露”的字段，不能处理未知字段或临时扩展字段

与此同时，本地数据库连接已经改为普通配置文件存储，不再通过系统 keyring 读取。既然 `desktop-config.json` 已经成为真实配置来源，就需要补齐：

- 原始 JSON 编辑能力
- 配置文件 / 配置目录入口

## 目标

- 在 desktop 设置页中新增原始 `desktop-config.json` 编辑器
- 提供配置文件路径入口，便于查看、排障和外部编辑
- 在不破坏未知字段的前提下支持直接读写整个配置文件
- 为常见 key 提供基础提示和结构引导

## 非目标

- 不把设置页做成完整的通用 IDE
- 不把 keyring 中的 publish target 或登录凭据迁移到 JSON 文件
- 不在本轮提供 JSON Schema 文件下载或外部共享
- 不在本轮做配置版本迁移器

## 现状问题

### 1. 设置页只覆盖“已知字段”

当前设置页中的数据库连接、repo 路径等配置项都是单独表单。这种方式适合高频基础项，但不适合：

- 临时实验字段
- 尚未稳定的嵌套配置
- 需要手工修复的异常配置

### 2. 缺少配置文件可见性

用户虽然可以通过设置页修改值，但无法直接知道：

- 配置文件存在哪里
- 当前文件内容是什么
- 是否需要手工比对或备份

这会增加排障成本。

### 3. 表单扩展速度慢于配置演进速度

每增加一个配置项，都要补：

- 前端字段
- Tauri 命令
- 保存逻辑
- 校验文案

这适合稳定设置，但不适合频繁变化的 desktop 本地配置。

### 4. 配置文件已成为真实数据源

当前 `desktop-config.json` 已经不只是导出视图，而是本地设置的真实权威来源。既然如此，就应允许高级用户直接编辑它，而不是只能绕道表单。

## 设计决策

### 1. 在数据库设置页增加原始配置入口，并使用独立页面承载编辑器

在 `/settings/database` 标题右侧新增固定入口：

- `编辑原始配置`
- `打开配置文件夹`

点击 `编辑原始配置` 后进入独立页面，例如 `/settings/config-file`。该页面：

- `desktop-config.json` 的当前路径
- 打开配置文件夹按钮
- 复制配置文件路径按钮
- JSON 编辑器
- 重新加载按钮
- 保存按钮

这样可以避免两个问题：

- 在数据库表单页中同时显示大块编辑器，页面会过于拥挤
- 原始 JSON 编辑是高级操作，不应与按项配置混在同一阅读流里

原始配置页不显示左侧设置边栏，直接把空间留给编辑器本体。

### 2. 编辑器采用 Monaco，并接入 JSON schema 提示与校验

本轮采用 Monaco，理由是当前需求已经明确偏向“接近 VS Code 的原文编辑体验”，需要：

- 更强的原文编辑手感
- 明确的错误标记和 diagnostics
- 基于 JSON schema 的 key 提示
- 基于 JSON schema 的基础校验

Monaco 更适合这种需求，因为它本身就是 VS Code 编辑器能力的独立抽取版本。

首轮编辑器至少支持：

- JSON 语法高亮
- 错误标记
- 格式化
- 常见 key 自动提示
- 基础类型校验

本轮仍然只服务于 `desktop-config.json`，不扩展为通用 IDE 框架。

### 3. 只对配置文件本身做整文件读写

新增一组专门的 desktop 配置命令，例如：

- `desktop_get_raw_config()`
- `desktop_set_raw_config(jsonText)`
- `desktop_get_config_file_info()`

其中：

- `desktop_get_raw_config()` 返回当前配置文件的原始 JSON 文本与路径信息
- `desktop_set_raw_config(jsonText)` 负责解析、校验并原样写回
- `desktop_get_config_file_info()` 返回：
  - 配置文件绝对路径
  - 配置目录绝对路径
  - 文件是否存在

这样可以保证：

- 表单模式和原始编辑模式共享同一个底层配置源
- 原始编辑器不会丢失未知字段

### 4. 保留表单设置，不用 JSON 编辑器替代表单

原始 JSON 编辑器是高级入口，不替代表单设置。

最终应保持两层入口并存：

- 表单设置
  - 面向高频常用项
  - 适合普通用户
- 原始配置
  - 面向高级用户与排障场景
  - 适合一次性查看、修复或批量调整

这样可以避免设置页直接退化成“只有一块 JSON 文本框”。

### 5. 提供基础 key 提示和宽松 JSON schema 校验

编辑器中的提示和校验采用 Monaco JSON schema diagnostics。

建议至少覆盖：

- 顶层：
  - `version`
  - `externalDatabaseConnectionString`
  - `games`
- 常见嵌套：
  - `games.hearthstone`
  - `games.hearthstone.hsdata.repoPath`
- `games.hearthstone.publish.publishTargetId`
- `games.hearthstone.publish.environment`
- `games.hearthstone.publish.targetFingerprint`

但 schema 需要保持宽松：

- 顶层和主要嵌套对象提供 `properties`
- 大部分对象保留 `additionalProperties: true`

这样可以同时满足两点：

- 已知字段有明确提示和类型校验
- 未知字段、临时字段、未来扩展字段不会被硬拦住

### 6. 配置文件入口至少包含“打开文件夹”和“复制路径”

本轮建议提供：

- `打开配置文件夹`
- `复制配置文件路径`

可选再提供：

- `在默认编辑器中打开配置文件`

其中“打开配置文件夹”是必须项，因为它最稳，也最符合排障需求。

### 7. 原始保存前必须经过编辑器校验与写盘前最小结构校验

原始编辑器不能把任意文本直接写盘。保存前至少需要：

1. JSON 解析成功
2. 顶层必须是对象
3. 若存在 `version`，必须是数字
4. 若存在 `games`，必须是对象

本轮不强制对每个嵌套字段做完全严格校验，因为这会再次把原始编辑模式变回“强表单模型”。

但至少要保证文件整体仍然是一个合法配置对象。

前端 Monaco 校验负责：

- 提示错误位置
- 提供已知 key 补全
- 暴露基础类型错误

Tauri 保存前校验继续负责最终兜底，避免前端绕过时写入非法内容。

### 8. 原始编辑不处理 keyring 项

必须明确一个边界：

- `desktop-config.json` 不包含所有 desktop 凭据

仍留在 keyring 的内容，例如：

- publish target connection string
- 登录相关凭据

不应出现在原始 JSON 编辑器中。

页面文案需要明确提示：

- “该编辑器只修改配置文件中的内容”
- “系统凭据存储中的敏感值不在这里展示”

## 交互流程

### 1. 打开页面

- 读取配置文件路径信息
- 读取当前原始 JSON 文本
- 在编辑器中展示格式化后的文本

### 2. 编辑但未保存

- 本地维护脏状态
- 页面显示“未保存更改”
- 切换页面或刷新前提示确认

### 3. 保存

- 先在前端解析 JSON
- 若失败，停留在编辑器并显示错误
- 若成功，调用 `desktop_set_raw_config(jsonText)`
- 保存成功后刷新：
  - 原始文本
  - 表单视图依赖的当前值

### 4. 重新加载

- 放弃未保存更改
- 重新读取磁盘上的配置文件内容

### 5. 打开文件夹 / 复制路径

- 读取 `desktop_get_config_file_info()`
- 调用 desktop 本地能力：
  - 打开目录
  - 复制路径

## 风险与取舍

### 1. 原始编辑会绕过表单级语义约束

这是预期取舍。

原始编辑器的目的就是允许高级用户直接处理配置文件，因此它不应被表单逻辑完全限制。但仍需保留最小 JSON 与结构校验，避免彻底写坏文件。

### 2. 编辑器补全不能一开始做得过重

如果首轮就追求：

- 完整 schema 验证
- 深层自动补全
- 重构级编辑体验

复杂度会明显失控。因此本轮只做静态 key 提示。

### 3. 文件路径入口会让用户更容易接触明文配置

这也是预期取舍。

既然本地数据库连接已经决定放入配置文件，就不应再假装它不可见。更合理的做法是：

- 明确边界
- 提供入口
- 让高级用户能够理解真实存储位置

## 验收标准

- desktop 设置页存在原始 `desktop-config.json` 编辑器
- 设置页能展示配置文件路径并提供至少一个打开入口
- 原始编辑器支持：
  - 语法高亮
  - JSON 错误提示
  - 保存
  - 重新加载
- 编辑器能提供基础 key 提示
- 保存原始配置时不会丢失未知字段
- 页面文案明确区分：
  - 配置文件内容
  - keyring 中的敏感凭据
