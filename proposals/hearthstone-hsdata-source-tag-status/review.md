# hsdata sourceTag 导入与投影状态展示设计评审

> 稳定的运行时边界、数据归属和同步规则以 [../../docs/project-architecture.zh-CN.md](../../docs/project-architecture.zh-CN.md) 为准。本文只记录本需求的评审结论；若有冲突，以主架构文档为准。

## 评审结论

建议通过。

当前设计已经从“前端临时推导投影状态”收敛为“后端持久化状态 + 前端直接展示状态”，方向正确，且和现有 desktop 导入页的工作流一致。

## 阻塞问题

无。

## 主要判断

### 1. 将投影状态持久化到 source_versions 是必要的

如果继续从领域表按 `build` 反推：

- 多个 `sourceTag` 共享同一 `build` 时会串扰
- 无法稳定表达 `processing`、`failed`、错误信息和成功时间
- 前端展示会依赖隐式推断，后续很难维护

当前改为直接读取 `projectionStatus`、`projectedAt`、`projectionError`，口径更稳定。

### 2. 首轮只接入 desktop 导入页是合理的范围控制

当前真正需要这个状态的人，是在本地执行导入和投影的人。

把首轮展示落在现有导入页，而不是同时扩到 site-console，可以减少：

- 接口扩散
- 重复 UI
- 还没验证交互前就提前固化多个入口

### 3. 前端按 sourceTag 合并本地来源和数据库状态是正确的接法

当前页面右侧已有“可用版本”列表，用户已经在这个列表里选择来源。

因此更合理的做法是：

- 保留现有本地来源列表
- 再加载 `source_versions` 状态
- 按 `sourceTag` 合并后展示 badge

而不是再额外生成一份数据库 sourceTag 表，否则页面会出现两个近似列表，增加判断成本。

## 非阻塞建议

### 1. sourceTag 缺失的本地来源要有明确降级展示

当本地来源暂时无法解析出 `sourceTag` 时，页面不应强行匹配数据库状态。

建议展示为：

- 可选择来源
- 但不显示导入/投影状态
- 文案上明确这是“未解析 sourceTag”，不是“未导入”

### 2. 投影失败信息优先放在选中详情区

`projectionError` 往往较长，不适合直接塞进右侧列表。

首轮让右侧列表只承担“状态提示”，把具体错误放到当前选中来源详情区，信息密度更合适。

### 3. 状态刷新时机必须写进实现计划

这个页面同时依赖本地文件和数据库状态，最容易出问题的不是展示组件，而是刷新时机。

至少需要在以下时机统一刷新状态：

- 页面初次进入
- 同步远端版本后
- 导入完成后
- 投影完成后

否则页面很容易出现“刚操作完，但 badge 还是旧值”的错觉。

## 通过条件

- 实现阶段必须继续以 `source_versions` 为唯一状态来源
- 前端不得再按 `build` 临时推导是否已投影
- 首轮保持在 desktop 导入页内完成，不扩展到新的 sourceTag 管理页面
