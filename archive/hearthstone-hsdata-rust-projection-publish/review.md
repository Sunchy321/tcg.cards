# hsdata Rust 投影与远端发布评审

## 评审结论

通过。

当前设计已经满足进入 `specs/` 的条件，可以作为后续实现的正式依据继续推进。

## 认可点

### 1. 运行时边界已经明确

设计清楚区分了：

- desktop Rust 负责本地导入、投影、发布准备
- 本地 PostgreSQL 负责构建权威结果
- remote 负责 serving 与最小发布控制面

这与当前项目的整体架构方向一致，没有继续保留“重型流程跨 TS / Rust 分裂执行”的灰区。

### 2. 投影与发布术语已经拆开

设计没有把“本地投影”和“远端更新”继续混称为 projection，而是把远端更新明确收口到 `publish`、`batch`、`manifest`、`apply` 这组术语中。

这有助于后续页面、状态字段和错误信息保持语义稳定。

### 3. 发布模型的粒度选择合理

当前按整卡 row family 发布：

- `entities`
- `entity_localizations`
- `entity_relations`

这比逐字段或逐行级别的细粒度合并更容易保证一致性，也更符合现有本地投影结果的组织方式。

### 4. 发布侧表先按有限范围建模是合理的

当前 `publish_batches`、`publish_batch_cards`、`publish_baselines` 并不是任意表发布框架，而是面向当前 Hearthstone card family 的受控发布模型。

这个边界是合理的，避免了过早抽象。

## 需要继续遵守的约束

### 1. 发布模型不要被误解释为通用任意表发布系统

虽然命名已经改成 `publish_*`，但当前模型仍然只适用于受控的有限表集合。

后续实现步骤 6-8 时，必须继续保持这个边界，不要在没有重新设计资源模型前，把它扩展成“任意表、任意主键、任意依赖关系”的通用发布框架。

### 2. Rust 投影中的硬编码兼容映射需要继续收口

当前 Rust 投影里为了补齐历史缺失 enum 兼容，已经存在若干硬编码 map。

这些 map 可以作为兼容层存在，但不应继续分散在主投影流程里。后续实现应把这类兼容映射集中到单独位置，避免：

- 规则分散
- 维护时漏改
- 很难区分“表驱动规则”和“运行时兼容补丁”

### 3. `multiclass` 历史编码风险不能在运行时靠猜测兼容

当前实现可以默认把 `MULTIPLE_CLASSES` 当作位掩码处理，但评审中已经确认，旧项目遗留资料里还存在一套不等价的历史组合枚举编码。

这意味着后续如果要补老版本兼容：

- 不能根据数值大小或局部模式做模糊判断
- 不能把两套编码混成一个“自动猜测”的通用分支
- 应优先按 build / sourceTag 分段决定兼容规则

## 风险判断

当前最大的剩余风险不是表结构，而是后续实现时的两类漂移：

- Rust 与既有 TS 基线之间的行为漂移
- `publish` 模型在实现中被不受控地泛化
- `multiclass` 在旧版本样本上可能出现历史组合枚举编码

只要后续继续以基线对照和有限范围发布为前提推进，这两个风险是可控的。

## 结论

该设计包通过评审，进入 `specs/`。

后续实现应继续按当前设计推进，并优先处理：

1. publish target profile 与批次绑定校验
2. Rust 投影硬编码兼容映射的集中化
