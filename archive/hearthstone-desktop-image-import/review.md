# 桌面端炉石图片导入与本地渲染端联动评审

> 稳定的运行时边界、数据归属和工作区职责以 [../../docs/project-architecture.zh-CN.md](../../docs/project-architecture.zh-CN.md) 为准。本文只记录本需求的评审结论；若有冲突，以主架构文档为准。

## 评审结论

建议通过。

这次改动的核心是把桌面端补齐为“本地图片资产闭环入口”，与当前桌面端承担本地数据处理、本地仓库访问和本地运行时编排的职责一致。

## 阻塞问题

无。

## 主要判断

### 1. 桌面端应通过 runtime 调用渲染端，而不是前端直连

如果前端直接访问渲染端：

- 二进制传输会落在浏览器层
- 本地 bucket 写入仍需要绕回 Tauri 或命令行
- 错误处理会分散在前端和本地脚本之间

由 runtime 统一负责更符合当前边界。

### 2. 新增图片配置应进入 `games.hearthstone.image`

渲染端地址和 bucket 目录都明显是炉石图片流程的游戏级配置，不应塞入全局数据库设置或 publish target 设置。

放进：

- `games.hearthstone.image.rendererBaseUrl`
- `games.hearthstone.image.bucketDir`

结构最清楚，也便于后续扩展更多图片相关配置。

### 3. 现有本地脚本逻辑必须收敛为共享核心

如果 desktop runtime 重新手写一套：

- key 规则可能分叉
- PNG 校验和转码规则可能不一致
- CLI 与桌面端后续维护成本会持续升高

因此共享核心模块是必要约束，不应只做代码复制。

### 4. 页面与 Web 控制台保持“形态一致、流程不同”是合理折中

用户真正需要的一致性是：

- 相同筛选项
- 相似的状态反馈
- 相近的操作位置

而不是强行保留 ZIP 导入这类不适合桌面端的交互。

### 5. 本地 bucket 到 R2 同步应与本地导入解耦

把“本地落盘”和“远端同步”塞进同一个任务会让失败语义变差：

- 本地图片已经可用，但远端同步失败时很难表达最终状态
- 远端同步重试不应要求重新渲染
- 本地结果验证和远端发布验证应分别进行

因此把“同步到 R2”设计为独立动作是合理的。

## 非阻塞建议

### 1. 首版先固定一个最小渲染端协议

不要一开始就做通用任务框架。

首版只要满足：

- 提交 requirements
- 查询任务状态
- 拉取 PNG 结果

就足够支撑桌面图片页。

### 2. 图片任务状态建议保留在 runtime 内存中

即使首版不做数据库持久化，也建议 runtime 在进程内保留当前任务摘要。

这样页面刷新后仍有机会重新读取最近状态，而不是完全丢失上下文。

### 3. 设置页里应加入最小连接测试

只保存 URL 和目录还不够，至少要支持：

- 测试渲染端 Base URL 是否可达
- 检查 bucket 目录是否存在或是否可创建

否则图片页上的失败体验会过于靠后。

## 通过条件

- 桌面端图片页替换占位页，并具备导出与任务触发入口
- 桌面设置页新增渲染端地址和 bucket 目录配置
- desktop runtime 能读取图片配置并调用渲染端
- 本地导入逻辑与现有 CLI 规则保持一致
- 本地 bucket 能通过独立同步动作推送到 R2
- 小批量任务可以成功写入本地 bucket 目录
