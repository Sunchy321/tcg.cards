## 目标

把万智牌导入体系的领域边界在 P0 阶段一次性收口，确保 P1 开始时可以直接进入数据库建模与迁移，而不会在来源职责、字段路径、匹配规则、执行模式或审批事实来源上反复返工。

## 实施原则

- 只做万智牌，不做跨游戏抽象
- 只做结构化配置，不做脚本化规则
- 先固定边界，再进入迁移与编码
- 用户无关留痕落 `magic_data`，用户审批动作落 `magic_app`，不再继续扩展主表 `__updations`
- `__lockedPaths` 优先级高于任何自动应用规则
- 来源未覆盖或单条记录未提供的字段不得被解释为 `null`
- 前端只读页面读取已发布快照，不执行导入、不审批、不写库

## 阶段拆解

### 1. 来源目录收口

**目标**

确认 P0 的来源范围、来源职责、默认策略与默认执行模式。

**任务**

- 固化 `magic/gatherer`、`magic/scryfall`、`magic/mtgch`、`magic/mtgjson`
- 明确每个来源负责的实体范围
- 明确每个来源的默认策略与默认执行模式
- 明确 `mtgjson` 在 P0 中仅用于对账与候选生成

**输出物**

- 来源目录常量
- 来源职责说明
- 来源默认配置样例

**验收标准**

- 任意来源都能回答“负责哪些字段、默认走哪种执行模式、是否允许自动应用”

### 2. 实体与匹配边界收口

**目标**

把导入候选限定到 6 类实体，并固定每类实体的主键与允许匹配键。

**任务**

- 固化 `card`、`cardLocalization`、`cardPart`、`cardPartLocalization`、`print`、`printPart`
- 为每类实体定义 `targetKey`
- 为每类实体定义允许的 `matchKey`
- 明确禁止模糊卡名匹配

**输出物**

- 实体类型定义
- `targetKey` / `matchKey` 校验模型
- 未匹配处理约束

**验收标准**

- 任意标准化记录只能匹配 0 或 1 个目标实体
- 不存在“先模糊匹配再覆盖”的兜底逻辑

### 3. 字段路径注册表

**目标**

建立万智牌专用字段路径注册表，作为规则、审批和展示的共同基础。

**任务**

- 按 6 类实体梳理可导入字段
- 标记主键字段、系统字段与保留字段
- 支持占位符字段路径，如 `card.legalities.{format}`
- 设计字段路径校验失败时的错误模型

**输出物**

- 字段路径注册表
- 占位符路径模型
- 字段元信息结构

**验收标准**

- 规则配置中引用未注册字段时加载直接失败
- 审批与批量分组都能基于注册表获取字段元信息

### 4. 来源覆盖与字段缺失语义收口

**目标**

明确来源字段覆盖能力和单条标准化记录的字段状态，避免把缺失字段误生成“清空”候选。

**任务**

- 固化 `supported`、`conditional`、`unsupported` 三类来源覆盖状态
- 固化 `provided`、`explicit_null`、`not_provided`、`not_applicable`、`parse_failed` 五类字段状态
- 明确 `not_provided`、`not_applicable`、`parse_failed` 不生成 `FieldChange`
- 明确 `explicit_null` 只能在字段可空且规则允许时参与 diff

**输出物**

- 来源字段覆盖矩阵结构
- 标准化字段状态模型
- diff 输入过滤规则

**验收标准**

- 来源不支持字段不会进入 diff
- 单条记录缺失字段不会进入审批队列
- 来源明确空值与来源未提供字段在数据结构上可区分

### 5. matcher schema 收口

**目标**

限制 `overwrite_when_matched` 的表达能力，确保后续实现简单、可解释、可审计。

**任务**

- 固化 `is_null`、`is_empty_string`、`equals`、`in`、`contains_any`
- 固化 matcher 输入值类型
- 固化 matcher 组合方式与 `fallbackAction`
- 禁止脚本、正则、跨字段表达式

**输出物**

- matcher zod schema
- matcher 评估约束
- `fallbackAction` 枚举

**验收标准**

- 任意 matcher 都能在不执行脚本的情况下解释和序列化
- 审批页可以直接展示 matcher 命中原因

### 6. 决策矩阵收口

**目标**

把万智牌来源 × 字段族 × 风险等级的默认决策固化下来。

**任务**

- 明确 `auto_apply` 白名单字段
- 明确 `batch_review` 的字段族与分组键要求
- 明确 `manual_review` 的高风险字段边界
- 明确 `approval_required` 的兜底行为

**输出物**

- 来源默认策略矩阵
- 字段族风险矩阵
- 决策模式判定顺序

**验收标准**

- 任意候选字段都能得到可解释的默认执行模式
- 不存在“未命中任何规则但仍自动应用”的路径

### 7. 配置快照与前端只读页面

**目标**

设计配置快照生成机制和只读页面读取方式，展示万智牌数据源、字段来源状态与导入策略，帮助维护者在实现导入链路前确认配置是否合理。

**任务**

- 定义快照生成时机，如发布配置后生成
- 定义快照编译校验与版本号规则
- 定义快照输出结构与存储位置
- 复用或扩展 `apps/site-console/app/pages/magic/data-source/index.vue`
- 展示 4 个 P0 来源的职责、默认策略与默认执行模式
- 展示字段来源状态矩阵
- 展示 `sourceId + fieldPath` 的导入策略矩阵
- 提供来源、实体类型、字段路径、覆盖状态、执行模式筛选
- 明确页面只读取当前生效快照，不提供规则编辑、导入执行、审批操作或写库能力

**输出物**

- 快照信息架构
- 快照结构与版本规则
- 页面筛选与只读边界说明

**验收标准**

- 可以查看每个来源覆盖哪些字段
- 可以查看每个字段在不同来源下是 `supported`、`conditional` 还是 `unsupported`
- 可以查看每个字段来源组合的 `strategy` 与 `decisionMode`
- 页面读取的是已发布快照，而不是实时规则表
- 页面不会触发任何导入、审批、自动应用或数据库写入行为

### 8. 事实边界与遗留兼容

**目标**

固定导入链路中哪些表是事实来源，哪些状态只是缓存，同时定义与旧机制的衔接方式。

**任务**

- 固化 `magic_data` / `magic_app` 表归属
- 固化 `import_review_actions` 与 `import_apply_logs` 的 authoritative 角色
- 固化 `decisionStatus` 的派生定位
- 固化 `__lockedPaths` 的硬约束行为
- 固化 `__updations` 的停用方向

**输出物**

- 事实边界说明
- 状态回写规则
- legacy 兼容说明

**验收标准**

- 审批、应用、回滚的唯一事实来源清晰
- 新链路不会继续污染主表 `__updations`

### 9. P1 输入清单

**目标**

把 P0 的结论沉淀成 P1 migration 可直接消费的输入。

**任务**

- 列出需要新增的 `magic_data` / `magic_app` 表
- 列出每张表必须包含的关键字段
- 标出日志冷热分层需要预留的字段
- 给出首批来源配置样例

**输出物**

- P1 migration 输入清单
- 首批 seed/config 草案

**验收标准**

- P1 可以直接开始 schema 设计与 migration 编写
- 不需要再回头讨论 P0 的领域边界

## 风险与缓解

### 风险 1：字段路径膨胀

- 影响：规则难维护，审批分组不可控
- 缓解：通过注册表和字段族元信息控制路径数量，不允许自由拼接字符串

### 风险 2：来源职责重叠

- 影响：同一字段在多个来源之间来回覆盖
- 缓解：先固定来源职责，再允许字段级 override

### 风险 3：自动应用范围过大

- 影响：错误写入主领域表
- 缓解：P0 只对白名单低风险字段开放 `auto_apply`

### 风险 4：遗留机制与新机制并存过久

- 影响：审批事实分裂
- 缓解：新导入链路不再写 `__updations`，并尽早规划迁移

## 建议顺序

1. 先输出来源目录、实体边界与字段路径注册表
2. 再输出来源覆盖能力、字段缺失语义与 matcher schema
3. 然后输出决策矩阵、配置快照与前端只读页、事实边界与 legacy 兼容
4. 最后整理 P1 migration 输入，进入数据建模阶段
