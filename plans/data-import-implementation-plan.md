## 目标

建立一套统一的数据导入框架，使系统能够从多个外部来源接收数据，在字段级策略与风险分层控制下，走自动应用、批量审批或逐条审批三类路径，并在全量留痕的前提下安全更新主领域表。

## 实施原则

- 所有更新先留档，再按执行模式决定自动应用或审批
- 来源适配器与规则引擎解耦
- 规则负责生成建议动作，执行模式负责决定自动应用、批量审批或逐条审批
- 主领域表只允许由自动应用服务或审批后应用服务写入
- 自动应用必须内建可回滚能力
- 第一版优先结构化配置，避免引入脚本化规则系统

| P2 配置与规则读取 | 让来源与规则可配置 | 实现规则集读取、来源绑定、字段规则匹配顺序、默认动作回退 | Config service / rule service | P1 | 可为不同来源返回不同字段规则 |
| P3 导入留档主链路 | 打通“拉取到留档” | 写入 `import_runs`、原始载荷、标准化记录；支持失败记录与幂等重跑 | Import service、raw record pipeline | P1 | 导入失败可追踪；重复运行不丢留痕 |
| P4 匹配与 diff | 生成可执行候选 | 实现实体匹配、字段级 old/new diff、变更集聚合、批量分组键计算 | Diff service、change set service | P2, P3 | 能稳定产出 `changeSet + fieldChange + batchGroupKey` |
| P5 策略评估与执行模式 | 让规则真正生效 | 实现 `overwrite`、`ignore`、`overwrite_when_matched`、`approval_required` 的建议动作；实现 `auto_apply`、`batch_review`、`manual_review` 的决策 | Policy engine | P2, P4 | 所有变更都有建议动作与执行模式 |
| P6 执行服务 | 打通自动应用与审批链路 | 实现自动应用、批量审批、逐条审批、审批动作记录与状态回写 | Apply service、review service、audit log | P5 | 低风险可自动应用；中风险可批量审批；高风险可逐条审批 |

| P8 日志冷热分层 | 控制长期存储体积 | 接入对象存储；按值大小选择 inline / ref；补充 hash 校验；评估按时间分区与归档 | Cold storage service、partition / archive plan | P7 | 大字段不长期挤占热表，且仍可审计和回滚 |

## 风险与缓解

### 风险 1：规则表达能力不足

- 影响：来源差异无法配置
- 缓解：先用结构化 matcher，必要时再扩展 matcher 类型，而不是直接引入脚本

### 风险 2：自动应用范围过大

- 影响：错误覆盖主表，且影响扩散更快
- 缓解：第一版只对白名单低风险字段启用 `auto_apply`

### 风险 3：审批积压

- 影响：新数据长期无法生效
- 缓解：提供批量审批、按来源过滤、按字段策略过滤

### 风险 4：批量审批分组过粗

- 影响：不同语义更新被错误一起通过
- 缓解：批量分组键必须包含来源、字段、策略与 matcher 原因

### 风险 5：来源配置过多导致维护成本上升

- 影响：规则不可读
- 缓解：规则集可复用，来源只做必要 override

### 风险 6：主表被绕过更新

- 影响：留痕链路断裂
- 缓解：明确主表只允许自动应用服务或审批后应用服务写入

### 风险 7：更新日志体积膨胀

- 影响：热表查询变慢，数据库存储成本上升
- 缓解：P1 预留 hash / ref / storage mode 字段，P8 再接入对象存储和归档

## 建议实施顺序

1. 先做 P0 和 P1，固定数据库和模型边界
2. 再做 P2、P3、P4，打通“导入到待审批”
3. 然后做 P5、P6，补齐执行模式、自动应用与审批链路
4. 接着做 P7，用回滚和真实来源验证设计是否足够稳定
5. 最后按数据规模推进 P8 的日志冷热分层
