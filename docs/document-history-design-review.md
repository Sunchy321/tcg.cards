# `document-history-design.md` 评审意见

## 结论清单

### 必须修改

- 拆分状态字段语义：将 `DocumentVersion` 的版本生命周期状态与导入处理状态分开建模
- 明确审核状态事实来源：在 `DocumentNodeChange.reviewStatus` 与 `DocumentChangeReview.status` 之间确定唯一事实来源
- 明确所有冗余字段的 authoritative / derived 边界，并补充统一重建机制
- 增加导入溯源字段：至少补充 source file hash、parser version、import run ID
- 明确 `reviewRevision` 的存储位置、递增规则和作用域

### 建议修改

- 收紧 `path` 模型定义，包括 path 语法、排序规则与索引方案
- 为不同文档类型预留特化匹配策略入口，不要完全依赖统一启发式
- 将 `intro`、`glossary`、`example` 等特殊节点纳入单独匹配规则
- 细化审核 override 后的修复流程，包括实体回写、缓存刷新和局部重算
- 重新评估正文 gzip 压缩与部分缓存字段是否需要在 v1 一次性实现

### 可后置

- `split` / `merged` 的高精度自动识别优化可以放到 v1 之后
- 阈值调优样本集与多文档独立参数校准可以在基础链路跑通后再做
- 全文检索、统计后台和更复杂的审核运营规则可以作为后续增强项
- 更激进的存储优化与冷数据归档可以等真实数据规模增长后再决定

## 总体评价

这份设计文档整体质量较高，已经具备比较明确的工程落地方向。文档不仅覆盖了目标、边界、数据模型、变更检测流程，还考虑了审核闭环、缓存失效、幂等导入与失败恢复等实际问题。

从成熟度来看，它已经不只是概念方案，而是一个可以进入实现阶段的设计稿。不过，在真正开始编码之前，仍有一些关键细节需要进一步收口，尤其是状态建模、一致性约束和部分算法边界。

**结论：** 这是一份可实施、方向正确的设计文档，但在若干核心细节上还需要进一步明确，才能作为稳定的实现基线。

## 优点

### 1. 领域分层清晰

文档将 `DocumentDefinition`、`DocumentVersion`、`DocumentNode`、`DocumentNodeEntity`、`DocumentNodeChange`、`DocumentChangeReview` 分得比较清楚，能够明显看出：

- `DocumentVersion` + `DocumentNode` 负责描述某一版本的结构快照
- `DocumentNodeEntity` 负责跨版本追踪同一个逻辑实体
- `DocumentNodeChange` 负责表达版本之间的变更结果
- `DocumentChangeReview` 负责人审确认与覆盖

这种分层方式的好处是职责边界清楚，后续无论是增加新文档类型、多语言能力，还是接入审核后台和全文检索，都更容易扩展。

### 2. 复杂变更建模方式合理

将 `split` 和 `merged` 这类复杂关系从 `details JSON` 中拆出来，单独使用 `DocumentNodeChangeRelation` 表建模，是一个很好的设计决策。

这样做的优势包括：

- 复杂关系更容易查询
- 更便于人工审核
- 更便于人工 override
- 后续重算和修正时不必解析一大段非结构化 JSON

如果复杂变更只存在于 JSON 字段里，系统后期维护成本会明显升高。

### 3. 工程落地意识比较强

这份文档不是停留在抽象层，而是已经考虑到了很多实现时才会遇到的问题，比如：

- 唯一键与索引
- 外键关系
- 缓存失效策略
- 导入顺序
- 幂等重试
- 失败恢复
- 审核工作流

这说明作者对“上线后会发生什么”是有预判的，这一点非常加分。

### 4. Diff 语义定义得比较好

文档明确区分了：

- `Snapshot Diff`
- `Reviewed Change View`

这个区分非常重要。它可以避免一个很常见的问题：用户界面看到的历史结果，与后台重新实时计算出来的结果长期不一致。

提前把这两种语义分开，能让后续 API 设计、缓存策略和前端展示都更稳定，不容易出现“同一个版本对比结果为什么前后不一样”的争议。

### 5. 范围控制比较克制

非目标部分写得比较现实，尤其是以下几点：

- v1 不追求完全自动识别所有 `split` / `merge`
- v1 不做跨文档自动实体映射
- v1 不做复杂语义级 diff

这说明文档没有试图一步到位解决所有问题，而是给第一版保留了合理边界，有利于降低初期实现风险。

## 缺点与风险

### 1. 部分状态字段存在语义重叠

当前文档里有几处状态建模还不够干净。

#### `DocumentVersion.status`

主表结构中定义的是：

- `active`
- `superseded`

但在幂等与失败恢复部分，又提出了导入过程状态：

- `pending`
- `processing`
- `completed`
- `failed`

这两组状态不是同一个维度，不建议复用同一个字段。

更合理的做法是：

- 保留一个字段表示版本发布或生命周期状态，例如 `active` / `superseded`
- 另加一个字段表示导入任务状态，例如 `importStatus`

#### `DocumentNodeChange.reviewStatus` 与 `DocumentChangeReview.status`

这两个字段也存在明显的语义重叠。

潜在问题是：

- 一个字段可能表示 `needs_review`
- 另一个字段可能已经是 `confirmed`

如果两边都被当作事实来源，就很容易出现状态漂移。

建议：

- 明确谁才是唯一事实来源
- 如果另一个字段保留，就只作为派生缓存使用

### 2. 冗余缓存字段较多，一致性压力较大

设计中保留了不少派生字段，例如：

- `DocumentNode.sourceContentHash`
- `DocumentNode.sourceFingerprintHash`
- `DocumentNode.sourceContentRefId`
- `DocumentNodeEntity.currentNodeRefId`
- `DocumentNodeEntity.currentNodeId`
- `DocumentNodeEntity.currentVersionId`

这些字段确实能优化查询性能，但也会显著提高一致性维护成本。尤其是在以下场景中：

- 重复导入
- 审核 override
- 历史回填
- 中途失败后的恢复

文档虽然已经意识到了这类风险，但在真实实现里，这些字段仍然会成为比较高风险的维护点。前提是必须确保所有写操作都经过统一的服务边界，不能让业务侧绕过导入服务直接更新。

### 3. 对 v1 来说有些部分可能偏重

文档中的数据量估算显示，几年后的总体数据规模仍然只有几十 MB 左右。

在这种规模下：

- 正文 gzip 压缩是否有必要，未必成立
- 某些缓存与派生字段是否值得在 v1 一次性全部实现，也值得重新评估

这不是设计错误，但可能会拖慢第一版落地速度。对于 v1，更简单的实现也许已经足够。

### 4. 路径查询方案还需要进一步明确

文档采用了 materialized path，并举了 `startsWith: '702'` 这类查询例子。

这个方向本身没有问题，但仍缺少几个关键定义：

- PostgreSQL 是否仅靠普通前缀索引即可满足需求
- 是否考虑使用 `ltree`
- path 的规范化规则是什么
- 混合数字与字符串片段时，path 排序规则如何保证稳定

如果这些问题不提前说清楚，后面很可能在查询性能或排序行为上再返工。

### 5. 指纹规范化策略可能过于激进

当前指纹生成逻辑会：

- 全部转小写
- 去掉标点
- 合并空白

这作为第一版启发式规则是合理的，但规则文本里某些标点和格式变化未必完全没有意义。

潜在风险包括：

- 原本不同的文本被规范化成同一份指纹
- 不同文档类型下，指纹碰撞和误匹配率差异较大

文档已经提出要做阈值调优，这很好，但实际上规范化规则本身也应该纳入样本验证，而不只是调阈值。

### 6. 候选集策略默认结构较稳定，但特殊节点未必适用

当前候选集约束依赖：

- 相同 `level`
- 相同 `nodeKind`
- 同章节
- 相近 path
- 同父节点

这套规则对结构规则比较稳定的正文条目会比较有效，但对以下类型可能不够稳：

- `intro`
- `glossary`
- `example`
- 没有稳定编号、由解析器生成 `nodeId` 的文档

这些节点更适合走文档类型特化策略，而不是完全复用同一套通用启发式。

### 7. 审核闭环已经提出，但操作规则还不够细

文档已经定义了哪些情况需要人工审核，例如：

- `split`
- `merged`
- 低置信度匹配
- 存在多个竞争候选

这部分方向是对的，但运营层面的规则还不够完整，例如：

- 审核队列按什么优先级处理
- override 是否允许回滚
- 审核修正后如何回写实体缓存
- 是否支持只重算某一个版本对

这些问题不是解决不了，但如果不提前设计，后面实现审核后台时会比较容易卡住。

## 实施前建议补充

### 高优先级

1. 明确拆分状态维度
   - 将“版本生命周期状态”和“导入处理状态”拆成不同字段
   - 将“审核事实来源”和“审核缓存状态”拆清楚

2. 更严格定义事实字段与派生字段
   - 每个冗余字段都要明确是 authoritative 还是 derived
   - 为所有派生字段定义重建方式

3. 增加源文件溯源字段
   - 例如 source file hash
   - parser version
   - import run ID
   - normalized content version

这会显著提升重导入、排障和回滚时的可控性。

4. 明确 `reviewRevision`
   - 存在哪里
   - 什么时候递增
   - 按 document 维度还是按版本对维度维护

### 中优先级

5. 收紧 path 模型定义
   - 明确 path 的语法
   - 明确 path 的排序规则
   - 明确数据库索引实现方式

6. 为不同文档类型保留特化匹配策略入口
   - `CR`、`MTR`、`IPG` 不一定适合完全共用一套匹配参数
   - `intro`、`glossary`、`example` 应考虑单独处理分支

7. 细化审核后的修复行为
   - override 后旧 `entityId` 如何处理
   - 哪些缓存需要重建
   - 是软失效还是直接重写

### 低优先级

8. 重新评估 v1 是否需要正文压缩
   - 按当前数据量，未必有必要一开始就做 gzip
   - 可以等真实数据规模上来后再补

9. 考虑进一步缩小 v1 自动识别范围
   - 自动处理 `added`、`removed`、`modified`、`renamed`
   - `split`、`merged` 默认进入审核优先流程

这样会更稳，也更容易先把系统跑起来。

---

## 最终结论

这是一份质量不错的设计文档。

它最突出的优点是：

- 领域拆分清晰
- 工程实现意识强
- 审核闭环完整
- 同时考虑了相邻版本 diff 与任意版本 diff

它当前最大的风险点在于：

- 状态字段存在重叠
- 派生字段较多，容易带来一致性压力
- 部分算法与 path 模型细节还不够严格

如果先把这些问题收口，再进入实现阶段，这份设计文档完全可以作为后续开发的基础版本。
