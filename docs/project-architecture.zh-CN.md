# 项目架构

## 目的

本文档定义项目的高层运行时边界和数据归属模型。

它以架构原则为主，而不是以具体实现细节为主。某些模块目前仍在迁移中，但后续新增工作应遵循本文档描述的边界。

## 运行时形态

工作区主要对面向客户端或接口的应用使用三种运行时形态：

- `site-*`
  面向终端用户的网站部署。
- `app-*`
  可安装应用，目前包括 desktop 应用。
- `service-*`
  暴露给其他运行时使用的后端接口或控制面服务。

不同运行时可以暴露不同能力，但不能因此混淆数据归属。

当 `apps/` 下的新应用属于这些主要运行时形态时，应遵循这些前缀规则。对于主要承担定时任务、而不是承担后端接口职责的特殊运行时，可以保留更直接的描述性命名。

## 应用列表

当前 `apps/` 工作区包含以下项目级应用：

- `app-console-desktop`
  desktop 管理端，也是领域数据重型工作流的主要本地运行时。这个应用拥有最完整的本地操作能力，包括访问本地仓库、本地 PostgreSQL 和 desktop 专属工作流。
- `app-console-mobile`（规划中）
  轻量移动管理终端。它的管理层级整体上接近 `site-console`，但运行在移动设备上，而不是运行在 Web 上。它后续也可以承载一部分不适合在浏览器中实现的轻量管理能力。
- `service-internal`
  面向 app 形态管理端的后端接口。它提供可安装管理应用所使用的远端后端 API。
- `site-console`
  Web 管理端。它是受限的远端管理界面，不应被视为领域数据重型工作流的执行权威端，也不能替代 desktop 独占的本地能力。
- `site-{game}`
  面向某一个游戏的公开站点。当前实例包括 `site-hearthstone` 和 `site-magic`。
- `site-main`
  项目主站和导航入口。
- `watcher`
  用于定期检查和提醒的运维运行时。它刻意保留了描述性命名，因为它的主要职责是定时监控和自动化，而不是为网站或 app 提供后端 API。当前主要用于变更检测和通知，后续也可能包含定时数据更新工作流。

## 管理运行时关系

面向管理的几个运行时并不共享同一层 authority，但它们可以复用同一套后端代码分层。

- `site-console`
  Web 形态的远端管理界面。它应保持在受限的 Web 能力层级，并以内嵌方式暴露较低能力的远端 router，因为网站部署应尽量避免为了自身管理流量再发起跨服务请求。
- `app-console-mobile`（规划中）
  移动管理客户端，整体能力预期高于 `site-console`、低于 desktop。它最终对外获得的能力，不必等于 `service-internal` 当前承载的完整 router 面。
- `service-internal`
  面向 app 形态管理端的远端后端 API 面。它在合适的地方复用同一套管理后端代码分层，对外暴露的远端 router 能力高于 `site-console`，但之所以以独立服务存在，是因为 app 形态客户端无法像 `site-console` 那样避免远端服务请求。
- `app-console-desktop`
  完整的管理运行时。它的能力不能只由远端 router 定义，因为它还拥有本地仓库、本地 PostgreSQL 和完整本地数据工作流等 desktop 本地能力。某些 desktop 操作还会通过直连远端数据库完成，而不是经过 `service-internal`。

因此，能力分层应优先从运行时角度描述，而不应只从 router 角度描述。

在远端 router 暴露层面，当前目标顺序是 `site-console` < `service-internal`。

desktop 的能力位于这个远端 router 顺序之上，因为它还包含本地独占能力，以及一部分不进入远端 router 的远端数据库直连能力。

`service-internal` 内部可以承载面向 desktop 的完整 router 面，但这并不意味着每一个 app 形态客户端都应获得其中全部能力。

## 包列表

`packages/` 工作区包含供多个应用复用的共享基础模块：

- `auth`
  共享认证集成层，包括 console 认证客户端创建、角色辅助逻辑和服务端认证初始化。
- `console-api`
  共享管理 API 面，以及提供给 console 客户端使用的相关领域后端逻辑。
- `console-core`
  共享的 console 基础原语，例如错误处理和布局元数据。
- `console-platform`
  console 客户端的平台抽象层，包括 API 访问、路由集成、会话处理、存储、toast 集成和能力接口。
- `console-shell`
  共享的 Nuxt 管理端外壳，包括可复用的管理页面、路由结构和面向 console 的 UI 组合。
- `db`
  数据库 schema 定义、schema 来源布局，以及 local、remote 两条部署轨道和它们共享 schema 来源的迁移工具。
- `eslint-config-custom`
  工作区共享的 lint 配置。
- `model`
  共享领域模型、校验 schema，以及游戏数据和管理 API 的类型契约。
- `search`
  共享搜索解析、命令建模、翻译和执行辅助逻辑。
- `shared`
  工作区级的小型共享常量和通用定义。
- `tsconfig`
  工作区共享的 TypeScript 配置预设。
- `ui`
  共享 UI 层，以及可复用的基于 Nuxt UI 的组件或展示模块。

## 数据归属维度

数据库布局包含两套彼此独立的维度。

第一套维度描述 schema 来源和部署轨道：

- `shared`
  共享 schema 来源，用于放置同时被 local 和 remote 两端轨道包含的表。它不是一个独立部署的 migration 轨道。
- `local`
  desktop 本地部署轨道，用于保存构建状态、导入状态、缓存和本地处理元数据。
- `remote`
  远端部署轨道，用于保存服务状态、面向应用的控制面数据和部署相关记录。

第二套维度描述每个游戏的业务 namespace 语义：

- `{game}`
  共享领域事实。
- `{game}_data`
  导入侧事实、构建侧事实，以及其他不应被视为可导出共享领域事实或面向用户应用状态的系统侧数据。
- `{game}_app`
  面向用户的应用状态。

在第一套维度里，只有 `local` 和 `remote` 是部署轨道。`shared` 只描述共享 schema 定义放在哪里，然后再被纳入这些部署轨道。

这两套维度是正交的，不能合并成同一套分类。

schema 轨道回答的是某个工作流应落在哪个存储与迁移轨道里。游戏 namespace 回答的是一张表表达的是哪一类事实。

这意味着 `local` 或 `remote` 轨道都可以包含 `{game}` 和 `{game}_data` 表。正常情况下，`{game}_app` 应归属远端应用工作流，而不是本地 desktop 工作流；如果存在例外，必须由具体工作流显式说明并给出理由。

新增表时，必须分别判断这两个维度，而不能假设其中一个维度会自动推出另一个维度。

## 权威模型

desktop 应用是领域数据重型工作流的执行权威端。

这意味着：

- desktop 拥有最完整的本地运行时上下文
- desktop 可以访问本地仓库和本地 PostgreSQL
- desktop 可以执行批量导入、重量导入，以及其他领域数据重型工作流
- remote 默认不是领域数据重型工作流的执行权威端

remote 运行时是受限副本、发布目标或受限管理界面。

某个 remote 工作流可以拥有一部分字段或记录，但这种 ownership 必须显式定义。不能因为 remote 是集中部署的，就默认它天然更权威。

能力意图应分三层理解：

- 领域数据重型工作流
  对于依赖本地仓库、本地 PostgreSQL 构建状态、批量导入、重量导入或重量投影的工作流，desktop 是唯一预期的执行面。
- 领域数据轻量编辑
  Web、mobile 和 desktop 三类管理运行时都可以提供轻量编辑能力，但仍需服从各自预期的远端接口层级和运行时工作流约束。
- 非领域数据管理工作流
  对于不定义也不变换领域数据的管理功能，例如用户管理一类的工作流，默认不由 desktop 独占，可以在多个管理运行时之间共享。

## 当前已定义的 Remote-Owned 范围

当前项目级别明确归属 remote 的范围，刻意收敛在那些主要运行时上下文位于远端基础设施中的类别：

- 供远端应用和管理端使用的认证、账户、会话和 API 凭证状态
- 通过远端应用工作流产生的 `{game}_app` 状态，包括用户创建内容、用户交互状态和远端评审决策
- 依赖托管基础设施、而不是依赖 desktop 本地构建执行的远端运维型 `{game}_data`，例如 knowledge 索引状态、embedding 状态、托管资源元数据，以及其他同类远端处理记录
- 仅为了运行 remote 运行时而存在的部署侧控制面记录

这些 remote-owned 类别可以引用共享领域数据或 desktop 产出的数据，但不能静默覆盖由 desktop 拥有的构建真相、导入真相或本地处理真相。

## 游戏数据架构

游戏数据工作流默认应收敛到 desktop 本地执行。

目标稳定形态是：

1. desktop 读取并准备本地游戏源数据
2. desktop 将构建状态和导入状态写入本地 PostgreSQL
3. desktop 执行本地处理
4. desktop 在需要时将选定事实同步到 remote

这意味着本地数据库是游戏数据工作流的构建权威层。

除非某个具体工作流被明确设计为例外，否则 remote 不应继续承担本地构建真相、本地导入真相或本地处理真相的主执行面。

## 本地与远端同步策略

local 和 remote 是两个独立数据库，因此多端修改的核心矛盾是同步与合并，而不是单写入端问题。

每个管理运行时都可以在其允许的能力范围内执行修改。

在实践中，desktop 一侧通常会产生更多领域数据变更，因为批量导入和重量导入在那里执行。

当同一数据集可能被两侧修改时，工作流必须显式定义合并策略，例如基于时间戳的优先级、字段级归属，或其他可确定的规则。

默认设计目标是不把通用的整行 last-write-wins 视为隐式的全局规则。

如果某个数据集同时混合人工维护字段与构建生成字段或导入生成字段，就应先拆分责任，而不是让整行共享一套不加区分的同步规则。

## 同步方向原则

同步应该服从 authority，而不是服从实现便利。

对于由 desktop 主导的领域数据工作流，默认规则是：

- `desktop -> remote` 是主方向

只有同时满足以下条件时，才允许 `remote -> desktop`：

- remote 拥有的字段子集已经被显式定义
- 回流字段不会覆盖 desktop 拥有的真相
- 这种同步行为对用户和开发者都是可解释的

系统应避免通用的整行双向合并。

这一默认方向不覆盖那些已经被显式定义为 remote-owned 的类别。对于被定义为 remote-owned 的工作流，应按该归属定义同步行为，而不是强行套回 desktop 主导模式。

## 当前架构规则

当某个游戏工作流依赖本地仓库、本地文件或本地 PostgreSQL 构建状态时，除非有明确反例，否则应把 desktop 视为执行权威端。

当一张表同时混合人工维护字段与构建生成字段或导入生成字段时，应拆分责任，而不是依赖模糊的合并归属。

## 文档规则

`docs/` 下的架构文档用于描述稳定边界和 authority 规则。
