## 1. 目标

基于 `docs/document-frontend-display-design.md` 的细化方案，把当前文档模型逐步落到前端，使 `magic-cr` 从旧 `RuleItem` 体系迁移到通用文档体系，并为后续多文档复用建立骨架。

## 2. 实施原则

- 先冻结 View Model，再写页面
- 先做公共站点阅读链路，再做历史与对比
- 先做可用审核后台，再考虑复杂编辑体验
- 公共站点默认只消费 `reviewed`
- 控制台优先暴露诊断信息与审核能力

## 3. 范围

### 3.1 本期纳入

- 公共站点阅读页迁移
- 公共站点结构化对比页
- 公共站点 `entityId` 历史页
- 控制台审核队列与详情
- 审核提交与缓存失效

### 3.2 本期不纳入

- 多语言翻译编辑后台
- 图形化关系拖拽编辑
- 全文搜索
- 可视化统计报表

## 4. 分阶段计划

## P0 View Model 冻结

### 目标

冻结公共前端消费的数据契约，避免页面开发期间反复调整。

### 核心任务

- 定义 `DocumentReaderPage`
- 定义 `DocumentComparePage`
- 定义 `DocumentEntityHistoryPage`
- 定义 `DocumentReviewQueuePage`
- 定义 `DocumentReviewDetail`

### 产出物

- model / schema 草案
- BFF 输出结构
- 字段说明文档

### 验收标准

- 前端页面不再直接依赖底层表结构
- `nodeId` / `entityId` / `changeId` 使用边界明确

## P1 公共阅读链路

### 目标

让 `magic-cr` 阅读页完全由通用文档模型驱动，并让用户直接进入最新版阅读。

### 核心任务

- 实现 `getDocumentReader`
- 明确 `/document/[slug]` 的最新版解析规则
- 聚合 `.title` 节点
- 生成 outline 树
- 生成 section 列表
- 改造现有 `/rule` 页面或新增 `/document/[slug]`

### 产出物

- BFF 接口
- 阅读页组件
- 目录树组件
- section 组件

### 验收标准

- `magic-cr` 可完整阅读
- 访问 `/document/[slug]` 时默认打开最新版
- 版本切换、目录定位、锚点定位可用
- 节点历史入口已挂入页面

## P2 公共对比链路

### 目标

将现有 diff 页升级为结构化变更页。

### 核心任务

- 实现 `getDocumentCompare`
- 对 `added` / `removed` / `modified` / `moved` / `renamed` / `split` / `merged` 做统一输出
- 实现章节与类型筛选
- 把 `reviewState` 和 `reviewRevision` 纳入页面

### 产出物

- 对比 BFF
- 变更卡片组件
- 过滤器组件

### 验收标准

- diff 页不再只依赖文本双栏
- `split` / `merged` 可被结构化表达
- 页面可区分稳定结果与待审核结果

## P3 实体历史链路

### 目标

把历史页从 `itemId` 视角迁移到 `entityId` 视角。

### 核心任务

- 实现 `getDocumentEntityHistory`
- 生成 timeline 数据
- 支持跳回对应版本阅读页

### 产出物

- 历史 BFF
- 时间线组件
- 历史页

### 验收标准

- rename / move 后历史仍连续
- 每个 timeline entry 都可回跳阅读页

## P4 控制台审核链路

### 目标

为复杂变更和低置信度变更建立可用审核后台。

### 核心任务

- 实现 `listDocumentReviewQueue`
- 实现 `getDocumentReviewDetail`
- 实现 `submitDocumentReview`
- 展示关系列表、审核记录、override 结果

### 产出物

- 审核列表页
- 审核详情面板
- 审核动作栏

### 验收标准

- 审核人员可完成确认、驳回、override
- `reviewRevision` 在提交后递增

## P5 缓存与旧模型收口

### 目标

保证新旧链路切换后结果一致，缓存可失效。

### 核心任务

- 对比页缓存键纳入 `mode + reviewRevision`
- 审核提交后触发相关缓存失效
- 收口旧 `RuleItem` 读取逻辑
- 评估 `/rule` 是否作为兼容壳层保留

### 产出物

- 缓存策略
- 路由迁移策略
- 兼容层方案

### 验收标准

- 审核后公共页与控制台结果更新正确
- 旧入口可平滑过渡

## 6. 风险与应对

### 风险 1：BFF 不稳定导致页面反复返工

应对：

- P0 先冻结字段和语义

### 风险 2：旧 `rule` 页面与新文档页长期双轨

应对：

- 从第一阶段就规划兼容壳层和淘汰路径

### 风险 3：最新版解析不稳定

应对：

- 明确最新版判定规则与 query 覆盖规则
- 在 BFF 层统一处理，不把判定逻辑散落到页面

### 风险 4：审核后缓存不一致

应对：

- 明确 `reviewRevision` 进入 compare 缓存键

### 风险 5：公共站点误展示未审核结果

应对：

- 默认模式固定为 `reviewed`
- 仅控制台允许切换 `snapshot`

### 风险 6：复杂关系交互过重拖慢进度

应对：

- 第一版用列表式关系编辑
- 图形化交互后置
