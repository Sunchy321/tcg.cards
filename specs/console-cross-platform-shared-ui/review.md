# console 三端共享与桌面壳选型评审

## 评审结论

建议按当前方案推进，并明确采用“先抽共享层、后复评桌面壳”的路线。

## 通过理由

1. 方案准确识别了当前问题的根因，重点不在于 Nuxt 与 Vite 的框架差异，而在于共享边界缺失
2. 方案把 `site-console`、`app-console-desktop` 与未来 `app-console-mobile` 放在同一目标结构下评估，避免只优化两端复用
3. 方案补充了后端双落点约束，明确 `site-console` 保留 SSR / BFF 薄后端，独立 app 共用后端能力收口到 `service-internal`
4. 方案避免过早把桌面端绑到 Nuxt 运行时，有利于保持桌面与移动平台能力边界清晰
5. 方案允许按“核心逻辑、SSR / BFF 与后端拆分、共享 UI、平台适配、页面迁移、壳层复评”分阶段推进，实施风险可控
6. 方案已经结合当前仓库实际问题，例如页面复制、`#model` alias 泄漏和 Nuxt 运行时依赖外溢，判断依据充分

## 主要风险

### 1. 共享层抽象过度

如果一开始就试图抽象完整页面框架和全部场景，容易形成新的中间层复杂度，反而降低开发效率。

建议坚持从高重复页面块和高重复状态逻辑开始，而不是一次性追求所有页面统一。

### 2. `site-console` 的 server 侧能力继续向共享层泄漏

当前 `site-console` 同时承担页面层和服务端能力，迁移过程中若没有严格约束，`useRequestEvent`、`#db`、`#schema`、`#model` 等依赖仍可能被带入共享客户端层。

这会直接破坏三端复用目标。

### 3. 平台适配接口设计不清

若 `toast`、`router`、`session`、`storage`、`api` 等适配接口设计过于粗糙，后续页面仍会被迫依赖平台实现细节。

这会导致共享组件名义上抽出，实际上仍然只能在单一壳层中运行。

### 4. 迁移周期内会有双轨维护成本

在共享层逐步建立之前，`site-console` 与 `app-console-desktop` 仍需要短期并行维护既有页面与迁移后页面块。

这是可接受成本，但需要提前承认，而不是假设迁移会一次完成。

### 5. 若 BFF 边界定义不清，会把 `service-internal` 重新做成页面后端

如果没有明确“哪些能力必须留在 `site-console` 用于 SSR / BFF，哪些能力应该迁到 `service-internal`”，迁移时很容易出现：

- `site-console` 继续保留过多业务后端
- `service-internal` 反而开始承担页面专属聚合

这会让两边职责再次重叠，后续仍然难以收敛。

## 结论建议

1. 先接受本提案的总体方向
2. 后续实施时优先处理共享核心，并同步拆清 `site-console` 的 SSR / BFF 角色与 `service-internal` 的独立 app 后端角色
3. 在共享层完成一轮真实页面迁移后，再单独评估“桌面端是否改为 Nuxt SPA + Tauri”

当前阶段不建议把“迁 Nuxt”作为主任务启动点。

## 实施后复评补充

在当前实施结果下，已经完成：

1. 共享核心层、平台适配层与共享 UI 层拆分
2. `YamlEditor` 共享化
3. `magic/announcement`、`hearthstone/announcement`、`hearthstone/set`、`hearthstone/tag` 回收为共享页面

基于这轮真实迁移结果，评审补充结论如下：

1. 原评审判断成立，桌面端当前不需要迁移到 Nuxt
2. 现阶段的实际收益来自共享层，而不是来自桌面壳框架统一
3. 当前若继续推进 Nuxt 迁移，会增加壳层复杂度，但不会明显减少已剩不多的桌面端专属样板
4. 后续应继续沿共享包边界推进，而不是转向壳层替换
