## 目标

将桌面端本地 PostgreSQL 落地为稳定可运维的权威构建数据库，承接导入流水线型 `_data` 与来源缓存；同时保留远端正式表和必要远端 serving 数据，建立清晰的发布、导出与协同机制。

## 实施原则

- 本地 PostgreSQL 负责构建，远端 PostgreSQL 负责 serving
- 先固定边界，再迁移数据与链路
- 先迁导入流水线，再迁来源缓存，再落地发布链路
- dump/export 作为正式能力设计，不依赖临时脚本
- 第一版优先显式拉取与显式发布，不做双向实时同步

## 实施步骤

### 1. 固定职责边界

- 明确哪些 `_data` 属于本地权威构建层
- 明确哪些 `_data` 保留远端 serving 角色
- 明确哪些远端表采用直连、镜像或发布写入
- 固定 `knowledge_*` 保留远端

### 2. 设计本地 PostgreSQL 生命周期

- 设计数据库初始化流程
- 设计启动、停止与迁移执行时机
- 设计本地数据目录位置
- 设计本地数据库损坏时的恢复手段
- 明确凭证保存、轮换与撤销方式

本阶段还应固定以下运行时约束：

- 本地 PostgreSQL 生命周期跟随桌面 runtime，而不是单个窗口
- app 关闭主窗口但仍在后台执行任务时，本地 PostgreSQL 不应停止
- 只有 app 真正退出时，才优雅关闭本地 PostgreSQL
- 不将本地 PostgreSQL 设计为系统长期常驻 daemon

若设置界面支持远端发布目标连接配置，也应在本阶段明确：

- 默认本地构建数据库仍是 app 托管的本地 PostgreSQL
- 设置界面配置的是远端发布目标，而不是本地运行库类型
- 发布目标连接只作为发布链路和必要远端校验的输入
- 不允许让同一个远端目标连接承担完整本地构建数据库职责
- 发布批次创建时必须绑定 `publishTargetId`、环境和 fingerprint

### 3. 迁移导入流水线型 `_data`

- 先迁 `hsdata_import_*`
- 再迁 `source_versions`、`raw_entity_snapshots`、`raw_entity_snapshot_tags`
- 再迁 `magic_data.import_*`
- 最后迁 `gatherer`、`mtgch`、`scryfall`

每一阶段都要求：

- 本地可独立写入
- 本地可断点续传
- 本地可失败恢复
- 远端旧路径仍可回退

对 `magic` 需要额外固定一条边界：

- 本地投影结果先作为待审核候选
- 不默认直接进入远端正式表
- 发布链路只能消费审核通过的结果

本阶段还应同步固定术语：

- 本地使用 `import / build / materialize`
- 审核层使用 `review / approve`
- 远端发布层使用 `finalize / apply / refresh`

### 4. 设计导出与恢复能力

- 工作区快照导出：
  - 用于完整共享本地构建环境
  - 以 `pg_dump` 为主
- 逻辑导出：
  - 用于分享最终结果或指定批次结果
  - 以 `JSON / NDJSON / CSV` 为主
- 恢复入口：
  - 从 dump 恢复
  - 从逻辑导出恢复

### 5. 设计远端残留表协同

- 远端直连型：
  - 体量小
  - 调用频率低
  - 不做本地持久镜像
- 本地镜像拉取型：
  - 本地构建频繁读取
  - 远端仍为 authority
  - 支持会话开始拉取与发布前强制刷新
- 远端发布写入型：
  - 正式领域结果
  - 远端 serving copy
  - 由发布链路统一写入

### 6. 处理 R2 状态表与远端简单编辑表

- 为 R2 状态表定义：
  - 远端 authority
  - 本地按需拉取
  - 发布后回写
- 为远端简单编辑表定义：
  - 发布前强制拉取
  - 版本或 checksum 校验
  - 冲突时阻止覆盖

### 7. 定义远端轻量同步任务表

- 定义发布任务主表
- 定义任务分片表
- 定义 lease / heartbeat 字段
- 定义重试次数、失败原因和结果 checkpoint
- 定义任务完成、取消和恢复语义

这组表只承担发布控制面职责，不保存完整构建中间态。

### 8. 建立发布链路

- 定义本地构建结果的发布批次
- 定义本地发布 manifest
- 定义批次与 `publishTargetId` / fingerprint 的绑定关系
- 定义远端写入顺序
- 定义分片维度
- 定义 lease、heartbeat 与重试
- 定义回滚策略
- 定义发布成功确认

对 `hsdata`，发布链路应进一步固定为：

- 先生成 `cardId` 级别 manifest
- 以“本次 manifest vs 上次成功 manifest”做 diff
- 将 `added / changed / removed` 的卡作为发布对象
- 对变更卡按整卡粒度写入 `entity`、`entity_localizations`、`entity_relations`
- 默认将远端正式结果表视为 `publish-owned`

对 `magic`，当前阶段只补充约束，不展开细节方案：

- 发布链路前必须存在审核通过门
- 发布对象不是“全部投影结果”，而是“审核通过后的候选结果”
- 审核层与正式发布层的边界需要后续单独设计

对其他允许远端修改的游戏，还应额外固定：

- 发布前执行远端 live drift check
- 发现 drift 时阻断发布或进入显式冲突处理
- 不得只依赖“上次成功发布基线”作为唯一 truth

发布链路应固定为：

- `finalize` 负责校验并创建任务
- `apply` 负责按分片写入正式表
- `refresh` 负责刷新派生结果

不允许把整次发布设计成一个超长同步 RPC。

### 9. 更新控制台与运维文档

- 调整控制台中的 `_data` 语义描述
- 更新 README
- 更新故障处理手册
- 更新数据共享与恢复说明

## 回退策略

### 回退原则

- 每次只迁一类表或一条流水线
- 任一阶段失败时，优先回退写入入口，不立即删除远端旧表
- 未完成发布链路前，远端正式读路径保持不变

### 分阶段回退

- P2 失败：恢复远端 `hsdata_import_*` 任务状态为主入口
- P3 失败：恢复远端 `source_versions/raw_entity_snapshots` 为主快照层
- P4 失败：恢复远端 `magic_data.import_*` 为主导入层
- P5 失败：恢复远端来源缓存
- P8 失败：停止远端任务调度，保留本地构建结果，等待修复控制面
- P9 失败：停止发布，保留本地构建结果，不污染远端正式表

## 风险与缓解

### 风险 1：桌面端托管 PG 复杂度高

- 影响：安装、升级与故障恢复成本上升
- 缓解：先固定最小托管模型，再逐步自动化

### 风险 2：导出功能只有文件输出，没有恢复语义

- 影响：使用者不知道应共享完整工作区还是逻辑结果
- 缓解：明确区分工作区快照导出与逻辑导出，并同步设计恢复入口

### 风险 3：远端残留表冲突处理不清

- 影响：本地发布覆盖远端人工变更
- 缓解：按游戏区分 `publish-owned` 与 live drift check 策略；构建前拉取、发布前校验、冲突时阻断发布

### 风险 4：发布目标切换导致误发环境

- 影响：同一批次被错误发布到其他环境或其他远端库
- 缓解：批次绑定 `publishTargetId`、环境与 fingerprint，执行前再次校验

### 风险 5：混合模型边界过宽

- 影响：本地和远端职责重新耦合
- 缓解：只为必要表保留混合协同，不扩大远端残留表范围

### 风险 6：远端任务控制面设计过重

- 影响：轻量任务表重新演变为完整远端构建中心
- 缓解：任务表只保存 job、partition、lease、retry 和 checkpoint，不承载大体量导入中间态

## 验收标准

- 桌面端本地 PostgreSQL 可稳定初始化、迁移和恢复
- 构建型 `_data` 与来源缓存可在本地稳定运行
- 工作区快照导出和逻辑导出都可用
- R2 状态表与远端简单编辑表有明确协同策略
- 远端轻量同步任务表可稳定承接任务认领、恢复和进度跟踪
- 本地构建结果可通过受控发布链路写入远端正式表
- 控制台与运维语义已与本地 PG 架构一致

## 建议的后续动作

3. 决定设置界面如何暴露远端发布目标连接，以及其权限、校验和凭证存储边界
4. 确认 Sealos 上远端数据库角色与受限凭证的实际可行性
