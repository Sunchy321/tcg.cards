# 本地 PostgreSQL 权威构建层提案评审

## 评审结论

接受该设计方向。当前版本已经把最关键的结构性问题收口到可以进入后续细化设计的程度，尤其是：

- 明确了“本地权威构建层 + 远端 serving 层”的主边界
- 明确了 `hsdata` 与 `magic` 不共享同一种发布前提
- 明确了发布目标连接是 publish target configuration，而不是运行库模式切换
- 明确了发布术语需要拆分为本地构建、审核和远端发布三组动作

当前 proposal 适合继续推进，但后续实现必须严格遵守这些新边界，不能在落地阶段重新回退成“远端兼做构建中心”。

## 评审要点

### 1. 按游戏区分发布一致性策略是必要的

当前设计不再试图用一条统一规则覆盖全部游戏，这是正确的。

原因很直接：

- `hearthstone` 可以把远端正式结果表尽量收敛为 `publish-owned`
- 其他游戏可能存在远端人工修改、远端简单编辑表或带外写入动作

如果继续追求一套统一的“只比上次成功发布基线”的发布算法，最后只会把 `hearthstone` 的理想条件错误推广到并不满足这些条件的游戏上。

现在 proposal 里把这件事写成“按游戏分策略”后，发布链路的正确性前提就更清楚了：

- `hearthstone` 走基线 diff 主路径
- 允许远端修改的游戏必须补 live drift check 或等价冲突控制

这条边界应该继续保留，不要在实现时为了追求通用化而重新抹平。

### 2. 术语拆分方向是对的，而且应该尽早落实

把链路拆成：

- `import / build / materialize`
- `review / approve`
- `finalize / apply / refresh`

是一个正确决定。

此前最大的风险不是算法本身，而是同一个 `projection` 同时表示：

- 本地从 raw/import 层生成结果
- 远端发布后的派生刷新

这种命名会直接污染：

- 状态字段
- 任务名
- UI 进度语义
- 日志和告警文本

现在设计里已经把三段动作拆开，后续实现应该尽量跟随这套术语，不要在新任务表、状态字段和控制台页面里继续扩散旧词。

### 3. 远端发布目标连接可以动态配置，但必须是“目标身份绑定”，不是“临时读取当前设置”

当前设计把发布目标连接提升为 profile，并要求批次绑定：

- `publishTargetId`
- `environment`
- fingerprint

这点非常关键，而且方向正确。

如果发布批次不绑定目标身份，动态配置本身就会变成误发风险来源。相反，只要把目标身份写进批次元数据，再在执行前重验一次，就能把“切环境后误发”的问题显式化、可阻断化。

这里最重要的不是 profile 的 UI 形式，而是要坚持：

- 批次在创建时就绑定目标
- 执行时必须复验目标一致性
- 不允许用“当前设置是什么就发到哪里”替代批次绑定

### 4. `magic` 的审核前提已经被正确压成边界，而不是被提前展开成半成品方案

当前 proposal 对 `magic` 的处理是：

- 先明确“投影结果不是默认可发布结果”
- 先明确“审核通过后才能进入发布链路”
- 暂不提前设计审核模型细节

这是合理的。

如果现在强行把审核表结构、审核粒度和审核后发布协议全部写死，反而容易在缺少足够上下文时做出过度承诺。当前写法把关键边界保住了，同时没有把后续审核设计提前锁死，这是比较稳的。

### 5. 本地 PostgreSQL 生命周期收口为 app runtime sidecar 是正确的

这份 proposal 没有把本地 PostgreSQL 设计成：

- 系统长期后台 daemon
- 跟单个窗口生命周期绑定

而是明确为 app runtime 托管的 sidecar。这是当前桌面端场景下最合理的形态。

这样既能支持长任务，也不会让数据库从用户认知上脱离应用本身。

## 需要继续注意的问题

- 当前设计已经写入 `publishTargetId` 和 fingerprint 绑定规则，但后续实现时还要决定 fingerprint 的实际组成，避免它过弱或过于脆弱。
- `magic` 审核后发布对象的粒度仍然是后续必须单独回答的问题，但当前 proposal 没有在这里过度展开，这一点是可接受的。
- 若未来远端环境不止一个，远端轻量同步任务表也需要与发布目标身份形成清晰隔离。

## 评审建议

建议按以下顺序进入下一轮细化设计：

1. 先把发布目标 profile 与批次绑定模型写实。
2. 再把 `hearthstone` 与其他游戏的 drift / conflict 策略拆成明确规则。
3. 最后在实现文档里系统替换旧的 `projection` 术语。

总体上，这份 proposal 现在已经达到了可继续推进的质量，不需要回退重写方向。
