# `tauri-dual-app` 评审意见

## 评审结论

建议通过当前设计方向，并进入后续规格化与实施准备阶段。

这份设计已经从“应该做网站还是 app”这种产品形态讨论，收敛到更关键的工程问题：

- 哪些能力必须中心化
- 哪些能力必须本地执行
- 哪些能力需要按网页端、手机端、桌面端分级开放

从这个角度看，当前方案已经具备较好的可实施性，整体判断是：

- 产品边界清晰
- 服务边界基本合理
- 权限方向正确
- 任务模型可支撑后续演进
- 查询侧缓存设计已达到可以继续往实现层细化的程度

我认为这份设计已经可以作为后续实现拆分的基础，不必再回到“把所有能力继续堆进单个网站”那条路。

## 总体评价

这份设计最有价值的地方，不是选用了 `Tauri`，而是完成了三件更重要的事情：

- 把查询产品线与管理产品线彻底分开
- 把管理能力按 `light` / `medium` / `full` 固定为执行能力分级
- 把“中心服务负责状态与协调，桌面端负责本地重任务”明确为后端基本模型

这三点一旦成立，后续很多实现细节都会自然收敛：

- 为什么网页端不能直接做 Git 与本地工具
- 为什么手机端不应该承接 `full` 任务
- 为什么数据库访问必须尽量集中在服务端
- 为什么任务状态不能只靠一个任务表硬撑

相比继续扩展当前公开网站，这个方向更适合长期演进；相比一开始就拆很多微服务，这个方向也更稳。

## 评审通过的关键理由

### 1. 产品边界已经足够清楚

当前设计已经把查询侧和管理侧拆成两条产品线：

- 查询侧：`site-main`、`site-<game>`、`app-main`
- 管理侧：`site-admin`、`app-admin-mobile`、`app-admin-desktop`

这比“一个 console 同时承载用户查询和管理任务”清晰得多。

尤其查询侧的定义是成立的：

- `site-main` 负责入口、聚合、全游戏搜索
- `site-<game>` 负责单游戏深内容与详情逻辑
- `app-main` 负责统一查询壳层与用户态能力

这个边界本身就能显著降低后续架构混乱的概率。

### 2. 服务拆分比较克制，适合当前阶段

当前方案采用：

- `service-api`：只读查询服务
- `service-admin`：管理服务 + 桌面任务网关

这个拆分是合理的。

它既避免了把查询接口与管理写接口混在一起，也避免了在时机未成熟时把系统过早拆成很多小服务。

对当前阶段来说，这种“逻辑清晰、部署克制”的做法比微服务化更合适。

### 3. 任务模型是这份设计里最重要的正确决定之一

`tasks`、`task_runs`、`device_leases` 这三层语义是必要的。

原因很简单：

- 管理任务不是纯同步请求
- 桌面端任务不是一次 HTTP 调用就能结束
- 同一个任务可能有多次尝试、失败重试、设备切换、租约超时

如果没有这三层模型，后续桌面执行链、日志链、审计链和重试链都会变得很脆弱。

因此我认为：

- `tasks` 承担业务态
- `task_runs` 承担执行尝试态
- `device_leases` 承担当前执行权

这个抽象是对的，不建议再回退。

### 4. 权限设计方向正确

这份设计没有把权限问题简单理解为“谁能访问页面”，而是拆成了：

- 身份认证
- 授权判断
- 资源约束

同时又区分了不同主体：

- `anonymous`
- `user`
- `device`
- `api_key`
- `service`

这是对的，因为后续真正复杂的部分并不是用户登录，而是：

- 桌面设备如何代表用户执行本地任务
- 哪些接口允许服务间调用
- 哪些接口不能被公开 key 访问
- 同一个角色在不同游戏、不同能力级别下能做什么

当前设计已经具备承载这些问题的结构。

### 5. 查询侧缓存设计已经从“优化想法”提升到“正式能力”

这一点我认为是本轮设计里另一个明显进步。

当前缓存不再只是“本地存一点数据”，而是被设计成完整能力，包括：

- 缓存对象分类
- 缓存元数据
- 主动同步触发
- 软失效 / 硬失效
- 容量预算与淘汰
- 联网模式切换
- 离线回退路径
- 服务端 manifest 与 delta 支撑

这说明设计已经意识到：

- `app-main` 的离线体验不是 UI 小功能
- 而是查询产品线的一部分基础设施

这个判断是正确的。

## 仍需注意的实现约束

虽然我认为方向已经可以通过，但下面这些约束必须在实现中被严格执行，否则设计很容易再次漂移。

### 1. 不要让 `app-main` 重新长出完整单游戏详情系统

当前设计把：

- 跨游戏壳层
- 单游戏深内容

明确分到了不同位置。

后续实现时，无论是否复用现有网页页面，都不应改变这条原则：

- `app-main` 可以承载跨游戏能力
- 但不应无节制地重新承接所有单游戏展示逻辑

否则查询侧会再次变成多套事实来源并存。

### 2. 不要让 `service-api` 与 `service-admin` 再次互相渗透

如果实现时图省事，把管理写接口逐步加回 `service-api`，或者把查询读模型随意挂进 `service-admin`，前面建立的服务边界会很快失效。

这一点需要在代码组织、路由归属和鉴权中同时保持一致。

### 3. 不要把能力分级误写成权限分级

`light` / `medium` / `full` 的语义应该一直保持为：

- 任务执行能力分级

而不是：

- 用户角色分级
- 页面可见性分级

页面能否看见、任务能否查看、数据能否访问，仍然需要单独权限模型决定。

### 4. 缓存系统不能只做“写入”，必须同时做“治理”

当前缓存设计已经写得比较完整，但实现时最容易缩水。

尤其要避免只做下面这些“看起来有缓存、实际上不可控”的方案：

- 只缓存最近请求结果
- 不区分缓存对象类型
- 不做失效
- 不做空间治理
- 不区分匿名与登录用户作用域

如果发生这种缩水，后续离线体验、数据陈旧问题和多账号污染都会集中爆发。

### 5. 桌面协议必须版本化

桌面端要连接：

- 本地文件系统
- Git 仓库
- 第三方卡图工具
- 对象存储上传链路

这意味着桌面端不是一个“稳定不变的小壳”，而是会持续演进的执行器。

因此协议版本、能力协商、心跳和租约机制都不应被弱化。

## 我认同的风险排序

### 高风险

- 任务模型被实现层简化回单表
- 桌面任务协议没有版本和能力协商
- 查询缓存最后退化成无边界的本地 JSON 存储

### 中风险

- `service-admin` 内部未来是否需要进一步拆出独立长连接/任务网关
- 查询侧共享层是否会逐步膨胀
- 内嵌页、快照页、原生壳之间的边界在实现中被反复突破

### 低风险

- 继续使用 `Tauri` 作为桌面与移动宿主
- 继续沿用中心数据库 + 对象存储
- 管理与查询继续维持双产品线

## 建议的实施关注点

如果进入下一阶段，我建议实现顺序仍然优先围绕以下三条主线展开：

- 先固化 `service-api` / `service-admin` / 桌面协议边界
- 先落稳 `tasks`、`task_runs`、`device_leases`
- 先把查询缓存协议做成正式接口，再谈 UI 细化

换句话说，后续最先要做的不是“把 app 壳搭出来”，而是先把：

- 服务职责
- 执行协议
- 缓存协议

这三层地基打稳。

## 最终意见

这份设计已经达到了“可以指导后续拆分与实现”的水平。

我给出的最终判断是：

- 方案方向正确
- 架构层次基本成立
- 关键抽象已经足够明确
- 可以进入后续规格化与实现准备

如果要用一句话概括这份设计的价值，那就是：

**它已经把系统从“一个不断膨胀的网站”重新定义为“查询产品线 + 管理产品线 + 中心协调服务 + 桌面执行器”的组合。**
