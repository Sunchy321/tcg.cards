## 1. 目标

围绕 `Tauri` 技术栈建立双应用体系：

- **查询应用**：只读为主，带少量用户态写操作
- **管理应用**：支持网页端、手机端、桌面端三种形态，并按能力分级执行不同任务

实施过程中重点解决：

- 管理能力与查询能力的分离
- 本地重任务的桌面端承接
- 三端统一的 API、任务、鉴权与审计协议
- 中心服务、数据库、对象存储之间的连接方案

## 2. 实施原则

- 先拆边界，再做宿主
- 先让网页端跑通共享模型，再引入 Tauri
- 先支持桌面端全能力，再扩展手机端中等能力
- 所有正式客户端默认不直连数据库

## 3. 阶段计划

| 阶段 | 目标 | 核心任务 | 输出物 | 依赖 | 验收标准 |
|------|------|----------|--------|------|----------|
| P0 边界确认 | 固化双应用和三端能力模型 | 明确查询应用边界；明确管理应用边界；定义 `light` / `medium` / `full`；梳理现有功能归属 | 产品边界清单、能力矩阵 | 无 | 所有现有核心功能都能归到某个应用与某个能力级别 |
| P1 服务分层 | 建立统一后端边界 | 设计后端模块化单体模型；明确 `service-api` 与 `service-admin` 边界；设计查询 API、管理 API、桌面任务网关；设计统一权限模型；明确对象存储上传流程；明确审计与任务模型 | API 边界文档、任务模型草案 | P0 | `service-api` 与 `service-admin` 职责清晰，读、写、桌面执行三类协议入口边界明确，权限模型统一且内部模块职责明确 |
| P2 管理网页端收口 | 让现有管理网页端只承载轻量能力 | 识别 `site-console` 页面；下线或隐藏桌面专属入口；接入任务分级；统一 API client，并作为 `site-admin` 演进目标 | 管理网页端 v1 | P1 | 网页端可安全执行全部 `light` 任务，且不暴露 `full` 本地能力 |
| P3 管理共享模块 | 抽离三端共用的领域层 | 提取页面状态、表单模型、任务模型、API client、权限判断、能力开关 | `packages/app-admin` 等共享包 | P2 | 网页端能完全基于共享模块运行 |
| P4 Tauri 桌面端 | 交付全能力管理终端 | 创建 `app-admin-desktop`；实现文件系统、Git、第三方工具桥接；接入设备鉴权与桌面任务执行 | 管理桌面端 v1 | P3 | 可执行 `full` 任务，包括本地仓库读取、卡图生成、产物上传 |
| P5 Tauri 手机端 | 交付中等能力移动终端 | 创建 `app-admin-mobile`；接入登录、通知、任务列表、审核、轻中量操作；屏蔽 `full` 能力 | 管理手机端 v1 | P3 | 手机端可完成状态查看、审核与 `medium` 任务，不暴露桌面专属能力 |
| P6 查询应用独立化 | 将用户查询体验从管理体系中拆出 | 让现有 `site-magic`、`site-hearthstone`、未来 `site-<game>` 与 `app-main` 共享查询架构；固定 `app-main` 通过内嵌页面消费单游戏详情能力；设计主动缓存与离线快照；对接只读 API；保留收藏、偏好等用户态操作 | 查询应用 v1 | P1 | 各游戏查询站点与查询 app 都不依赖管理端上下文，读操作链路清晰，`app-main` 不复制单游戏详情逻辑，且常用内容可离线读取 |
| P7 存储与上传优化 | 完善大文件与产物链路 | 下发短期上传凭证；支持桌面端直传对象存储；完善失败恢复与日志回传 | 上传链路 v1 | P4 | 大文件和大量产物不需通过中心 API 中转即可完成上传 |

## 4. 详细任务拆解

### P0 边界确认

| 序号 | 任务 | 说明 |
|------|------|------|
| 0.1 | 梳理查询应用职责 | 搜索、列表、详情、规则、收藏、偏好等归入查询应用 |
| 0.2 | 梳理管理应用职责 | 导入、审核、上传、重试、卡图生成、数据源管理等归入管理应用 |
| 0.3 | 定义能力标签 | 为每类任务标注 `light` / `medium` / `full` |
| 0.4 | 输出归类清单 | 为现有页面、脚本、接口建立归属表 |

### P1 服务分层

| 序号 | 任务 | 说明 |
|------|------|------|
| 1.1 | 定义查询 API | 面向查询应用和管理只读场景 |
| 1.2 | 定义 `service-api` 边界 | 仅承载 Query API，不承载管理写接口 |
| 1.3 | 定义 `service-admin` 边界 | 承载 Admin API 与 Desktop Gateway |
| 1.4 | 定义管理 API | 面向管理写操作、审核、任务创建 |
| 1.5 | 定义桌面任务网关 | 面向桌面端任务拉取、进度上报、结果回写 |
| 1.6 | 定义统一权限模型 | 明确 `actor`、`action`、`resource`，以及 RBAC + ABAC 的组合方式 |
| 1.7 | 定义内部模块职责 | 明确 Auth / Device、Query Read Model、Admin Command、Task Orchestrator、Asset Transfer、Audit / Notification 的边界 |
| 1.8 | 定义缓存与同步协议 | 明确 manifest、`versionTag`、`contentHash`、delta sync、容量预算、搜索分片与离线快照边界 |
| 1.9 | 定义对象存储上传策略 | 区分服务端代理与客户端直传 |
| 1.10 | 定义 `tasks`、`task_runs` 与 `device_leases` | 为后续实现统一任务调度与桌面执行尝试建模做准备 |
| 1.11 | 定义设备表与审计表字段 | 为设备授权、执行追踪和操作留痕做准备 |

### P2 管理网页端收口

| 序号 | 任务 | 说明 |
|------|------|------|
| 2.1 | 盘点现有 `site-console` 页面 | 标注为查询、轻量管理、中量管理、桌面专属，并收口到 `site-admin` |
| 2.2 | 收紧网页端入口 | 从网页端移除或禁用桌面专属操作 |
| 2.3 | 统一轻量任务流 | 将网页端操作转为调用管理 API 和 `tasks` 业务态流转 |
| 2.4 | 增加能力提示 | 在 UI 中明确提示“该操作仅桌面端可用” |

### P3 管理共享模块

| 序号 | 任务 | 说明 |
|------|------|------|
| 3.1 | 抽离管理领域模型 | `tasks`、`task_runs`、日志、导入、审核、设备等模型共享 |
| 3.2 | 抽离 API client | 统一鉴权、错误、分页、轮询逻辑 |
| 3.3 | 抽离能力开关层 | 根据运行端判断可见功能和可执行任务 |
| 3.4 | 抽离核心页面状态 | 表单、筛选、详情、日志等状态逻辑复用 |

### P4 Tauri 桌面端

| 序号 | 任务 | 说明 |
|------|------|------|
| 4.1 | 创建 `app-admin-desktop` | 为管理应用提供桌面运行时 |
| 4.2 | 设计本地能力接口 | 文件系统、目录选择、Git、工具调用、长任务事件 |
| 4.3 | 接入桌面登录与设备注册 | 登录后绑定 `deviceId`、平台、版本等信息 |
| 4.4 | 接入桌面任务执行 | 拉取 `full` 任务、执行、上报进度和结果 |
| 4.5 | 迁移本地脚本能力 | 将现有上传脚本、卡图导入脚本纳入统一能力层 |

### P5 Tauri 手机端

| 序号 | 任务 | 说明 |
|------|------|------|
| 5.1 | 创建 `app-admin-mobile` | 为管理应用提供移动运行时 |
| 5.2 | 实现通知与任务列表 | 支持提醒、查看状态、快速进入审核 |
| 5.3 | 实现审核与轻中量操作 | 完成审批、确认、重试等操作 |
| 5.4 | 屏蔽桌面专属能力 | 不暴露文件系统、Git、第三方工具等能力 |

### P6 查询应用独立化

| 序号 | 任务 | 说明 |
|------|------|------|
| 6.1 | 统一查询架构 | 让 `site-magic`、`site-hearthstone`、未来 `site-<game>` 与 `app-main` 遵循同一查询产品模型 |
| 6.2 | 固定查询集成方式 | 明确 `app-main` 负责跨游戏壳层能力，单游戏详情通过内嵌 `site-<game>` 页面消费 |
| 6.3 | 设计主动缓存策略 | 明确缓存对象分层、用户选中游戏预取、详情快照范围、失效规则、容量淘汰与联网模式切换 |
| 6.4 | 创建 `app-main` | 为统一查询体验提供安装型 app 宿主 |
| 6.5 | 对接查询 API | 聚焦高频查询体验、缓存与离线回退 |
| 6.6 | 接入用户态轻写操作 | 收藏、偏好、历史等 |
| 6.7 | 与管理应用彻底解耦 | 不共享管理入口、不依赖管理上下文 |

### P7 存储与上传优化

| 序号 | 任务 | 说明 |
|------|------|------|
| 7.1 | 设计短期上传凭证 | 桌面端可直接上传对象存储 |
| 7.2 | 设计断点与重试 | 大文件上传失败后支持恢复 |
| 7.3 | 设计产物索引回写 | 上传完成后将产物元数据写回中心系统 |
| 7.4 | 设计日志回传 | 让网页端和手机端可查看桌面任务执行日志 |

## 5. 推荐优先级

建议优先顺序如下：

1. 先完成 `P0 + P1`
2. 再完成管理网页端收口 `P2`
3. 然后做管理共享模块 `P3`
4. 再落地桌面端 `P4`
5. 手机端 `P5` 放在桌面端稳定之后
6. 查询应用独立化 `P6` 可与后期体验重构结合推进

## 6. 决策要点

在进入实现前，建议先确认以下事项：

- 查询应用是否需要在第一阶段同步推出移动端
- 管理网页端是否继续沿用现有 `site-console` 域名与部署方式
- 桌面端是否允许持有对象存储短期上传令牌
- 设备注册是否需要人工审批
- 哪些现有脚本必须在第一阶段并入桌面端统一能力层

## 7. 完成定义

当以下条件成立时，可认为该方案进入可实施状态：

- 双应用边界清晰
- 三端能力矩阵已确认
- 网络与数据库访问原则已确认
- 桌面端全能力职责已确认
- 中心服务职责已确认
- 后续实现可以直接据此拆分具体开发任务
