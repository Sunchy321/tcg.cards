# Tauri 双应用架构设计草案

## 1. 背景

当前系统同时存在以下几类需求：

- 面向普通用户的查询、浏览、收藏、登录后轻交互
- 面向运营或数据人员的导入、审核、上传、重试、批处理
- 强依赖本地环境的工作，例如读取本地 Git 仓库、调用第三方卡图生成工具、处理本地文件与产物

现有 `site-console` 已经同时承载了网页界面、服务端逻辑和部分本地脚本入口。随着管理能力变重，继续沿“单一网页站点”扩展，会遇到以下问题：

- 本地能力与浏览器安全边界冲突
- 同一界面同时承载轻操作和重操作，心智模型混杂
- 手机端、网页端、桌面端的能力边界不清晰
- 网络访问、数据库访问、凭证管理缺少统一约束

本提案将产品形态明确为“两类应用 + 管理应用三端能力分级”，并以 `Tauri` 作为桌面与移动端的统一技术栈。

## 2. 目标

- 定义两个应用：查询应用、管理应用
- 以 `Tauri` 作为原生壳技术栈，统一桌面端与移动端实现方向
- 为管理应用划分网页端、手机端、桌面端三种能力层级
- 明确哪些能力只能在桌面端执行
- 设计应用与中心服务、数据库、对象存储、本地工具之间的连接方式
- 为后续拆分现有 `site-console` 提供可执行实施计划

## 3. 非目标

- 本提案不包含具体代码实现
- 本提案不要求一步完成所有端形态发布
- 本提案不在 v1 中支持“任意端执行全部管理任务”
- 本提案不要求把数据库直接嵌入桌面端本地运行

## 4. 设计原则

### 4.1 按任务强度分端

- 网页端只处理轻量请求
- 手机端处理轻到中等请求
- 桌面端处理全部请求，特别是需要本地能力的重任务

### 4.2 按权限与能力分层

- 用户查询能力与运营管理能力分离
- 本地特权能力不暴露给普通网页端
- 高风险操作需要显式审计与状态追踪

### 4.3 中心化数据，端侧差异化执行

- 业务主数据、任务状态、审计日志统一保存在中心数据库
- 各端只获取其应有的能力，不复制完整后端职责
- 本地重任务由桌面端执行，但结果回写中心系统

### 4.4 先统一协议，再统一外壳

- 先定义服务边界、任务协议、能力分级
- 再分别落地网页端、移动端、桌面端

## 5. 应用拆分

### 5.1 查询应用

查询应用面向普通用户，核心特征是“以只读为主，带少量用户态写操作”。

这里的“查询应用”是产品分层概念。在当前模型下，它同时包含两种交付形态：

- 一组按游戏拆分的独立网站，例如 `site-magic`、`site-hearthstone`，以及未来的 `site-<game>`
- 一个对应的安装型主查询 app，用于承载统一的查询体验与用户态能力

查询侧的产品边界明确如下：

- `app-main`：承载 `site-main` 的内容能力，并整合所有 `site-<game>` 的查询入口与聚合体验；应用内提供配置，允许用户自定义显示哪些游戏
- `site-main`：作为入口页与内容聚合页，提供跨游戏导航、内容汇总与全游戏搜索；不承载各游戏详情页的主要逻辑
- `site-<game>`：承载单游戏的完整详情页、搜索结果细节、规则与领域展示逻辑；详情页仅存在于各 `site-<game>`

查询侧集成方式固定如下：

- `app-main` 原生承载跨游戏聚合页、全游戏搜索、用户配置、收藏、历史等跨游戏能力
- `app-main` 不实现独立的单游戏详情页，也不复制 `site-<game>` 的主要展示逻辑
- 当用户在 `app-main` 中进入某个游戏的详情、规则或复杂搜索结果细节时，统一以内嵌页面方式消费对应 `site-<game>` 路由
- 因此，单游戏复杂展示逻辑的唯一事实来源仍是各 `site-<game>`

主要职责：

- 卡牌、规则、版本、公告等信息查询
- 搜索、筛选、对比、详情浏览
- 用户登录后的轻写操作，例如收藏、历史、偏好设置
- 只读 API 的消费与展示

特点：

- 大部分业务是只读
- 网站形态继续按游戏拆分、独立域名部署
- 同时保留一个统一的查询 app 形态
- 查询 app 以只读查询为主，可附带少量用户态写操作
- 查询 app 必须包含主动缓存能力，用于离线使用与访问加速
- `site-main` 与 `app-main` 都提供跨游戏搜索
- 详情页逻辑保留在各 `site-<game>`，不回流到 `site-main`

### 5.1.1 查询侧职责矩阵

| 形态 | 入口导航 | 内容聚合 | 全游戏搜索 | 单游戏详情页 | 用户自定义游戏显示 |
|------|------|------|------|------|------|
| `site-main` | 支持 | 支持 | 支持 | 不支持 | 不支持 |
| `app-main` | 支持 | 支持 | 支持 | 通过内嵌对应 `site-<game>` 页面消费，不自带主详情逻辑 | 支持 |
| `site-<game>` | 可选 | 不作为主职责 | 支持单游戏范围内搜索与结果细节 | 支持 | 不支持 |

### 5.1.2 `app-main` 与 `site-<game>` 的固定集成方式

采用 **“跨游戏原生壳 + 单游戏内嵌页”** 模型：

- `app-main` 自身负责：
  - 首页聚合
  - 全游戏搜索入口
  - 用户选择显示哪些游戏
  - 收藏、历史、偏好等用户态能力
- `site-<game>` 继续负责：
  - 卡牌详情
  - 单游戏规则详情
  - 单游戏复杂检索结果细节
  - 其他高耦合单游戏展示逻辑

进入单游戏深层内容时：

- `app-main` 保留统一导航、账号态和壳层体验
- 内容区以内嵌页面承载对应 `site-<game>` 路由
- 不在 `app-main` 中再复制一套游戏详情页面实现

这样做的好处是：

- 保持“详情页仅存在于 `site-<game>`”这一约束
- 避免在 `app-main` 中重复实现各游戏展示逻辑
- 允许 `site-<game>` 独立演进其复杂页面

代价是：

- `app-main` 与 `site-<game>` 之间需要统一登录态与跳转协议
- 内嵌页面的导航、回退、主题和埋点需要额外约定

### 5.1.3 查询 app 主动缓存策略

`app-main` 必须内建主动缓存与离线读取能力。

目标：

- 提升常用内容的首屏速度
- 在弱网或离线场景下仍可浏览已缓存内容
- 让用户围绕“我关心哪些游戏”主动控制缓存范围

缓存范围建议至少包含：

- 用户配置
- 用户已选择显示的游戏列表
- 首页聚合流
- 全游戏搜索入口所需的基础索引与建议词
- 最近浏览与收藏的内容摘要
- 用户选中游戏的常用列表页摘要
- 用户最近访问过的单游戏详情快照

建议把缓存划分为四层：

#### 配置缓存

- 用户偏好
- 选中的游戏
- 主题与本地设置

#### 聚合缓存

- `site-main` / `app-main` 共用的跨游戏聚合数据
- 全游戏搜索建议词与热门入口

#### 游戏摘要缓存

- 选中游戏的列表页、卡片摘要、规则目录摘要
- 适合快速刷新和批量预取

#### 详情快照缓存

- 最近访问详情页的结构化快照
- 用于离线回退，不作为线上主渲染路径的替代实现

为了避免把缓存做成模糊概念，建议再把缓存对象固定为以下几类：

- `config entry`：用户配置、显示游戏集合、主题、本地开关
- `cross-game feed chunk`：首页聚合流、热门入口、跨游戏推荐块
- `search index shard`：全游戏搜索建议词、前缀索引、热门关键词
- `game summary chunk`：某个游戏的目录、列表、摘要卡片、规则树摘要
- `detail snapshot`：结构化详情快照，附带少量必要资源引用
- `asset blob`：快照依赖的图片、图标等静态资源

其中业务缓存的事实来源应保持清晰：

- 配置缓存来自本地用户态存储
- 聚合缓存、搜索索引、游戏摘要缓存来自 `service-api`
- 详情快照来自单游戏详情对应的结构化快照接口
- 静态资源缓存由 `asset blob` 与 WebView 自身 HTTP 缓存共同承担，但不把 WebView 缓存视为唯一离线保障

建议每个缓存条目统一具备以下元数据：

- `cacheKey`
- `scope`：至少区分匿名用户、登录用户、游戏、语言
- `versionTag`
- `contentHash`
- `fetchedAt`
- `staleAt`
- `expireAt`
- `lastAccessAt`
- `priority`
- `size`
- `source`

这样可以统一处理命中、失效、清理和诊断。

#### 缓存优先级

建议把主动缓存优先级固定为三档：

- **P0 必保缓存**：配置缓存、已选游戏集合、首页聚合流、搜索建议词
- **P1 常用缓存**：已选游戏的常用列表、规则目录、最近浏览摘要、收藏摘要
- **P2 机会缓存**：最近访问详情快照、手动钉住的离线内容、收藏详情快照

当空间不足时，先淘汰 `P2`，再淘汰 `P1`，`P0` 只在用户登出、切换账号或显式清理缓存时移除。

#### 主动同步触发

主动缓存触发时机建议包括：

- 用户首次登录后
- 用户修改“显示哪些游戏”配置后
- 用户进入某个游戏后
- 应用空闲、充电、可用网络满足条件时
- 手动触发“立即同步缓存”时

进一步建议细化为：

- **启动同步**：应用启动后先拉取 `config entry`、`cross-game manifest`
- **配置变更同步**：用户增减游戏时，立即补齐或回收对应 `game summary chunk`
- **访问驱动同步**：用户打开某游戏列表或详情后，提升该游戏缓存优先级，并预取相邻摘要或详情快照
- **后台同步**：仅在网络良好且设备空闲时执行增量同步，不阻塞前台交互
- **手动同步**：用户显式要求时允许突破保守阈值，立即刷新已选游戏缓存

#### 新鲜度与失效规则

建议采用“软失效 + 硬失效”双层模型：

- **软失效**：到达 `staleAt` 后仍可读，但界面要允许后台刷新
- **硬失效**：到达 `expireAt` 后默认不再作为可信在线数据，只能作为离线兜底并明确提示

建议默认新鲜度策略如下：

- 配置缓存：本地写入后立即生效，无固定 TTL
- 聚合缓存：短周期软失效，适合频繁 revalidate
- 搜索索引与游戏摘要缓存：中周期软失效，通过 manifest 增量更新
- 详情快照缓存：较长周期软失效，但一旦对应 `versionTag` 变化则立即失效

强制失效触发条件建议包括：

- 用户登出或切换账号
- 用户显示游戏集合变化
- 语言或内容地区切换
- 权限范围变化导致可见内容集合变化
- 服务端 manifest 声明条目删除、版本变更或 `contentHash` 不一致

#### 容量与淘汰策略

建议把缓存清理分成两套机制：

- **业务淘汰**：按 `priority` + `lastAccessAt` 清理 `detail snapshot` 与低优先级摘要块
- **资源淘汰**：按引用计数或最近访问时间清理 `asset blob`

同时建议保留两类显式保留能力：

- **自动保留**：最近访问、收藏、历史中的高频内容
- **手动保留**：用户手动标记“离线保留”的游戏或详情

#### 本地存储模型

考虑到桌面端与手机端都需要统一行为，建议 `app-main` 使用“元数据数据库 + 文件资源目录”模型：

- 本地数据库：保存缓存元数据、manifest、索引分片、摘要内容、快照正文
- 本地文件目录：保存图片、压缩资源包、较大搜索索引文件
- 本地队列表：保存待同步任务、失败重试、最近同步结果

数据库表建议至少包含：

- `cache_entries`
- `cache_manifests`
- `search_shards`
- `detail_snapshots`
- `asset_refs`
- `sync_jobs`

这样既方便桌面端容量管理，也方便手机端受限空间下做精细裁剪。

#### 详情快照格式约束

为了避免离线快照逐步演变成第二套详情页实现，建议详情快照只保存“可稳定复用的结构化结果”，而不是保存整页 HTML。

快照建议包含：

- 基础标识、标题、副标题、更新时间
- 结构化区块数据，例如摘要、属性表、规则段落、关联条目
- 所需资源引用
- 来源版本信息与生成时间
- 渲染提示信息，例如“此内容来自缓存快照”

不建议放入：

- 依赖在线脚本计算的复杂交互状态
- 临时实验区块
- 只能由在线环境动态拼装的个性化重逻辑

#### 联网模式策略

建议把运行状态拆成三种模式：

- **online**：默认走线上主路径，同时本地读缓存做首屏加速
- **weak**：优先显示本地缓存，再后台尝试刷新线上内容
- **offline**：直接走缓存与详情快照回退，不发起实时详情请求

状态切换应由网络质量、最近失败率与用户手动选择共同决定，而不是只看“是否联网”。

### 5.1.4 离线回退策略

`app-main` 在线时仍以“跨游戏原生壳 + 单游戏内嵌页”为主路径。

但在离线或弱网场景下，需要增加回退能力：

- 聚合页与搜索入口优先读取本地缓存
- 最近访问过的详情页允许显示“缓存快照”
- 如果目标内容没有可用快照，则明确提示需要联网

也就是说：

- **在线主路径**：内嵌 `site-<game>` 页面
- **离线回退路径**：读取本地详情快照

这不改变“单游戏详情逻辑唯一事实来源仍在 `site-<game>`”这一原则，只是允许 app 保存其结构化读取结果作为离线回退副本。

建议把查询读取流程进一步固定为：

1. 原生壳先查询本地缓存是否存在可用摘要或详情快照
2. 若处于 `online`，先显示本地摘要首屏，再拉起内嵌 `site-<game>` 页面
3. 若处于 `weak`，优先显示本地内容，并给出“正在尝试刷新”的轻提示
4. 若处于 `offline`，直接展示本地快照，并明确标注“离线缓存内容”
5. 若本地没有可用快照，则给出可理解的失败提示，而不是空白页

对于离线回退，建议补充以下约束：

- 全游戏搜索在离线模式下只搜索本地索引覆盖范围，并明确提示结果不是全量
- 收藏、历史、偏好等用户态轻写操作可先写本地队列，待恢复联网后再同步
- 不允许在离线模式下伪造需要强一致性的状态，例如在线审核结果、远程最新计数
- 当快照已过硬失效期但用户仍主动查看时，可展示只读内容，但必须附带陈旧提示

#### 内嵌页与快照的切换规则

建议固定以下切换原则：

- 能在线成功打开内嵌页时，以内嵌页为准
- 内嵌页加载失败但存在可用快照时，自动降级到快照
- 快照页不负责承接复杂互动，只负责稳定展示信息
- 当网络恢复后，从快照返回同一路径时，优先尝试重新拉起内嵌页

#### 账号与隐私边界

虽然大部分查询内容可以公开读取，但缓存仍应区分用户边界：

- 匿名缓存与登录用户缓存分开存储
- 收藏、历史、偏好、私人笔记等用户态数据只保存在对应用户作用域下
- 用户登出时默认清理用户私有缓存，但可保留公开只读缓存
- 如果未来支持多账号切换，禁止不同账号共享用户态缓存条目

### 5.2 管理应用

管理应用面向内部运营、数据、审核人员，核心特征是“任务驱动 + 人工判断 + 分端能力受控”。

主要职责：

- 数据源管理
- 导入任务创建与重试
- 审核、确认、驳回
- 卡图生成流程调度
- 本地 Git 仓库读取与数据上传
- 本地产物处理与上传
- 任务日志、结果、审计查看

管理应用分为三种形态：

#### 5.2.1 网页端

适合：

- 查看任务状态
- 发起轻量任务
- 审核、确认、驳回
- 查看日志、结果摘要
- 做只读管理查询

不适合：

- 读取本地 Git 仓库
- 调用本地第三方卡图生成工具
- 执行长时间本地计算
- 处理大量本地文件

#### 5.2.2 手机端

适合：

- 接收通知
- 查看任务进度
- 审核与批准
- 发起中等复杂度但不依赖本地环境的任务
- 查看异常并触发重试

不适合：

- 本地仓库操作
- 大文件导入
- 卡图生成
- 高强度批处理

手机端建议补充轻量离线缓存能力：

- 最近任务列表
- 待审核项摘要
- 最近通知
- 草稿型输入

这样可以提升弱网下的查看与确认效率，但不应让手机端在离线状态下提交高风险管理写操作。

#### 5.2.3 桌面端

适合：

- 全部管理能力
- 读取本地文件和 Git 仓库
- 调用本地 CLI、图像工具、生成器
- 处理长任务、批量任务、产物上传
- 承担“本地执行器”角色

桌面端是唯一必须支持“全能力”的管理终端。

桌面端建议补充本地任务缓存与恢复能力：

- 可执行任务队列摘要
- 最近执行日志索引
- 本地产物元数据
- 未完成任务的本地恢复点

这主要用于加速与恢复，不等于桌面端可以脱离中心服务长期离线运行。

## 6. 技术栈策略

### 6.1 总体策略

- 桌面端：`Tauri Desktop`
- 手机端：`Tauri Mobile`
- 网页端：复用同一套前端域模型与页面组件，但运行于普通 Web

### 6.2 代码组织建议

建议将“产品应用”与“运行时外壳”拆开：

```text
apps/
  site-main/               门户或导航站
  site-magic/              Magic 查询网站
  site-hearthstone/        Hearthstone 查询网站
  site-<game>/             未来其他游戏查询网站
  app-main/                查询应用宿主
  site-admin/              管理应用网页端
  app-console-desktop/     管理应用桌面端宿主
  app-console-mobile/      管理应用手机端宿主
  service-api/             查询 API 服务
  service-admin/           管理与桌面执行服务

packages/
  query-core/              查询应用共享前端模块
  app-console/             管理应用共享前端模块
  app-console-capabilities/  管理能力协议与类型
  app-api-client/          API client、鉴权与请求封装
```

其中：

- `site-magic`、`site-hearthstone`、未来的 `site-<game>` 与 `app-main` 共享 `packages/query-core` 中的跨游戏通用能力，但不共享“单游戏详情页实现”
- `site-admin`、`app-console-desktop`、`app-console-mobile` 共享 `packages/app-console`
- `app-console-desktop` 与 `app-console-mobile` 仅补充各自 Tauri 宿主、命令与能力桥接
- 网页端复用管理 UI，但只开启轻量能力

### 6.3 为什么不把三端做成完全三套应用

因为三端的业务模型相同，差别主要在于：

- 能力边界
- 运行时权限
- 本地资源访问方式

因此更适合“共享业务模块 + 按端注入能力”，而不是完全三套独立代码。

## 7. 管理应用能力分级

### 7.1 能力级别

定义三档能力：

- `light`：网页端可执行
- `medium`：手机端可执行
- `full`：仅桌面端可执行

这里的能力分级用于描述**任务执行能力**，不直接等于任务查看权限。任务可见性仍应由角色、资源范围和审计规则单独控制。

### 7.2 任务分级表

| 任务类型 | 网页端 | 手机端 | 桌面端 | 说明 |
|------|------|------|------|------|
| 查询任务状态 | 支持 | 支持 | 支持 | 纯读取 |
| 审核、批准、驳回 | 支持 | 支持 | 支持 | 人工决策 |
| 创建轻量导入任务 | 支持 | 支持 | 支持 | 不依赖本地文件 |
| 触发远端重试 | 支持 | 支持 | 支持 | 调用中心服务 |
| 查看错误日志 | 支持 | 支持 | 支持 | 纯读取 |
| 中等批量审批 | 可选 | 支持 | 支持 | 手机端需要优化交互 |
| 上传拍照或轻量附件 | 不优先 | 支持 | 支持 | 偏移动端场景 |
| 导入本地文件 | 不支持 | 不支持 | 支持 | 依赖文件系统 |
| 读取本地 Git 仓库 | 不支持 | 不支持 | 支持 | 强本地依赖 |
| 调用卡图生成工具 | 不支持 | 不支持 | 支持 | 第三方本地工具 |
| 生成大量产物后上传 | 不支持 | 不支持 | 支持 | 高 I/O、高耗时 |
| 本地调试工具链 | 不支持 | 不支持 | 支持 | 开发/运维型操作 |

## 8. 网络与数据库设计

## 8.1 总体原则

- 客户端默认不直接连接数据库
- 所有端优先通过中心 API 访问业务数据
- 只有桌面端在极少数受控场景下允许直连对象存储或上传通道
- 数据库连接集中在中心服务层管理

## 8.2 中心服务拆分

建议后端分为三层：

### 8.2.0 推荐后端模型

推荐采用 **“模块化单体 + 三类协议入口”**，而不是一开始就拆成大量微服务。

原因：

- 当前业务仍在快速变化，任务模型、权限模型、桌面执行协议都未完全稳定
- 查询、管理、桌面执行虽然入口不同，但共享大量领域模型、鉴权、审计、对象存储与任务状态
- 过早拆微服务会显著增加发布、联调、追踪、权限同步和数据一致性成本

因此，推荐先按“逻辑服务”分层，再按需要决定是否拆成独立部署单元。

推荐的逻辑模型如下：

```text
clients
  site-magic / site-hearthstone / site-<game>
  app-main
  site-admin
  app-console-mobile
  app-console-desktop

service layer
  Query API
  Admin API
  Desktop Gateway

application modules
  Auth / Session / Device
  Query Read Model
  Admin Command
  Task Orchestrator
  Asset Transfer
  Audit / Operation Log
  Notification

infrastructure
  PostgreSQL
  R2 / Object Storage
  Search Index
  Third-party upstream APIs
```

这里要点是：

- **入口分开**：读、写、桌面执行协议分开
- **领域共享**：任务、鉴权、设备、审计在同一后端内统一维护
- **部署渐进**：早期可以是一个 `service-*` 应用中按模块组织，后期再拆

### 8.2.0.1 v1 部署建议

v1 不建议直接做大量 `service-*`。

当前已确定的服务拆分为：

- `service-api`：只承载 Query API
- `service-admin`：承载 Admin API 与 Desktop Gateway

也就是说：

- **查询服务独立**
- **管理写操作与桌面执行协议先合并**

当出现以下信号时，再考虑把 `service-admin` 内的 Desktop Gateway 单独拆成独立服务：

- 桌面任务轮询、日志上传、产物回写流量明显上升
- 长连接、轮询或事件流与普通管理请求互相干扰
- 桌面协议迭代频率与管理 API 发布节奏明显分化

### 8.2.0.2 模块职责

#### Auth / Session / Device

负责：

- 用户登录态
- 管理角色与权限
- 桌面设备注册
- 设备令牌签发与校验

它不直接承载业务页面逻辑，而是为 Query API、Admin API、Desktop Gateway 提供统一身份上下文。

#### Query Read Model

负责：

- 面向查询应用的只读聚合输出
- 面向站点与查询 app 的统一读模型
- 缓存友好的列表、详情、搜索、摘要接口

建议进一步拆成两类读模型：

- `cross-game read model`：供 `site-main` 与 `app-main` 使用，负责入口页聚合、全游戏搜索、跨游戏摘要
- `game read model`：供 `site-<game>` 使用，负责单游戏详情、规则、版本、列表、搜索结果细节

前端共享边界建议同步固定：

- `query-core` 只承载跨游戏通用能力与壳层能力，例如导航、全游戏搜索入口、收藏、历史、偏好、聚合卡片
- 单游戏详情页组件、复杂规则展示组件、游戏专属搜索结果细节组件保留在各 `site-<game>`

这一层应尽量避免把后台导入态、任务态直接暴露给用户端。

### 8.2.0.3 App 缓存支持

后端需要把 app 主动缓存视为正式能力，而不是纯客户端优化。

建议服务端为 app 提供以下支撑：

- 缓存清单（manifest）
- 版本标识（`versionTag` / `contentHash`）
- 增量同步（delta sync）
- 资源签名 URL 或受控下载入口
- 最近访问详情快照的结构化读取结果

建议至少提供两类缓存清单：

- `cross-game manifest`：面向 `app-main` 的聚合页、全游戏搜索与用户选中游戏集合
- `game manifest`：面向单个游戏的摘要内容、目录、常用列表和可缓存详情快照

缓存更新策略建议：

- 默认读取本地缓存并后台 revalidate
- 当 `versionTag` 或 `contentHash` 变化时刷新本地条目
- 对用户未选中的游戏不做大规模主动预取
- 对最近访问、收藏、历史记录相关内容给予更高缓存优先级

建议进一步明确服务端职责：

- Query API 返回结果需要尽量稳定、可缓存、可分块
- Manifest 需要显式声明条目类型、依赖资源、版本、删除列表
- 详情快照接口需要返回结构化结果，而不是只能返回页面 HTML
- 搜索索引需要支持分片下载和增量替换，避免每次全量重拉
- 资源下载入口需要支持签名、过期控制和断点续传

#### 建议的缓存协议元素

每份 manifest 建议至少包含：

- `manifestId`
- `scope`
- `generatedAt`
- `baseVersion`
- `entries`
- `deletedEntries`

每个 `entry` 建议至少包含：

- `entryKey`
- `entryType`
- `versionTag`
- `contentHash`
- `priority`
- `size`
- `resourceRefs`

#### 增量同步策略

建议 Query API 至少支持三种同步模式：

- **bootstrap**：首次安装或首次选择某游戏时拉取完整基础集
- **delta**：基于 `baseVersion` 获取新增、变更、删除条目
- **snapshot refresh**：针对单个详情快照按需刷新

同步接口需要允许客户端携带：

- 当前 manifest 版本
- 已缓存条目摘要
- 设备能力信息，例如存储预算、网络模式
- 用户当前选中的游戏集合

服务端则根据这些信息返回：

- 建议更新的条目集合
- 建议删除的条目集合
- 是否需要强制重建本地索引
- 可选资源下载列表

#### 查询服务与缓存服务边界

当前不建议单独拆一个缓存服务。

更合适的方式是：

- 由 `service-api` 继续承载缓存协议接口
- 在 `service-api` 内部把 manifest 生成、索引分片、快照输出作为独立模块
- 等缓存规模、下载压力、生成时延明显成为瓶颈后，再考虑拆成独立缓存分发层

#### Admin Command

负责：

- 审核、批准、驳回
- 创建任务、取消任务、重试任务
- 维护配置与来源状态
- 获取上传凭证

它的特点是：

- 强鉴权
- 强审计
- 以命令式写操作为主

#### Task Orchestrator

负责：

- 统一任务表状态流转
- 任务去重、重试、超时、取消
- `light` / `medium` / `full` 能力校验
- 分派桌面端任务

它是管理体系的核心，不应散落在各个页面 handler 中。

#### Asset Transfer

负责：

- 上传凭证生成
- 大文件与产物元数据登记
- R2 key 规划
- 上传完成后的结果确认

它不负责本地生成产物，但负责“让产物安全地进系统”。

#### Audit / Operation Log

负责：

- 谁在什么时间发起了什么操作
- 任务状态变化轨迹
- 审核动作记录
- 桌面端执行上报摘要

所有管理写操作都应进入这一层。

#### Notification

负责：

- 手机端提醒
- 桌面任务状态通知
- 失败告警
- 可选的邮件或系统消息投递

这一层在 v1 可以较轻，但接口边界应预留。

### 8.2.1 只读查询 API

面向查询应用与管理应用的读请求：

- 卡牌查询
- 文档查询
- 任务状态查询
- 审核结果查询
- 日志摘要查询
- 缓存清单与增量同步

特点：

- 可部署在 Cloudflare / 边缘环境
- 优先做缓存与权限裁剪
- 不承载本地重任务

### 8.2.2 管理 API

面向管理应用的写操作和任务调度：

- 创建任务
- 改状态
- 提交审核结果
- 获取上传凭证
- 申请桌面任务执行令牌

特点：

- 必须有严格鉴权
- 必须有审计日志
- 不直接在请求链路中做重型本地工作

该组接口归属于 `service-admin`。

### 8.2.3 桌面任务网关

用于桌面端获取“全能力任务”执行指令：

- 拉取桌面专属任务
- 上报进度
- 上传执行日志
- 回写结果和产物索引

这层属于 `service-admin`，但协议上应与管理网页/手机端使用的 Admin API 单独分组。

### 8.2.4 为什么是三入口而不是一个通用 API

不建议把所有请求都塞进一个“万能 API”：

- 查询应用追求读性能、缓存命中和稳定输出
- 管理应用追求权限、审计、命令语义和状态机约束
- 桌面网关追求长任务、设备身份、进度上报和产物回写

三者的请求模式、错误模型、限流对象、幂等要求都不同。

因此建议：

- Query API：面向读
- Admin API：面向写
- Desktop Gateway：面向执行

它们可以共享内部模块，但不应共享同一套对外语义。

在当前拆分下：

- `service-api` 对外提供 Query API
- `service-admin` 对内提供 Admin API 与 Desktop Gateway

## 8.3 数据库访问原则

### 8.3.1 默认模式

所有应用都通过 API 访问数据库，不在客户端直接持有 `DATABASE_URL`。

原因：

- 避免数据库凭证下发到客户端
- 便于统一权限、审计、限流
- 便于后续增加缓存与服务编排

### 8.3.2 为什么不让桌面端直接连数据库

虽然桌面端是内部工具，但仍不建议默认直连数据库：

- 凭证泄露风险高
- 版本升级后 SQL 协议与业务约束难统一
- 难以做审计与细粒度授权
- 手机端、网页端无法复用相同权限模型

### 8.3.3 可接受的例外

只有在以下场景，才允许设计受控直连：

- 临时维护工具
- 一次性迁移脚本
- 明确仅开发环境使用

这类能力不属于正式产品能力，不纳入管理应用主架构。

## 8.4 对象存储与大文件连接

对于 R2 或类似对象存储，建议如下：

- 普通读：由中心服务代理或签名 URL 下发
- 普通写：由管理 API 下发短期上传凭证
- 大文件或大量产物写入：桌面端使用短期凭证直传对象存储

这样可以避免：

- 所有大文件都经过 API 服务转发
- 桌面端永久持有高权限对象存储密钥

## 8.5 本地工具连接设计

桌面端通过 `Tauri` 提供受控能力桥接：

- 文件系统访问
- Git 命令调用
- 第三方卡图生成工具调用
- 本地目录选择
- 长任务进度事件

这些能力不应由前端页面直接拼接命令，而应通过一组稳定命令接口暴露，例如：

- `selectRepo()`
- `scanRepo()`
- `generateCardImages()`
- `uploadArtifacts()`
- `reportTaskProgress()`

前端只调用能力接口，不直接操作底层命令细节。

## 8.6 鉴权模型

### 8.6.0 权限设计原则

权限模型建议统一采用 **“认证 → 授权 → 资源约束”** 三层判断。

也就是说，每个请求都应依次回答三个问题：

1. 你是谁？
2. 你能做什么？
3. 你能操作哪个资源？

不建议把这三类判断混写在各个 handler 的临时 `if` 里，而应下沉为统一的服务端权限层。

### 8.6.0.1 主体模型

后端统一识别以下主体类型：

- `anonymous`
- `user`
- `device`
- `api_key`
- `service`

建议在服务端 context 中统一归一为 `actor`：

- `actor.type`
- `actor.userId`
- `actor.deviceId`
- `actor.roles`
- `actor.allowedGames`
- `actor.taskCapabilityMax`
- `actor.capabilities`

这样不同入口可以共享同一套授权判断，而不是各自维护一套用户信息结构。

### 8.6.0.2 授权层级

建议采用：

- **RBAC 作为主模型**
- **ABAC 作为补充约束**

其中：

- RBAC 负责“这个主体大致能做哪类事”
- ABAC 负责“这个主体能操作哪些游戏、哪些任务级别、哪些资源”

建议角色至少包含：

- `owner`
- `admin`
- `operator`
- `reviewer`
- `viewer`

建议补充属性至少包含：

- `allowedGames`
- `taskCapabilityMax`
- `canApproveReview`
- `canManageSource`
- `canUploadAsset`
- `canUseDesktopRunner`

### 8.6.0.3 服务准入规则

#### `service-api`

可接受主体：

- `anonymous`
- `user`
- `api_key`
- `service`

职责：

- 仅提供 Query API
- 不承载管理写接口
- 不承载桌面执行协议

#### `service-admin`

可接受主体：

- `user`
- `device`
- `service`

职责：

- 提供 Admin API
- 提供 Desktop Gateway

约束：

- 普通 `api_key` 默认不允许进入 `service-admin`
- 桌面端执行协议仅允许 `device` 主体访问

### 8.6.0.4 动作模型

建议每个接口都声明明确动作，不在业务代码中隐式判断。

例如：

- `query.search`
- `query.card.read`
- `query.document.read`
- `admin.task.create`
- `admin.task.cancel`
- `admin.review.approve`
- `admin.source.manage`
- `admin.asset.upload`
- `desktop.task.claim`
- `desktop.task.heartbeat`
- `desktop.task.report`

服务端统一执行：

- `authenticate(request)`
- `authorize(actor, action, resource)`

handler 只声明需要的 `action` 与资源上下文。

### 8.6.0.5 资源约束

就算动作允许，仍应继续校验资源范围。

建议至少包括：

- 游戏范围：`allowedGames`
- 任务能力上限：`taskCapabilityMax`
- 是否只能访问自己创建的任务
- 是否只能操作某些来源
- 是否只能认领某些任务类型

这样可以避免“同一角色对所有资源全放开”的过粗授权。

### 8.6.1 查询应用

- 用户登录态为主
- 只读接口可配合匿名访问或轻权限访问

建议进一步细化为：

- 公开只读内容：允许 `anonymous`
- 个性化只读内容：要求 `user`
- 第三方程序化查询：允许 `api_key`

### 8.6.2 管理应用网页端与手机端

- 使用统一账号体系
- 必须走服务端权限校验
- 根据角色控制是否可执行 `light` / `medium` 能力

建议动作示例：

- `admin.task.create`
- `admin.task.cancel`
- `admin.review.approve`
- `admin.source.manage`

并结合：

- `allowedGames`
- `taskCapabilityMax`

做资源约束。

### 8.6.3 管理应用桌面端

- 使用同一账号体系登录
- 登录后向中心服务申请桌面执行令牌
- 桌面任务网关根据设备身份、用户身份、角色决定是否下发 `full` 任务

桌面端不应只靠“用户已登录”放行，建议增加设备维度授权：

- 用户登录
- 设备注册
- 设备签发短期执行令牌
- 服务端在每次认领任务时校验设备令牌、用户角色、游戏范围、能力范围、协议兼容性

建议额外记录：

- `deviceId`
- `deviceName`
- `platform`
- `appVersion`
- `lastSeenAt`
- `protocolVersion`
- `capabilities`

用于审计与设备授权。

建议桌面端设备令牌至少绑定：

- `userId`
- `deviceId`
- `roles`
- `allowedGames`
- `taskCapabilityMax`
- `protocolVersion`
- `capabilities`

### 8.6.4 任务权限校验落点

结合 `tasks` / `task_runs` / `device_leases`，建议按下列方式校验：

#### 创建任务

- 校验 `admin.task.create`
- 校验目标游戏是否在 `allowedGames`
- 校验目标 `capabilityLevel` 是否不超过 `taskCapabilityMax`

#### 认领任务

- 校验 `desktop.task.claim`
- 校验当前主体为 `device`
- 校验目标游戏是否在设备允许范围
- 校验目标任务是否允许当前设备能力执行
- 校验不存在冲突中的 `active` lease

#### 上报进度和结果

- 校验 `desktop.task.heartbeat` 或 `desktop.task.report`
- 校验 `runId + leaseToken + deviceId` 一致
- 校验任务当前执行权仍然有效

### 8.6.5 审计要求

所有管理写操作和桌面执行操作都应进入审计链。

建议至少记录：

- `actor.type`
- `actor.userId`
- `actor.deviceId`
- `action`
- `resourceType`
- `resourceId`
- `requestId`
- `taskId`
- `runId`
- `createdAt`

这样才能在后续排查“谁创建、谁认领、谁失败、谁取消”时保持可追溯。

## 9. 任务模型设计

### 9.1 任务定义表 `tasks`

`tasks` 用于描述“业务任务本身”，而不是每次执行尝试。

它负责描述：

- 任务类型
- 参数
- 期望能力级别
- 业务级当前状态
- 审计信息

建议字段：

- `id`
- `taskType`
- `capabilityLevel`：`light` / `medium` / `full`
- `status`：`queued` / `running` / `completed` / `failed` / `cancelled`
- `payload`
- `result`
- `error`
- `createdBy`
- `assignedDeviceId`
- `currentRunId`
- `runCount`
- `lastRunAt`
- `createdAt`
- `updatedAt`

说明：

- `tasks.status` 表示任务的业务态
- `tasks.result` 表示任务最终业务结果
- `tasks.error` 仅保留最终失败摘要，不承载所有尝试细节

### 9.2 执行尝试表 `task_runs`

`task_runs` 用于记录一次具体执行尝试。

它负责描述：

- 谁认领了任务
- 这是第几次执行
- 这次执行在什么设备、什么协议版本上发生
- 这次执行的进度、日志摘要与失败原因

建议字段：

- `id`
- `taskId`
- `attempt`
- `status`：`queued` / `leased` / `running` / `completed` / `failed` / `cancelled` / `expired`
- `runnerType`：`web` / `mobile` / `desktop`
- `deviceId`
- `protocolVersion`
- `capabilities`
- `leaseToken`
- `leaseExpiresAt`
- `startedAt`
- `heartbeatAt`
- `finishedAt`
- `result`
- `error`
- `logSummary`
- `createdAt`
- `updatedAt`

说明：

- `task_runs.status` 表示某次尝试的执行态
- `task_runs.result` 表示该次尝试产出的结果摘要
- 详细日志可继续落对象存储或独立日志表，`logSummary` 只保留索引和摘要

### 9.3 设备租约 `device_leases`

对于 `full` 任务，建议显式保留设备租约模型，避免多个桌面端同时执行同一任务。

建议字段：

- `id`
- `taskId`
- `runId`
- `deviceId`
- `leaseToken`
- `status`：`active` / `released` / `expired`
- `acquiredAt`
- `expiresAt`
- `releasedAt`
- `createdAt`
- `updatedAt`

如果第一版不单独建表，也至少应在设计里保留等价语义，不要把租约信息完全隐含在 `tasks` 或 `task_runs` 的少数字段中。

### 9.4 调度规则

- `light` 任务：网页端、手机端、桌面端均可发起与执行
- `medium` 任务：手机端和桌面端可执行
- `full` 任务：只能由桌面端执行

补充规则：

- `tasks` 决定业务态
- `task_runs` 决定执行尝试态
- `device_leases` 决定当前执行权
- 同一时刻一个 `full` 任务最多只允许一个 `active` lease

### 9.5 桌面端执行流

1. 用户在管理应用中创建 `full` 任务
2. 管理 API 写入 `tasks`
3. `service-admin` 为该任务创建首个 `task_runs`
4. 桌面端轮询或订阅可执行任务
5. 桌面端领取租约，写入 `device_leases`
6. 桌面端执行本地工作，并持续回传 `heartbeat`、进度、日志、产物
7. 执行成功或失败后，更新 `task_runs`
8. `service-admin` 根据执行结果推进 `tasks` 业务态
9. 其他端读取 `tasks` 摘要，并在需要时展开查看 `task_runs`

## 10. 推荐实施顺序

### P0 架构落位

- 明确查询应用与管理应用边界
- 明确管理能力分级
- 确认中心 API 边界

### P1 管理网页端收口

- 将现有 `site-console` 识别为 `site-admin` 的前身
- 清理只属于桌面端的功能入口
- 为任务分级打标签

### P2 抽离管理共享模块

- 抽离管理应用共享页面、状态、API client、领域模型
- 让网页端先消费共享模块

### P3 引入 Tauri 桌面端

- 创建管理应用桌面宿主
- 接入文件系统、Git、第三方工具调用
- 接入桌面任务网关

### P4 引入 Tauri 手机端

- 优先实现通知、审核、状态查看、轻中量操作
- 不承载 `full` 能力

### P5 查询应用独立化

- 将查询应用从管理能力中彻底分离，并保持“按游戏独立站点”模型
- 同时提供对应的统一查询 app 形态
- 保持查询侧以只读为主、用户态写为辅

## 11. 风险与权衡

### 11.1 使用 Tauri 的收益

- 桌面端与移动端可共享更大比例的前端模块
- 比纯网页更适合受控本地能力接入
- 比完全三套原生应用更节省成本

### 11.2 使用 Tauri 的代价

- 需要处理桌面与移动的宿主差异
- 需要设计能力桥接层
- 现有 Nuxt 全栈结构不能原样直接平移为全部端共享

### 11.3 最大架构风险

如果不尽早把“中心服务职责”和“桌面本地职责”切开，后续会再次回到“一个应用既像网站又像本地脚本集合”的混乱状态。

## 12. 结论

建议采用以下总体方案：

- 查询应用：独立产品线，以只读查询为主
- 管理应用：独立产品线，采用网页端 + Tauri 手机端 + Tauri 桌面端三形态
- 端能力分级：网页端 `light`，手机端 `medium`，桌面端 `full`
- 网络策略：所有端优先走中心 API，不默认直连数据库
- 数据库策略：数据库集中在服务端访问
- 本地重任务策略：仅桌面端通过 Tauri 能力桥接执行，并回写中心系统

该方案兼顾了：

- 当前本地重任务的现实需求
- 后续手机端扩展能力
- 长期的权限、审计、任务协调一致性
