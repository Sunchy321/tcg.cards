## 1. 实施策略

本计划采用“先基础设施，后模块迁移；先轻页面，后重任务；先桌面宿主，后多端复用”的顺序。

迁移期间保持以下约束：

- `site-console` 持续可用
- 新能力优先沉淀到共享层
- 桌面端不直接复制 Nuxt 专属依赖
- 本地能力通过 capability 层进入，不直接散落在页面中

## 2. 阶段计划

| 阶段 | 目标 | 核心任务 | 输出物 | 依赖 | 验收标准 |
|------|------|----------|--------|------|----------|

| P1 基础共享层 | 让后续迁移不再复制代码 | 抽离认证、API client、错误处理、权限判断、基础布局与通用表单模式 | `packages/app-console`、`packages/app-api-client`、capability 草案 | P0 | 后续页面迁移可复用统一 client 与布局 |
| P2 轻页面迁移 | 先让桌面端可日常进入使用 | 迁移首页、用户页、设置页、Magic 与 Hearthstone 入口页 | 桌面端基础页面 v1 | P1 | 基础导航完整，用户可在桌面端稳定浏览主入口 |
| P3 常规模块迁移 | 打通常规管理工作流 | 迁移公告、系列、标签等 CRUD 页面 | 常规模块 v1 | P2 | 桌面端可独立完成常规编辑、保存、删除操作 |
| P4 复杂模块迁移 | 接管复杂远端管理页 | 迁移规则、数据源、变更比对、树形查看与编辑器相关页面 | 复杂模块 v1 | P3 | 复杂模块可在桌面端运行，且不依赖 Nuxt runtime |
| P5 桌面重任务迁移 | 释放桌面端的真正价值 | 迁移图片导入、仓库上传、导入投影与本地文件/Git/工具链流程 | 桌面专属能力 v1 | P4 | `full` 能力不再依赖 `site-console` 或外部手工脚本 |
| P6 网页端收口 | 完成管理端职责重分配 | 将 `site-console` 收口为 `site-admin`，仅保留 `light` 能力 | 管理端职责新边界 | P5 | 桌面端承接完整能力，网页端只保留轻量能力 |

## 3. 详细任务拆解

| 0.3 | 建立脚本清单 | 盘点 `hsdata-upload`、图片导入脚本及其本地依赖 |
| 0.4 | 初始化桌面壳层 | 完成路由、布局、登录页、会话持久化、基础导航 |

## 4. 建议执行顺序

建议按以下顺序推进：

1. 完成 `P0`
2. 立即完成 `P1`
3. 用 `P2` 验证桌面端基础可用性
4. 用 `P3` 建立可复用的标准业务页面模式
5. 再推进 `P4`
6. 最后推进 `P5` 与 `P6`

## 5. 每阶段验收重点

### P0 / P1

- 桌面端能独立登录
- 不再依赖 Nuxt runtime 才能发起管理请求
- 共享层边界清楚

### P2 / P3

- 页面已可在桌面端正常导航和保存
- 表单与列表模式已稳定
- 页面逻辑没有继续绑死在宿主实现里

### P4

- 复杂页面已能脱离 Nuxt 插件体系运行
- 复杂状态已抽成可复用组合层

### P5

- 本地文件、Git、工具链能力不再依赖手工脚本
- 长任务具备进度、日志、失败重试能力

### P6

- 网页端与桌面端职责边界明确
- `full` 能力只在桌面端存在

## 6. 结论

建议从现在开始，把 `app-console-desktop` 作为管理端实现主线，把 `site-console` 视为待拆分、待收口的现有系统。

最先要做的不是直接搬某个复杂页面，而是先完成：

- 共享认证与 API client
- 桌面管理壳层
- capability 抽象

完成这些之后，再逐步迁移具体业务模块，整体风险最低。
