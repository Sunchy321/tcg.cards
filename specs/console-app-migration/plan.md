## 1. 实施策略

本计划采用“先共享 Nuxt layer，后页面回收；先轻页面，后重任务；先共享壳层，后宿主专属能力”的顺序。

迁移期间保持以下约束：

- `site-console` 持续可用
- 新能力优先沉淀到共享层
- 页面结构优先收敛到 `packages/console-shell`，不继续在宿主内复制薄 wrapper
- 本地能力通过 capability 层进入，不直接散落在页面中

## 4. 建议执行顺序

建议按以下顺序推进：

1. 完成 `P0`
2. 立即完成 `P1`
3. 先落 `P2`，固定 `console-shell` 作为共享 Nuxt 页面壳层
4. 用 `P3` 验证共享 layer 是否能稳定承载轻页面
5. 用 `P4` 建立可复用的标准业务页面模式
6. 再推进 `P5`
7. 最后推进 `P6` 与 `P7`

## 5. 每阶段验收重点

### P0 / P1 / P2

- 桌面端能独立登录
- 不再依赖 Nuxt runtime 才能发起管理请求
- 共享层边界清楚
- `console-shell` 的目录与依赖方向固定完成

### P3 / P4

- 页面已可在桌面端正常导航和保存
- 表单与列表模式已稳定
- 页面逻辑没有继续绑死在宿主实现里

### P5

- 复杂页面已能脱离 Nuxt 插件体系运行
- 复杂状态已抽成可复用组合层

### P6

- 本地文件、Git、工具链能力不再依赖手工脚本
- 长任务具备进度、日志、失败重试能力

### P7

- 网页端与桌面端职责边界明确
- `full` 能力只在桌面端存在

## 6. 结论

建议从现在开始，把 `packages/console-shell` 作为共享页面壳层主线，把 `app-console-desktop` 与 `site-console` 视为其宿主。

最先要做的不是直接搬某个复杂页面，而是先完成：

- 共享认证与 API client
- 共享 Nuxt layer 落点
- capability 抽象

完成这些之后，再逐步迁移具体业务模块，整体风险最低。
