# `magic-data-import-p0-design.md` 评审意见

## 评审结论

通过，可以进入万智牌导入 P0 的实施计划阶段。

## 通过理由

- 已经把范围收窄到万智牌，避免在 P0 同时处理跨游戏抽象
- 已经明确 6 类目标实体与可接受的匹配键，避免后续实现时引入模糊匹配
- 已经把 `gatherer`、`scryfall`、`mtgch`、`mtgjson` 的职责拆开，避免来源重叠时规则失控
- 已经给出字段路径注册表边界，并明确哪些字段不允许进入字段规则系统
- 已经明确来源字段覆盖能力与单条记录字段缺失语义，避免把缺失误判为清空
- 已经加入前端只读快照页与快照生成机制，用于展示数据源、字段来源状态与导入策略
- 已经把执行模式从“默认覆盖”收口为 `auto_apply`、`batch_review`、`manual_review`
- 已经明确 `magic` / `magic_data` / `magic_app` 的职责与 authoritative 边界
- 已经把 `__lockedPaths` 与 `__updations` 的兼容策略说清楚，便于平滑迁移

## 必须坚持的边界

- `fieldPath` 必须有注册表校验，不能允许自由字符串配置
- `mtgjson` 在 P0 不能默认落库，只能通过显式字段白名单放开
- `mtgch` 不得使用卡名模糊匹配中文数据
- `null` 只能表示来源明确声明为空，不能表示来源不支持或单条记录未提供
- `not_provided`、`not_applicable`、`parse_failed` 不得生成 `FieldChange`
- 前端只读页面只能读取已发布快照，不得直接拼装实时规则或触发导入、审批、自动应用或规则写入
- `decisionStatus` 只能是派生状态，不能成为审批事实来源
- 新导入链路不得继续把候选写入主表 `__updations`

## 需要注意的问题

### 1. `card` 与 `print` 的语义重叠

当前主表同时存在 `card.name / text` 与 `print.name / text`。实现时需要明确：

- `card.*` 更偏 oracle/卡牌级事实
- `print.*` 更偏印刷级展示事实

若来源只提供一套文字，必须在标准化阶段先决定它落到哪一层，不能在规则阶段再猜。

### 2. 批量审批分组键必须稳定

`batch_review` 如果只按 `fieldPath` 分组，容易把不同语言、不同来源、不同规则命中的更新混成一批。设计中要求至少包含 `sourceId`、`entityType`、`fieldPath`、`ruleSetId`、`locale/lang`、`reasonCode`，这一点实现时必须严格遵守。

### 3. `__lockedPaths` 的优先级

实现时必须保证：

- 锁定判断发生在自动应用之前
- 锁定字段即使来源可信，也不能绕过审批直接写入
- override 行为必须进入 `magic_app.import_review_actions`

### 4. 日志体积控制

文档已经把冷热分层放到进阶目标，这是合理的。但 P1 migration 阶段仍建议提前预留 `storageMode`、`hash`、`ref` 相关字段，避免二次迁移。

### 5. 缺失字段统计

`not_provided` 与 `not_applicable` 不应进入审批队列，但仍建议在 `import_runs` 或原始记录诊断信息中保留统计，方便识别来源质量下降或适配器字段映射错误。

### 6. 快照与真实规则的一致性

前端只读页面如果直接使用手写静态结构，容易和后续数据库规则产生分叉。P0 实现时应把展示结构设计成由“已发布规则 -> 快照”编译生成，而不是写成只服务页面的临时结构。

### 7. 快照发布与缓存

快照必须在“发布”时生成，而不是页面访问时即时生成。否则：

- 页面访问会承担不必要的编译成本
- 缓存失效策略会变复杂
- 很难解释用户看到的是哪一个配置版本

建议快照带上 `version` 和 `publishedAt`，并通过缓存或 `ETag` 分发给前端。

## 实施建议

- 第一版先实现静态字段路径注册表，再考虑从 schema 代码生成
- 第一版先实现来源字段覆盖矩阵与标准化字段状态，再实现 diff
- 第一版先把只读页面做成“读取快照的配置可视化”，不要加入编辑、执行或审批按钮
- 第一版先把 `auto_apply` 限制在外部 ID 与图片 URI 类字段
- 第一版先把 `mtgjson` 做成对账来源，待规则稳定后再放开写入
- 第一版优先打通 `scryfall`、`gatherer`、`mtgch` 三个真实来源
- 第一版在审批 UI 中就要预留按 `sourceId`、`fieldPath`、`locale/lang` 批量过审的能力
