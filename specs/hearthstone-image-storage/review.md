# 炉石图片 R2 存储与第三方导入方案评审

## 评审结论

有条件通过，可以作为炉石图片生成、缺图导出和第三方 PNG 压缩包导入的实施方案基础。

通过条件：

- 实施前确认首版 `zone/template/premium` 枚举是否覆盖真实渲染需求
- 实施前确认第三方工具接受单 JSON 需求文件，并按指定文件名返回 PNG ZIP
- 实施前确认单需求文件默认上限和硬上限
- 实施前确认 R2 bucket 与公开访问域名的最终命名
- 实施前确认控制台导入和本地脚本导入是否复用同一套校验转换逻辑

## 通过理由

- 图片二进制、图片索引和领域事实分层清楚
- R2 key 使用 `imageSpecVersion + variant + renderHash`，兼顾去重、可读性和批量运维
- 变体维度拆成 `zone/template/premium`，避免“普通”一词在不同维度下语义混淆
- 前端 URL 不暴露 WebP preset 和 format 维度，生产路径保持简单
- 第三方工具不持有 R2 凭证，生产上传收口在控制台或本地脚本
- 第三方工具只需要按文件名输出 PNG，职责边界更清晰
- 控制台导入阶段统一执行 PNG 校验、WebP 转换、R2 上传和索引 upsert
- 新表明确归类为 `hearthstone_data`，没有把系统生成状态混入主领域层或用户应用层

## 主要风险

### 1. 需求文件和压缩包过大

缺图数量可能很大，PNG 源图也明显大于 WebP。

建议：

- 单需求文件默认上限为 `200`
- 首版硬上限为 `500`
- 控制台导出前展示预计缺图数量、本次导出数量和剩余数量
- 超过上限时通过游标或下一批导出继续处理
- 导入端拒绝超过硬上限的需求文件和异常超大 ZIP

### 2. ZIP 安全风险

第三方返回的是压缩包，存在 zip bomb、路径穿越、重复文件名和未知文件风险。

建议：

- 解压前检查压缩包总大小、条目数量和预估解压大小
- 只接受需求文件中声明的 PNG 文件名
- 拒绝绝对路径、`..`、嵌套目录和重复文件名
- 忽略 `__MACOSX/` 与 `.DS_Store`
- PNG 解码前执行单文件大小限制

### 3. PNG 输出不符合目标样式

第三方工具可能输出尺寸错误、透明通道异常或非 PNG 文件。

建议：

- 需求文件显式包含 `style.width`、`style.height` 和 `transparentBackground`
- 导入端校验 PNG magic bytes、尺寸和基本解码结果
- PNG 不合规时不上传 R2，并记录为失败或拒绝项
- 首版使用小批量样本先验证第三方工具输出

### 4. 失败信息表达能力弱

第三方只返回 PNG ZIP 时，未生成的图片可能只能表现为文件缺失。

建议：

- 首版把缺失文件记录为 `missing-output`
- 控制台展示缺失文件列表，便于重新导出或重跑第三方工具
- 后续如需要详细错误，可在 ZIP 中增加可选 `errors.json`，但首版不要求

### 5. R2 与数据库索引可能不一致

可能出现数据库记录为 `ready`，但 R2 对象已丢失的情况。

建议：

- 默认缺图查询不实时 HEAD，避免慢查询
- 控制台提供手动 `verifyR2`
- 导入后可抽样 HEAD 校验
- 后续补充定时 reconcile

### 6. `renderHash` 不包含图片变体

`renderHash` 只表示渲染输入等价，不包含 `zone/template/premium`。

这是合理设计，但实施时必须保证：

- 图片资产唯一键包含完整变体元组
- R2 key 包含完整变体路径
- 需求文件中的 PNG 文件名不能只使用 `renderHash`
- `requestId` 必须由 `imageSpecVersion + renderHash + variant` 稳定计算

### 7. 导入端转换成本集中

PNG 到 WebP 的转换发生在控制台或本地脚本，导入大批量任务时可能占用 CPU。

建议：

- 首版控制单批数量
- 本地脚本支持并行转换，但限制并发
- 控制台导入大文件时走后台任务而不是同步请求
- 记录平均转换耗时和失败原因，为后续队列化做准备

### 8. 需求文件包含完整 `renderModel`

需求文件便于第三方工具离线渲染，但会暴露渲染输入。

建议：

- 需求文件只在控制台内部下载或受控传递
- 不把需求文件放到公开 R2 prefix
- 不在文件中包含 R2 凭证、数据库连接或用户信息
- 如果未来第三方工具不可信，改为只导出 requestId 并提供受控拉取 API

### 9. 图片协议版本升级成本

一旦 `imageSpecVersion` 变化，可能需要重导出大量任务。

建议：

- 只有不兼容变更才升级版本
- 小的渲染器 bugfix 优先通过 `status = stale` 与局部导出处理
- 默认生产 WebP preset 改变时必须升级版本
- 版本升级前先产出 diff 统计

## 需要在实施前确认的问题

- `play` 图是否需要与 `hand` 不同的尺寸
- `battlegrounds` 与 `normal` 是否足以覆盖全部玩法模式
- `signature` 是否对应当前模型中的“异画”，还是需要更细分的 cosmetic 类型
- `diamond` / `signature` 是否所有卡都允许生成，还是需要 eligibility 规则
- 图片公开 URL 是否走同一个 `assetBaseUrl`
- 单需求文件默认上限是否采用 `200`，硬上限是否采用 `500`
- ZIP 导入是优先在控制台完成，还是先落地为本地脚本

## 实施建议

- 第一版先实现 `card_image_assets` 与缺图查询
- 第一版导出文件只支持单 JSON 需求文件
- 第一版第三方结果只支持 PNG ZIP
- 第一版只支持固定 WebP 生产产物
- 第一版只实现手动导出与手动导入
- 第一版缺图查询默认只看索引表，R2 校验作为手动动作
- 第三方工具先用离线文件协议打通，再考虑 API 化
