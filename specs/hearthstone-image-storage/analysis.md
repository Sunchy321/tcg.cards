# 炉石图片 WebP 标准评估结论

数据来源：`output/webp-variants/preset__quality.json`

## 目的

本文基于当前一次完整评估输出，总结现行 `excellent / good / review` 评级标准的实际效果，并给出是否应当调整标准的结论。

这里关注的不是“单张图片能否挑出最优配置”，而是“在海量前端图片生产场景下，是否应继续使用当前标准作为统一 WebP preset 的主要决策依据”。

## 当前标准的实际含义

当前标准会把一个 variant 视为“可接受”，前提是它达到：

- `excellent`
- 或 `good`

否则一律记为 `review`。

当前统一 preset 的选择逻辑还增加了一层严格约束：

- 样本集中至少 `95%` 的图片需要达到 `good` 或 `excellent`
- 不允许出现错误

也就是说，当前标准本质上是一个**高保真标准**，而不是一个偏向“前端可接受即可”的生产标准。

## 当前标准暴露出的核心问题

### 1. 几乎所有 lossy preset 都在质量门槛阶段被提前淘汰

本次输出显示：

- `q60-m6` 到 `q88-m6` 全部为 `review`
- `q90-m6` 与 `q90-m4-fast` 的可接受率仅为 `0.04`
- `q90-m6-sharp-yuv` 的可接受率为 `0.20`
- `q92-m4-sharp-yuv`、`q92-m6-sharp-yuv`、`q95-m6` 的可接受率为 `0.28`

也就是说，在当前标准下，几乎所有常规 lossy 生产候选都没有进入统一推荐竞争。

结果不是“这些 preset 在生产上一定不可用”，而是“当前标准定义下，它们没有资格参与后续时间与体积比较”。

### 2. 生成时间和压缩比虽然已被纳入考虑，但影响很有限

从本次结果可见，不同 preset 的成本差距非常大：

- `near-lossless-60`
  - 平均压缩比：`0.399622`
  - 平均编码时间：`361.2 ms`
- `q92-m4-sharp-yuv`
  - 平均压缩比：`0.173230`
  - 平均编码时间：`54.64 ms`
- `q90-m4-fast`
  - 平均压缩比：`0.147920`
  - 平均编码时间：`40.24 ms`
- `q86-m4-fast`
  - 平均压缩比：`0.122935`
  - 平均编码时间：`39 ms`

这些 fast lossy preset 在体积和生成时间上明显更适合海量生产，但它们因为通过率远低于当前质量门槛，被直接排除在统一推荐之外。

这说明当前标准虽然形式上允许综合考虑成本，但实际决策依然由严格质量门槛主导。

### 3. 当前标准更像“素材保真标准”，而不是“海量前端生产标准”

从本次输出看，当前标准的行为与其设计目标是一致的：

- 它优先保证高保真
- 它容忍更大的体积
- 它并不主动为吞吐量和存储成本让路

因此它非常适合：

- 高质量存档
- 严格保真素材
- 需要尽量避免视觉退化的主资源生成

但对于以下目标并不匹配：

- 海量前端图片批量生成
- 对总体存储和流量极敏感
- 对生成吞吐量要求高
- 允许轻微但可接受的视觉退化

## 结论

### 结论一：按当前标准，当前统一推荐结果是正确的

如果坚持使用当前 `excellent / good / review` 评级体系与 `95%` 通过率门槛，那么：

- `near-lossless-60` 的统一推荐是正确的
- 当前系统行为与设计一致
- 当前输出没有出现逻辑错误

### 结论二：如果目标是海量前端生产，应当更新标准

是，应当更新标准。

原因如下：

- 当前标准过于严格
- 它会在第一阶段就淘汰几乎全部 lossy preset
- 这使得压缩比和生成时间优势无法真正参与统一 preset 决策
- 对海量前端图片场景来说，这会长期推高存储成本、传输成本和生成成本

换句话说：

- **当前标准适合“高保真评估”**
- **不适合作为“海量生产标准”的唯一依据**

## 建议方向

建议保留当前标准，但调整其角色：

- 保留为**高保真基准标准**
- 不再把它作为生产统一 preset 的唯一门槛

同时新增一套更贴近生产目标的标准，特点应包括：

- 以“前端肉眼可接受”为目标，而不是以“尽量接近原图”为唯一目标
- 允许一部分轻微但可接受的 lossy 输出进入候选
- 在通过基础可用性门槛后，再重点比较：
  - 平均压缩比
  - 平均编码时间
  - 统一通过率

## 生产标准提案

### 目标

生产标准用于海量前端图片生成场景下的统一 WebP preset 选择。

它的目标不是追求尽可能接近原图，而是在以下三项之间取得更适合生产的平衡：

- 前端可接受的视觉质量
- 更低的存储与传输体积
- 更高的批量生成吞吐量

### 与高保真标准的分工

后续建议并行保留两套标准：

- **高保真标准**
  - 延续当前 `excellent / good / review` 逻辑
  - 继续用于高保真评估、回归检查、上限参考
- **生产标准**
  - 用于统一 preset 选型
  - 允许更多 lossy preset 进入候选
  - 以聚合后的体积、编码时间和通过率决定最终结果

### 单图通过条件

基于当前 `quality` 结果，建议将生产标准的单图通过条件定义为 `pass`：

- `psnrCompositeRgb >= 35`
- `ssimCompositeLuma >= 0.9975`
- `visibleChangedPixelRatio <= 0.25`
- `alpha.mae <= 0.2`
- `alpha.maxDiff <= 2`

否则记为 `fail`。

这套阈值相较当前高保真标准明显更宽松，但仍然保留了对：

- 全图失真
- 可见变化比例
- 透明边缘稳定性

的基本约束。

### 统一 preset 候选门槛

对某一目标类型的一组样本，统计某个 variant 的：

- `passRate = passCount / sampleCount`

只有满足以下条件的 variant，才允许进入统一 preset 候选集合：

- `passRate >= 0.95`
- `errorCount = 0`

这意味着：

- 至少 `95%` 的样本要达到前端生产可用
- 不允许有处理错误

### 统一 preset 选择规则

在满足候选门槛的 variant 中，选择生产成本分最低者作为统一 preset。

生产成本分定义为：

```text
productionCostScore =
  normalizedAverageSizeRatio * 60
  + normalizedAverageEncodeMs * 30
  + normalizedQualityPenalty * 10
```

其中：

- `normalizedAverageSizeRatio`：平均压缩比归一化结果
- `normalizedAverageEncodeMs`：平均编码时间归一化结果
- `normalizedQualityPenalty = 1 - averageQualityScore / 100`

权重含义为：

- `60%` 体积成本
- `30%` 编码成本
- `10%` 质量惩罚

也就是说，在已经满足生产可用门槛之后，决策会明确偏向：

- 体积更小
- 编码更快

而不是继续盲目追求更高保真。

### 按当前样本推导出的生产结果

按照上述生产标准，本次样本中可以进入统一候选集合的 preset 已经不再局限于 `near-lossless` / `lossless`，而会包含一批 fast lossy 候选。

在这些候选中，当前样本的最低生产成本分对应 preset 为：

- `q86-m4-fast`

其聚合结果为：

- `passRate = 1.00`
- `averageSizeRatio = 0.12293468`
- `averageEncodeMs = 39.0`
- `averageQualityScore = 71.56708`

因此，如果目标是海量前端图片生产，当前样本下推荐的统一生产 preset 应当改为：

- **`q86-m4-fast`**

## 最终判断

- 如果当前目标是高保真主资源生成，可以继续沿用现行标准。
- 如果当前目标是海量前端图片生产，**应当更新标准**。
- 基于本次输出，`near-lossless-60` 是“当前严格标准下的正确答案”，但大概率不是“长期生产策略下的最终答案”。
