# 设计评审：炉石传说高级搜索交互重构

## 评审方式

本次设计通过 /grill-with-docs 技能进行交互式评审。以下是评审过程中的关键决策记录。

## 评审要点

### Q1: 弹窗容器选择

**问题**：旧方案使用页面跳转，新方案用什么 UI 容器？

**讨论**：
- 评估了 UModal（居中弹出）、USlideover（滑入面板）、UDrawer（抽屉）
- UModal 在 `max-w-lg` 下宽度不足 512px，无法容纳两列筛选器布局
- USlideover 支持 `top` / `bottom` 方向，有 `inset` 模式

**决策**：桌面端使用 USlideover `side="top" inset`，移动端使用 `side="bottom" inset`。

### Q2: DSL 回填

**问题**：打开弹窗时是否解析当前搜索词来回填筛选器？

**讨论**：
- 选项 A：空状态启动
- 选项 B：从已有 DSL 回填

**决策**：选项 B。使用 `#search/parser` 包解析 DSL，避免像 site-magic 那样手写 CMD_MAP。

### Q3: 实时同步策略

**问题**：筛选器调整时是否实时同步 DSL 到搜索输入框？

**背景**：当前 `/search/advanced` 页面行为是筛选器变化实时同步 DSL 文本到顶部输入框，但**不触发搜索**，需要用户主动点击"搜索"按钮。

**决策**：保持和当前一致——实时同步 DSL 到输入框，点击"搜索"才提交。

### Q4: 旧页面处理

**问题**：是否保留 `/search/advanced` 页面？

**讨论**：
- 评估了"完全保留"、"保留但不直接访问"、"删除并重定向"三个选项
- 考虑过"点击高级搜索 → 进入搜索界面并自动打开弹窗"的方案
- 空搜索界面打开弹窗的行为经过评估，结论是"不奇怪，反而是更好的体验"

**决策**：删除旧页面（项目处于 dev 阶段，URL 不保留）。

### Q5: 结果与筛选器是否同时显示

**问题**：筛选面板打开时，结果列表是否同时可见？

**讨论**：评估了"同时显示"、"模式切换独占"、"保留简略筛选标签"三个方向。

**决策**：不同时显示，筛选器在 Slideover 中。未来可能保留在结果模式显示简略筛选标签。

### Q6: 公共组件抽取

**问题**：是否提取筛选器表单为公共组件？

**讨论**：旧方案（保留旧页面 + 弹出窗口）需要公共组件。新方案中筛选器只有一个消费方（搜索结果页内的 Slideover），无需抽取。

**决策**：不抽取公共组件。筛选器表单内容整合到搜索结果页中（可直接写在页面或新建内部组件用于组织代码）。

## 评审结论

**状态**：设计通过 ✅

所有设计决策已在评审中确认，不存在未解决的争议或未评估的风险点。可以进入计划阶段。
